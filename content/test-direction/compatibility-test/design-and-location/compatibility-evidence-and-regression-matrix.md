# 兼容性证据收集与回归矩阵

## 这个知识点在解决什么问题
兼容性问题最容易出现“这台有问题，那台没问题”的沟通混乱。证据收集和回归矩阵解决的是：如何把设备、系统、场景、现象和修复状态组织成可复现、可比较、可持续回归的结构。

## 兼容性问题为什么更依赖证据
因为同一个问题在不同设备上可能表现完全不同：
- A 机型直接闪退
- B 机型只是布局错位
- C 机型偶发黑屏

如果缺少明确证据，就很难判断根因是否同一类。

## 证据至少包括什么
- 设备型号、系统版本、ROM 版本
- 复现步骤和出现概率
- 截图、录屏、日志、崩溃堆栈
- 网络、账号、语言、横竖屏等环境条件

如果是图形或布局问题，最好再补：
- 分辨率和缩放比例
- 页面入口位置
- 字体大小、系统语言
- 横竖屏切换时机

## 什么叫回归矩阵
回归矩阵是把“场景 x 设备组合 x 结果状态”结构化记录下来。

```text
登录流程  x  Android 13 / Xiaomi   -> pass
登录流程  x  Android 14 / Huawei   -> fail
支付流程  x  iOS 17 / iPhone 13    -> pass
```

这样修复后就能快速知道：
- 原问题组合有没有回归
- 相邻组合要不要一起补测

## 怎么在团队里落
- 固定高风险场景集
- 固定核心设备池
- 缺陷关闭前要求补最小回归矩阵
- 线上问题要补设备标签回流

## 一个最小回归矩阵可以长这样
```text
场景：支付页打开
修复前失败：Android 14 / 某品牌 / 深色模式
修复后回归：
  - 原机型原版本
  - 同品牌相邻机型
  - 同系统不同品牌
  - iOS 对照组
```

这样才知道修复的是“单点现象”，还是一类兼容性问题。

## 证据收集顺序怎么做更高效
1. 先固定环境
- 设备型号、系统版本、安装包版本

2. 再固定场景
- 页面入口、账号、网络、语言、横竖屏

3. 再固定现象
- 必现还是偶现
- 录屏、截图、崩溃日志

4. 最后补相邻对照组
- 同系统不同机型
- 同机型不同系统
- 原版本和修复版本

这样你更容易知道问题是“设备特有”，还是“系统级”或“业务代码通病”。

## 常见误区
- 只有一句“某安卓机有问题”，没有结构化环境信息
- 修复后只在原设备重测，不看相邻版本和同类机型
- 没有回归矩阵，导致每次兼容性修复都从头试
- 只留截图，不留日志和版本信息，后面无法定位根因
- 没有对照组，导致修复后也无法判断覆盖范围
