# 设备碎片化与版本策略怎么定

## 这个知识点在解决什么问题
兼容性测试最大的难点不是“设备很多”，而是你不可能把所有设备和系统版本全测一遍。设备碎片化与版本策略解决的是：在有限测试资源下，如何选出最有代表性、最容易出问题、最值得持续回归的组合。

## 碎片化具体体现在哪
- 机型品牌和硬件代际差异
- 系统版本差异
- 分辨率、刷新率、屏幕比例差异
- 芯片平台和 GPU 差异
- ROM 深度定制差异

这些差异会直接影响：
- 安装和启动
- 页面布局和适配
- 输入法、权限、通知、后台行为
- 图形渲染和性能表现

## 为什么不能平均抽样
兼容性不是抽彩票。选设备时要看业务风险，而不是简单按数量平均分。

更合理的维度通常是：
- 用户量占比
- 收入或关键业务占比
- 历史问题高发机型
- 新系统版本和重点发版节点
- 特定硬件能力要求

## 一个常见策略
```text
核心回归池
  -> 用户量大、问题高发、关键版本

专项验证池
  -> 新系统、新芯片、新机型

边缘风险池
  -> 低占比但历史上容易出问题的组合
```

## 一个更落地的设备池例子
例如一个移动应用可以这样分：
- 核心池：2 到 3 台主流中端安卓机 + 1 到 2 台主流 iPhone
- 新版池：最新 Android / iOS 版本
- 风险池：历史上常出权限、渲染、通知问题的品牌机型

这样每次版本回归时，不是“随机拿几台手机试”，而是按风险模型执行。

## 一个测试矩阵可以怎么列
```text
设备层：品牌 / 型号 / 芯片 / 内存档位
系统层：Android / iOS 大版本、小版本
场景层：安装、登录、支付、推送、音视频、后台切前台
```

这样做的好处是：你能看出到底是“设备问题”“系统版本问题”，还是“某类业务场景更敏感”。

## 测试开发视角该怎么落
- 维护一份设备矩阵，记录机型、系统、分辨率、CPU/GPU、问题历史
- 把兼容性问题和设备标签关联起来
- 每次大版本根据改动范围动态调整回归池

## 如何根据改动范围选机
- UI 大改版：优先多屏幕尺寸、多分辨率、多字体环境
- 图形渲染改动：优先多芯片、多 GPU、多高刷设备
- 权限/通知改动：优先多系统版本、多 ROM
- 支付/登录改动：优先主流机型和高业务占比机型

这比“每次所有设备全跑一遍”更贴近风险控制。

## 什么时候要动态扩池
- 升新系统版本
- 接入新渲染特性
- 权限模型变化
- 新增音视频、推送、后台保活类能力

这类改动对设备差异更敏感，回归池不能一直不变。

## 常见误区
- 只按新旧机型划分，不看真实用户覆盖
- 只测旗舰机，不测中低端主流机
- 测试池长期不更新，和线上用户结构脱节
- 每个版本都试图全覆盖，结果投入很大但风险控制仍然不聚焦
- 设备池只按硬件列，不把高风险业务场景一起纳入矩阵
