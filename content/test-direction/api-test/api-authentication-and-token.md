# 鉴权与 Token 机制

## 1. 概述

在接口测试中，鉴权与 Token 机制是非常常见的一类基础能力。很多接口并不是“谁都能调”，而是只有通过身份认证并携带有效凭证后，才能被允许访问。

因此，这一部分的核心问题就是：  
**系统如何确认“你是谁”，以及“你有没有权限访问这个接口”。**

其中：

- **认证（Authentication）** 更关注“你是谁”
- **鉴权（Authorization）** 更关注“你能做什么”
- **Token** 则通常是系统在认证成功后发给客户端的一种访问凭证

理解这部分内容，不仅有助于做接口调试，也有助于设计安全性测试、越权测试和登录态相关测试。

---

## 2. 什么是认证与鉴权

### 2.1 认证
认证是确认用户身份的过程。

例如：
- 用户输入用户名和密码登录
- 系统验证账号密码是否正确
- 如果正确，就确认这个用户身份有效

也就是说，认证回答的是：  
**当前发请求的人到底是谁。**

### 2.2 鉴权
鉴权是在身份确认之后，进一步判断该用户是否有权限执行某个操作。

例如：
- 普通用户只能查看自己的订单
- 管理员可以查看全部订单
- 某些接口只有特定角色才能调用

也就是说，鉴权回答的是：  
**这个人有没有权限做这件事。**

---

## 3. 什么是 Token

Token 可以理解为系统发给客户端的一种“访问凭证”。

通常流程是：

1. 用户先登录
2. 服务端校验身份成功
3. 服务端生成一个 Token 返回给客户端
4. 客户端后续访问受保护接口时，携带这个 Token
5. 服务端验证 Token 是否有效
6. 验证通过后，允许访问对应资源

因此，Token 的核心作用是：  
**让系统在后续请求中识别用户身份，而不必每次都重新输入账号密码。**

---

## 4. 为什么很多系统要用 Token

如果没有 Token，客户端每次访问接口都可能需要重新传用户名密码，这既不方便，也不安全。

使用 Token 的主要好处包括：

### 4.1 减少重复登录成本
登录一次后，后续请求可直接携带 Token 访问受保护资源。

### 4.2 降低账号密码重复传输风险
不需要在每次请求里都传账号密码。

### 4.3 便于做登录态管理
服务端可以通过 Token 判断用户当前是否处于已登录状态。

### 4.4 适合前后端分离场景
前后端分离项目中，Token 是非常常见的身份凭证机制。

---

## 5. Token 常见传递方式

在接口测试中，最常见的是通过请求头传递 Token。

### 5.1 放在 Authorization 头中
最常见的形式例如：

- `Authorization: Bearer xxx`
- `Authorization: token xxx`

这是目前最常见的方式。

### 5.2 放在自定义 Header 中
有些系统会自定义请求头，例如：

- `token`
- `access-token`
- `x-token`

### 5.3 放在 Cookie 中
某些系统会把登录态保存在 Cookie 里，后续请求自动携带。

### 5.4 放在请求参数中
这种方式相对不推荐，但有些旧系统或特殊场景仍会使用。

---

## 6. 接口测试中为什么要关注 Token 机制

接口测试中，很多问题都和 Token 机制直接相关，例如：

- 登录成功后拿不到 Token
- Token 格式不对导致访问失败
- Token 失效后系统仍允许访问
- 用户 A 的 Token 能访问用户 B 的数据
- Token 被篡改后系统未正确拦截
- Token 过期机制不清晰
- 刷新 Token 的逻辑异常

因此，Token 机制不仅是登录问题，也是接口安全性、权限控制和状态管理的重要组成部分。

---

## 7. 常见测试点

### 7.1 登录成功后是否正确返回 Token
登录接口成功后，需要检查：

- 是否返回 Token
- Token 字段名是否符合设计
- 返回格式是否正确
- 是否能被后续接口正常使用

### 7.2 不携带 Token 是否会被拦截
对于需要登录态的接口，不带 Token 请求时应返回未认证或未登录错误，而不是放行。

### 7.3 携带无效 Token 是否会被拦截
例如传空 Token、乱写 Token、截断 Token，系统是否都能正确识别并拒绝访问。

### 7.4 携带过期 Token 是否会被拦截
如果 Token 已经过期，系统应返回明确的过期或重新登录提示，而不是继续访问成功。

### 7.5 用户 A 的 Token 是否能访问用户 B 的数据
这是非常典型的越权测试点，重点验证系统是否只根据资源 ID 放行，而忽略了用户身份绑定。

### 7.6 Token 失效机制是否正确
例如：

- 退出登录后 Token 是否立即失效
- 密码修改后旧 Token 是否失效
- 黑名单机制是否生效

### 7.7 刷新 Token 逻辑是否正常
如果系统设计了 access token 和 refresh token，还需要验证刷新流程是否正确。

---

## 8. 常见测试方法

### 8.1 正常登录获取 Token
先调用登录接口，使用合法账号密码获取 Token，再带着 Token 请求业务接口，验证主流程是否打通。

### 8.2 删除 Token 再访问
不带 Token 请求受保护接口，检查系统是否正确返回未登录或未认证错误。

### 8.3 篡改 Token
故意修改 Token 内容，再访问接口，检查系统是否能识别并拒绝。

### 8.4 使用过期 Token
通过构造过期场景或等待 Token 失效后再访问接口，验证过期处理是否合理。

### 8.5 更换用户 Token 验证越权
使用用户 A 的 Token 访问用户 B 的数据，检查是否存在水平越权问题。

### 8.6 退出登录后复测
调用退出接口后，再使用旧 Token 访问业务接口，确认是否已经真正失效。

---

## 9. 常见场景

### 9.1 登录接口
最基础的 Token 获取入口，需要确认认证成功后是否能正确签发 Token。

### 9.2 用户信息接口
通常用于验证 Token 是否能正确识别当前用户身份。

### 9.3 受保护业务接口
例如订单接口、用户资料接口、后台管理接口等，常用于验证鉴权是否正确。

### 9.4 刷新接口
用于验证 Token 过期后的续期机制是否正常。

### 9.5 退出接口
用于验证旧 Token 是否会被正确失效。

---

## 10. 常见问题

接口测试中，和鉴权与 Token 相关的常见问题包括：

- 登录成功但未返回 Token
- Token 返回字段不规范
- 不带 Token 也能访问受限接口
- 伪造 Token 未被拦截
- 过期 Token 仍可继续访问
- 用户 A 的 Token 能访问用户 B 的资源
- 退出登录后旧 Token 仍然有效
- 刷新 Token 逻辑异常
- 错误信息不清晰，难以区分未登录、无权限和 Token 失效

这些问题很多都属于接口安全测试中的高风险问题。

---

## 11. 与权限控制的关系

需要注意的是，**有 Token 不等于一定有权限**。

例如：
- 用户成功登录，拿到了有效 Token
- 但他只是普通用户
- 某个接口只有管理员可访问
- 那么系统仍然应该拒绝访问

也就是说：

- **Token 解决的是身份识别**
- **权限控制解决的是操作边界**

测试时不能只验证“带了 Token 能不能访问”，还要继续验证“带了这个 Token 的用户是否真的有权访问”。

---

## 12. 测试示例

以“用户详情接口”为例，可以这样测试：

### 12.1 正常场景
先登录获取用户 A 的 Token，再请求自己的详情接口，确认正常返回。

### 12.2 未登录场景
不携带 Token 请求该接口，确认返回未认证错误。

### 12.3 无效 Token 场景
手动修改 Token 后再请求，确认接口拒绝访问。

### 12.4 越权场景
使用用户 A 的 Token，请求用户 B 的详情数据，确认系统是否正确拦截。

### 12.5 退出后复测
调用退出登录接口后，继续使用原 Token 请求详情接口，确认已失效。

---

## 13. 面试中如何表达

面试里如果被问到鉴权与 Token，可以这样组织回答：

- 先解释认证和鉴权的区别
- 再说明 Token 是登录成功后的访问凭证
- 再补充接口测试中常见的测试点，如未带 Token、伪造 Token、过期 Token、越权访问等
- 最后强调 Token 只解决身份问题，权限还要单独验证

这样回答会比只说“Token 就是登录后带着访问接口的东西”更完整。

---

## 14. 小结

鉴权与 Token 机制是接口测试中的高频基础内容，它不仅关系到接口能否正常访问，也关系到系统的安全边界是否清晰。

接口测试中通常需要重点关注：

- 登录后是否正确返回 Token
- 不带 Token 是否被拦截
- 无效或过期 Token 是否被正确处理
- 用户是否只能访问自己的资源
- 退出登录后旧 Token 是否失效
- Token 与权限控制是否真正对应

一个真正设计合理的鉴权系统，不只是“登录后能访问接口”，更应该做到“只有正确的人，才能在正确的边界内访问正确的资源”。
