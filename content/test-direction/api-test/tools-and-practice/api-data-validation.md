# MySQL / Redis 数据校验

## 摘要

MySQL / Redis 数据校验的价值，在于确认接口不仅“返回成功”，而且数据库、缓存和最终查询链路也真的一致。它是接口测试从表面验证走向结果闭环验证的关键一步。这个知识点不能只停在“返回值不够”，而要真正展开到：校验什么、怎么查、为什么查、查完之后如何定位问题。

---

## 核心问题

- 为什么接口测试不能只看返回值
- MySQL 校验和 Redis 校验分别在验证什么
- 数据校验和日志、链路、性能测试之间是什么关系

---

## 1. 为什么必须做数据校验

很多接口问题都属于：

- 表面成功，内部错误

例如：

- 返回成功但数据库未写入
- 数据库已更新但缓存仍是旧值
- 主表改了，明细表没改
- 接口报错了，但部分数据已经落库

所以对增删改和状态流转类接口来说，只看响应通常不够。

---

## 2. MySQL 校验到底在查什么

MySQL 校验重点通常包括：

- 数据是否真实写入、更新、删除
- 字段值是否正确
- 状态流转是否符合预期
- 是否产生重复数据、脏数据
- 主表和明细表是否一致

例如一个订单创建接口，不能只看接口返回 `success`，还要继续查：

- 订单主表是否有新记录
- 明细表是否插入完整
- 状态字段是否正确
- 金额、用户 ID、商品 ID 是否匹配请求

也就是说，MySQL 校验回答的是：

- 真实存储层到底发生了什么

---

## 3. Redis 校验到底在查什么

Redis 校验重点通常包括：

- key 是否写到正确命名空间
- value 是否正确
- TTL 是否合理
- 修改后是否已刷新为新值
- 删除或失效后是否仍残留旧缓存
- Redis 与 MySQL 是否一致

例如一个用户信息查询接口，如果命中了旧缓存，即使 MySQL 已经改对了，用户看到的仍然是错数据。

所以 Redis 校验回答的是：

- 外部高频读取结果是否已经同步

---

## 4. 一个更完整的测试闭环怎么做

对于写操作接口，一个更稳的测试闭环通常是：

1. 调接口
2. 看响应码和返回体
3. 查 MySQL
4. 查 Redis
5. 再调查询接口验证端到端结果
6. 必要时补日志或链路核对

这六步里：

- 前两步是表层验证
- 后三步是在确认系统内部状态真的一致
- 最后一步是在定位问题根因

---

## 5. 一个具体场景怎么拆

假设你在测“修改用户昵称”接口。

请求成功后，至少应该验证：

- MySQL 用户表里的昵称是否已经更新
- Redis 用户缓存里的昵称是否同步更新
- 再次调用查询接口是否返回新昵称

如果出现：

- 返回体是新昵称
- MySQL 也是新昵称
- 查询接口却还是旧昵称

那就很可能是：

- 缓存刷新链路有问题

而不是数据库写入问题。

---

## 6. 查询 SQL 和缓存查看为什么不能只背命令

例如，最基础的 MySQL 校验可能像这样：

```sql
SELECT id, nickname, status
FROM user_profile
WHERE id = 10001;
```

而 Redis 校验可能关注：

```text
GET user:profile:10001
TTL user:profile:10001
```

真正重要的不是命令本身，而是你知道：

- 应该查哪张表、哪个字段
- 应该查哪个 key、哪个 TTL
- 它们和这次请求的业务结果是什么关系

否则就会出现“SQL 会写，但查错了地方”的假校验。

---

## 7. 数据校验和日志、链路、性能测试的关系

### 和日志的关系

如果数据校验失败，日志能继续帮助判断：

- 请求有没有真正执行到写库逻辑
- 缓存刷新有没有触发
- 中间有没有报错或补偿失败

### 和链路的关系

数据校验本质上是在验证一条业务链路是否完整闭环。

### 和性能测试的关系

高并发下的数据校验更重要，因为很多问题只有在压力下才会暴露，例如：

- 重复写入
- 超卖
- 数据覆盖
- 缓存和数据库不一致

所以性能测试和数据校验经常必须一起做。

---

## 8. 实际测试里最容易漏什么

### 8.1 只查主表，不查关联表

很多业务真正的问题出在明细表、流水表、状态表。

### 8.2 只查数据库，不查缓存

现代系统里大量线上异常都藏在缓存同步里。

### 8.3 压测或重复提交后不查数据

很多并发问题在响应层根本看不出来，只能通过数据校验发现。

### 8.4 查错表、查错 key

这会把测试活动本身变成错误来源。

---

## 9. 一条实用原则

只要接口会：

- 改状态
- 改数据
- 改缓存

就不要只看返回值。始终问自己：

1. 数据有没有真的改进去
2. 关联数据有没有一起改
3. 缓存有没有同步
4. 查询链路最后看到的是不是同一个结果

只要这四个问题没回答清楚，接口测试结论就还不完整。

---

## 小结

数据校验是接口测试闭环里最能拉开差距的一环。只有把返回值、数据库、缓存和最终查询结果一起验证，测试结论才可靠。真正成熟的接口测试，不是“接口通了”，而是“系统内部和外部表现都一致”。
