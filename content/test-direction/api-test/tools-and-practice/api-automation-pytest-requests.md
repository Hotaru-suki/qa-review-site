# pytest + requests 接口自动化

## 摘要

`pytest + requests` 是接口自动化里非常常见的一套组合。`requests` 负责发请求，`pytest` 负责组织用例、参数化、前后置、断言和结果输出，适合把高频回归接口沉淀成长期可维护的自动化资产。接口自动化的目标不是“把所有接口都脚本化”，而是把高频、稳定、判断标准清晰的接口验证沉淀成可持续回归能力。

---

## 核心问题

- 接口自动化真正适合解决什么问题
- 为什么 `pytest + requests` 很适合做项目级回归
- 自动化脚本怎样分层，才不会很快变成一次性工具

---

## 1. 为什么要做接口自动化

当接口数量越来越多、版本迭代越来越快时，纯手工回归通常会遇到：

- 重复劳动高
- 回归效率低
- 漏测概率高
- 很难接入 CI

接口自动化的本质，是把高频、稳定、规则明确的接口验证沉淀成可以重复执行的回归能力。

它最擅长的不是“替代所有人工判断”，而是稳定执行那些：

- 每次都要回归
- 输入输出相对明确
- 断言可以程序化

的接口场景。

---

## 2. 为什么常用 pytest + requests

### `requests`

优点是：

- 轻量
- 上手快
- 发 HTTP 请求非常直接

适合把接口请求快速封装出来。

### `pytest`

优点是：

- 参数化方便
- fixture 适合做登录、环境准备、数据清理
- 断言简洁
- 报告和 CI 集成成熟

这套组合不是唯一选择，但在测试开发里足够均衡：学习成本不高，扩展性也足够。

---

## 3. 哪些接口适合优先自动化

优先选择：

- 核心业务链路接口
- 高频回归接口
- 契约稳定、规则清晰的接口
- 人工验证成本高但结果判断明确的接口

不建议一开始就把下面这些一股脑全自动化：

- 低频使用接口
- 频繁变更接口
- 高度依赖复杂外部联动的接口
- 结果难以程序断言的场景

自动化最怕“铺得太广，结果全都不稳定”。

---

## 4. 一个基础框架至少要有哪几层

一个能长期维护的接口自动化项目，通常至少分成：

### 配置层

管理：

- 环境地址
- 超时
- 账号信息
- 数据库连接

### 请求封装层

统一处理：

- 发请求
- 公共 Header
- 日志
- 超时和异常

### 用例层

组织：

- 模块场景
- 参数化数据
- 正常与异常路径

### 断言层

统一校验：

- 状态码
- 业务码
- 返回字段和值

### 数据校验层

验证：

- MySQL
- Redis
- 消息
- 其他下游结果

### 报告层

输出：

- 执行结果
- 失败日志
- 关键信息

分层的目的，是避免把 URL、参数、断言和数据库查询全部混在一个测试函数里。

---

## 5. 一个简单实现示意

例如一个极简登录接口自动化可以先这样写：

```python
import requests


def login(base_url, username, password):
    url = f"{base_url}/api/login"
    payload = {"username": username, "password": password}
    return requests.post(url, json=payload, timeout=5)


def test_login_success():
    resp = login("https://test.example.com", "alice", "123456")
    assert resp.status_code == 200
    body = resp.json()
    assert body["code"] == 0
    assert "token" in body["data"]
```

这只是起点。

如果项目继续做大，就不应该把：

- 地址
- 账号
- Token
- 断言规则
- 环境切换

都写死在用例里，而应逐步拆层。

---

## 6. 自动化为什么不能只断言返回值

这是接口自动化最常见的问题之一。

如果只断言：

- HTTP 200
- 业务码成功

很可能漏掉：

- 数据库没正确写入
- 缓存没更新
- 下游消息没发出
- 页面依赖字段不完整

所以更稳的自动化应尽量形成闭环：

- 接口返回正确
- 关键数据状态正确
- 必要副作用正确

否则脚本只是“接口可达检查”，不是完整验证。

---

## 7. 一个更实际的场景

以“创建订单接口”为例，自动化不应该只写：

- 调接口
- 断言返回成功

还应该继续验证：

- 订单表是否生成记录
- 状态字段是否正确
- 金额是否正确
- 如果有消息或库存变更，是否已触发

这时候自动化脚本才真正具备业务验证价值。

---

## 8. 常见误区

- 把自动化等同于“把手工点点点写成脚本”
- 一开始就追求覆盖全部接口
- URL、账号、token 全部硬编码
- 只断言状态码，不校验业务结果和数据变化

这些问题会让自动化项目很快变成难维护、易失效的一次性脚本堆。

---

## 9. 一个更稳的建设顺序

通常可以按这个顺序推进：

1. 先挑核心且稳定的接口
2. 先把请求和断言跑通
3. 再补环境管理和公共封装
4. 再补数据库或缓存闭环
5. 最后接入 CI 和报告

这比一开始就试图做一个“大而全”的框架更稳。

---

## 小结

接口自动化的关键，不在于“会不会写脚本”，而在于是否能把高频接口验证稳定地沉淀成长期可维护的回归体系。`pytest + requests` 之所以常用，不是因为它唯一，而是因为它足够轻、足够灵活，适合从项目实际需求出发逐步搭起自动化能力。
