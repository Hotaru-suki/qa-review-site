# 安全性测试（接口视角）

## 1. 概念

安全性测试主要关注接口是否存在认证、鉴权、数据泄露、恶意调用或非法访问等风险，目的是确认接口在对外提供能力的同时，不会因为设计缺陷或校验不足而被滥用、绕过或攻击。

在接口测试中，很多问题并不是“接口不能用”，而是“接口被不该访问的人访问了”“接口泄露了不该泄露的数据”“接口能被恶意构造请求利用”。  
因此，接口安全性测试的核心问题就是：  
**这个接口是否只对该访问的人开放，只返回该返回的数据，并且不会被轻易滥用。**

---

## 2. 为什么要关注安全性测试

接口天然是系统对外暴露能力的入口，一旦安全控制不足，就很容易成为风险点。实际项目中常见的问题包括：

- 未登录用户可以直接访问受限接口
- 普通用户通过修改参数查看其他用户的数据
- 返回报文中泄露手机号、身份证号、token 等敏感信息
- 错误响应中暴露 SQL、堆栈、服务内部路径
- 鉴权 token 失效后仍能继续访问接口
- 接口缺乏签名、时间戳或防重放机制，导致请求被伪造或重放

因此，安全性测试的核心意义在于：  
**验证接口不仅功能正确，而且访问边界清晰、数据暴露受控、异常信息安全。**

---

## 3. 常见关注点

### 3.1 未登录是否可访问受限接口
对于需要登录态或 token 的接口，需要验证未携带认证信息时是否会被正确拦截，而不是默认放行。

### 3.2 越权访问是否被拦截
需要重点验证用户是否只能访问自己有权限访问的数据和功能，而不能通过修改参数绕过权限控制。

越权通常包括：

- 水平越权：访问同级别其他用户的数据
- 垂直越权：低权限用户访问高权限功能

### 3.3 用户 A 是否可查看或修改用户 B 的数据
这是最常见的接口安全问题之一，例如通过修改 `userId`、`orderId`、`recordId` 等参数直接访问非本人资源。

### 3.4 敏感字段是否脱敏
接口返回中若包含敏感信息，应确认是否做了脱敏处理，例如：

- 手机号中间位隐藏
- 身份证部分隐藏
- 银行卡号部分隐藏
- 密码、密钥类字段绝不返回

### 3.5 错误信息是否泄露内部实现
异常报文中不应直接暴露：

- SQL 语句
- 堆栈信息
- 服务内部路径
- 框架异常原文
- 表名、字段名、类名等实现细节

### 3.6 是否存在 SQL 注入、参数篡改、重放等风险
对于输入参数、鉴权参数、签名参数等，需要关注是否可被恶意拼接、伪造或重复利用。

### 3.7 Token 失效后是否正确拦截
需要验证 token 过期、伪造、篡改、为空时，系统是否都能正确识别并拒绝访问。

---

## 4. 常用方法

### 4.1 改 token 测试
通过替换、删除、伪造、篡改 token，验证接口是否能正确处理以下场景：

- 不带 token
- 带无效 token
- 带过期 token
- 带其他用户 token

### 4.2 改关键参数测试越权
针对 `userId`、`orderId`、`regionId`、`recordId` 等资源标识字段，尝试替换为其他用户的数据标识，检查是否能越权访问。

### 4.3 构造非法输入
在输入参数中构造特殊字符、恶意拼接内容、异常长度内容等，观察系统是否存在注入、解析异常或安全绕过问题。

### 4.4 检查返回报文与日志暴露信息
关注接口报错时返回给前端的信息是否过多，同时确认日志中是否存在敏感信息明文输出风险。

### 4.5 测试签名、时间戳、鉴权逻辑
对于需要签名校验、时间戳校验、防重放控制的接口，需要验证这些机制是否真的生效，而不是形同虚设。

---

## 5. 常见问题

安全性测试中经常发现以下问题：

- 未登录可直接访问受限接口
- 普通用户可访问管理员数据
- 修改参数后可查看他人信息
- 敏感字段未脱敏直接返回
- 返回报文泄露内部异常细节
- token 失效后仍可访问
- 请求签名校验缺失或可绕过
- 接口对非法输入缺乏防护
- 日志中打印明文密码、token、手机号等敏感信息

这些问题很多不会影响接口“能不能用”，但会直接影响系统安全边界和数据安全。

---

## 6. 测试示例

以“用户详情接口”为例，安全性测试可以至少覆盖以下内容：

### 6.1 未登录访问
不携带 token 请求接口，检查是否返回未认证错误，而不是直接返回用户信息。

### 6.2 越权访问
使用用户 A 的 token，请求用户 B 的详情数据，验证系统是否正确拦截。

### 6.3 敏感字段检查
查看返回结果中是否直接暴露手机号、身份证号、密码摘要、token 等敏感信息。

### 6.4 错误信息检查
构造非法参数触发异常，确认返回报文中不会直接泄露服务内部实现细节。

### 6.5 token 失效场景
使用伪造或过期 token 调用接口，验证是否会被拒绝访问。

---

## 7. 与其他测试维度的区别

安全性测试主要回答的是：  
**接口有没有被越权访问、恶意利用或泄露敏感数据的风险。**

它与其他测试维度的区别如下：

- **可用性测试**：关注接口能不能调用
- **正确性测试**：关注接口结果对不对
- **健壮性测试**：关注异常输入下是否可控
- **安全性测试**：关注接口边界是否安全、数据是否暴露过度、权限是否可绕过
- **性能测试**：关注负载上来后快不快、扛不扛压

所以，安全性测试更强调访问控制、数据保护和恶意场景防护。

---

## 8. 小结

接口安全性测试是接口测试中非常重要的一部分，它重点关注的不是业务流程本身，而是接口在认证、鉴权、数据返回和异常处理层面是否存在安全风险。

它通常需要重点关注：

- 未登录是否可访问
- 越权访问是否被拦截
- 敏感字段是否脱敏
- 错误信息是否安全
- token 失效后是否能正确拒绝
- 是否存在注入、篡改、重放等风险

一个真正安全的接口，不只是“能正确返回数据”，更要做到“只向正确的人返回正确的数据，并且不会被轻易滥用”。
