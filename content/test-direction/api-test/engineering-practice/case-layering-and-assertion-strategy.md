# 用例分层与断言策略怎么设计

## 这个知识点在解决什么问题
接口自动化和回归用例一多，最容易出的问题不是“不会写脚本”，而是用例层次混乱、断言全堆在一处，导致维护困难、失败难定位。用例分层与断言策略解决的是：怎样把同一接口的验证拆成更稳定、更可维护的结构。

## 用例为什么要分层
一条接口常常同时涉及：
- 契约层
- 数据层
- 业务层
- 风险层

如果所有东西都塞进一条大用例，失败后很难知道到底坏在哪一层。

## 常见分层方式
- 冒烟层：主链路最小验证
- 功能层：正常与异常业务场景
- 数据层：数据库、缓存、消息校验
- 风险层：权限、幂等、限流、异常输入

## 断言为什么要有策略
断言不是越多越好，而是要回答：
- 这条用例最想证明什么
- 出错时希望一眼看出哪层失败

## 一个简单原则
- 响应断言和数据断言分开
- 主结果断言和副作用断言分开
- 共性断言抽公共方法，避免四处复制

## 一个接口怎么拆层
### 创建订单接口
- 冒烟层：返回 200、订单号非空
- 功能层：正常创建、金额校验、状态校验
- 数据层：订单表、库存表、流水表
- 风险层：幂等、越权、异常输入

这样做的好处是：
- 冒烟失败一眼就知道主链路坏了
- 数据层失败能更快定位到副作用链路
- 风险层不会和主流程断言搅在一起

## 常见误区
- 所有断言写在一个函数里
- 同一条用例既测主链路又测大量边角风险
- 断言只看响应，不看数据和副作用
- 公共断言逻辑四处复制，后面难维护
