# 从联调定位到回归闭环怎么串起来

## 这个知识点在解决什么问题
很多团队会把“接口调通”“缺陷修复”“自动化回归”拆成彼此独立的动作，结果同一个问题被反复发现、反复沟通。工程化的接口测试不是停在调试成功，而是要形成从发现问题到形成长期保护的闭环。

## 一个完整闭环包含什么
```text
联调发现异常
  -> 复现并缩小范围
  -> 固化输入、环境和证据
  -> 提交缺陷并推动修复
  -> 修复后回归验证
  -> 关键场景沉淀进自动化
  -> 结果进入报告和版本结论
```

## 联调阶段重点做什么
- 先确认请求是否正确构造
- 再确认返回是否符合契约
- 继续看数据是否真实落地
- 记录业务键、traceId、环境版本、时间点

如果问题一开始就没有证据留存，后面回归很难证明“这次修好的是不是同一个问题”。

## 缺陷记录怎么写才有用
一个好缺陷至少要让开发和后续回归的人回答四个问题：
- 什么场景下出现
- 怎样稳定复现
- 正确结果应该是什么
- 当前证据指向哪一层

## 回归为什么不能只点一次通过
回归至少分三层：
- 修复场景回归：原问题是否解决
- 相邻场景回归：相似接口、相邻状态是否被带坏
- 保护性回归：把高价值场景纳入自动化或固定检查项

## 一个缺陷如何从联调走到闭环
以“库存扣减偶发失败”为例：

1. 联调发现现象
- 下单接口返回成功
- 但部分订单库存未扣减

2. 先固定证据
- 记录订单号、商品 ID、请求时间、traceId
- 抓请求体、响应体、关键日志

3. 缩小范围
- 查订单表是否写入
- 查库存流水是否生成
- 看消息是否投递或消费失败

4. 缺陷提交
- 标明复现条件、影响范围、当前证据

5. 修复后回归
- 原场景再跑一次
- 并发、重试、相邻商品场景一起回归

6. 长期保护
- 把“下单成功但库存未扣”沉淀进接口自动化或发布检查

如果最后一步没做，这个问题大概率还会回来。

## 自动化在闭环里扮演什么角色
自动化不是为了替代联调，而是为了把历史问题变成长期保护：
- 高频主链路问题进冒烟
- 容易回归的问题进接口自动化
- 高风险边界问题进发布前专项检查

这样闭环才真正完成。

## 一个好闭环的最终产物
- 有复现路径
- 有缺陷记录
- 有修复验证
- 有长期保护项
- 有报告或知识沉淀

只做到前两步，问题仍然可能反复出现。

## 典型例子
支付回调接口曾因重复请求导致重复入账：
- 联调阶段通过日志和订单号发现问题
- 缺陷中记录了重复请求时间间隔、订单号和两次资金流水
- 修复后验证第一次成功、第二次被幂等拦截
- 最后把“同业务键重复回调”写进自动化和发布回归清单

这时问题才算真正闭环，而不是“开发说修了，测试再点一遍接口”。

## 常见误区
- 联调时只看页面或接口返回，不保存证据
- 缺陷只写“接口报错”，没有请求体、响应体和业务键
- 回归只验证单点修复，不验证相邻路径
- 自动化只做 happy path，不覆盖历史高风险缺陷
