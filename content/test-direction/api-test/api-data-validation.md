# MySQL / Redis 数据校验

## 1. 概述

在接口测试中，很多时候不能只看接口返回结果，还需要进一步校验数据库和缓存中的真实数据是否与接口行为一致。  
因此，MySQL / Redis 数据校验的核心作用就是：  
**确认接口不仅“返回成功”，而且系统内部数据也真的处理正确了。**

其中：

- **MySQL 校验** 主要用于确认数据是否真实落库、更新、删除或状态流转正确
- **Redis 校验** 主要用于确认缓存是否写入、更新、删除，以及缓存与数据库是否保持一致

这类校验是接口测试从“表面验证”走向“结果闭环验证”的关键一步。

---

## 2. 为什么要做数据校验

在实际项目里，仅看接口返回值往往是不够的，因为可能出现以下情况：

- 接口返回成功，但数据库并没有写入
- 数据库更新了，但缓存仍是旧值
- 接口返回的数据来自脏缓存，不是最新数据
- 状态流转看起来成功，但真实表中状态没改
- 主表改了，明细表没改
- 接口报错了，但部分数据其实已经写进去了

因此，做 MySQL / Redis 数据校验的核心意义在于：  
**用系统真实数据验证接口处理结果，而不是只信接口表面返回。**

---

## 3. MySQL 数据校验主要校验什么

### 3.1 数据是否成功写入
对于新增类接口，需要确认调用后数据库中是否真的新增了一条记录。

### 3.2 字段值是否正确
除了“有没有写进去”，还要看“写得对不对”，例如：

- 用户名是否正确
- 状态值是否符合预期
- 数量、金额是否正确
- 创建时间、更新时间是否正常

### 3.3 数据是否更新成功
对于修改类接口，需要确认目标字段是否已被正确修改，而不是返回成功但实际未生效。

### 3.4 数据是否删除或失效
对于删除类接口，要确认是物理删除还是逻辑删除，并校验实际结果是否符合设计。

### 3.5 状态流转是否正确
很多接口本质上是在改状态，因此需要重点确认数据库中的状态字段是否按预期变化。

### 3.6 是否产生重复数据
对于新增、重试、重复提交类场景，还要确认数据库中是否出现重复记录。

---

## 4. Redis 数据校验主要校验什么

### 4.1 缓存是否成功写入
接口执行后，如果设计要求写缓存，就要确认 Redis 中是否真的生成了对应 key。

### 4.2 缓存值是否正确
不能只看 key 是否存在，还要确认缓存里的内容是否与业务预期一致。

### 4.3 缓存是否及时更新
当数据库中的数据发生修改后，缓存是否同步刷新，是 Redis 校验中最常见的关注点之一。

### 4.4 缓存是否被正确删除
对于删除、失效、状态变更类场景，需要确认旧缓存是否已清理，避免读取到过期数据。

### 4.5 Redis 与 MySQL 是否一致
最终要确认缓存中的数据是否与数据库真实数据保持同步，而不是各自为政。

---

## 5. 常见适用场景

MySQL / Redis 数据校验尤其适用于以下接口场景：

### 5.1 新增类接口
例如创建用户、创建订单、写入事件记录等，重点检查数据是否真实写入数据库。

### 5.2 修改类接口
例如更新用户状态、修改配置、变更区域占用状态等，重点检查更新是否正确生效。

### 5.3 删除类接口
例如删除记录、逻辑下线、缓存失效等，重点检查数据是否真的删除或失效。

### 5.4 查询类接口
如果查询结果来自缓存，则需要确认 Redis 数据是否正确，以及缓存是否与数据库一致。

### 5.5 状态流转类接口
例如支付状态变化、占用状态变化、审批状态变化等，重点检查数据库和缓存是否同步更新。

---

## 6. 常用校验方式

### 6.1 调接口后直接查 MySQL
最基础的做法是接口调用完成后，立即查询数据库相关表，确认数据是否真实变化。

常见核对内容包括：

- 是否新增成功
- 是否修改成功
- 是否删除成功
- 目标字段值是否正确
- 时间字段是否正确更新

### 6.2 调接口后直接查 Redis
如果接口涉及缓存读写，就需要查看：

- key 是否存在
- value 是否正确
- 过期时间是否符合预期
- 是否仍残留旧缓存

### 6.3 MySQL 与 Redis 交叉比对
不仅分别查库和查缓存，还要确认二者数据是否一致。

### 6.4 多接口交叉验证
例如：

- 调用创建接口后查数据库
- 再调用查询接口确认返回一致
- 再查 Redis 看缓存是否同步

这种方式更容易形成闭环验证。

---

## 7. MySQL 校验时常见关注点

### 7.1 表是否正确
需要确认接口写入的是不是正确的业务表，而不是写错表或漏写关联表。

### 7.2 主键或唯一字段是否合理
例如订单号、用户 ID、事件 ID 是否唯一且正确生成。

### 7.3 关键字段是否正确
例如：

- 状态字段
- 数量字段
- 金额字段
- 时间字段
- 操作人字段

### 7.4 是否存在脏数据
例如：

- 半写入数据
- 无效默认值
- 重复数据
- 关联字段为空

### 7.5 主表与明细表是否一致
对于多表场景，需要确认主记录和明细记录之间逻辑一致。

---

## 8. Redis 校验时常见关注点

### 8.1 key 命名是否符合预期
可确认缓存是否写到了正确的命名空间和业务 key 下。

### 8.2 value 结构是否正确
例如 JSON 结构是否完整，字段值是否正确。

### 8.3 TTL 是否合理
对于有过期时间要求的缓存，需要校验过期时间是否正确设置。

### 8.4 更新后是否命中新值
修改类接口执行后，缓存是否已经更新为最新值，而不是继续返回旧值。

### 8.5 删除后是否仍有残留
对于缓存删除逻辑，需要确认旧 key 是否被真正清理。

---

## 9. 常见问题

在接口测试中，通过 MySQL / Redis 校验经常会发现以下问题：

- 接口返回成功，但数据库未写入
- 数据库已更新，但 Redis 未更新
- Redis 命中旧值，导致查询结果错误
- 删除成功，但缓存未清理
- 主表更新了，明细表没同步
- 数据库中状态已变，但接口返回还是旧状态
- 接口失败后部分数据已写入，造成脏数据
- 重复提交导致数据库出现重复记录

这些问题如果只看接口响应，往往很难发现。

## 10. 测试示例

以“区域占用状态更新接口”为例，可以这样做数据校验：

### 10.1 校验 MySQL
调用接口后，查询区域状态表，确认：

- 当前状态是否已从空闲变为占用
- 更新时间是否刷新
- 操作记录是否新增

### 10.2 校验 Redis
如果该状态有缓存，则继续检查：

- Redis 中对应 key 是否存在
- 缓存值是否为最新状态
- 是否仍残留旧值

### 10.3 交叉验证
再调用“状态查询接口”，确认：

- 接口返回值
- MySQL 数据
- Redis 数据

三者保持一致。

---

## 11. 与普通返回值校验的区别

只看接口返回值，能回答的是：  
**接口表面上有没有成功。**

而 MySQL / Redis 数据校验回答的是：  
**系统内部真实数据有没有跟着正确变化。**

因此两者区别在于：

- 返回值校验更偏接口层
- MySQL / Redis 校验更偏数据层
- 前者是表面结果
- 后者是内部事实

真正完整的接口测试，通常应该把两者结合起来。

---

## 12. 实践建议

### 12.1 新增、修改、删除接口尽量都做数据库校验
因为这类接口最容易出现“返回成功但数据没变”的问题。

### 12.2 使用缓存的查询或状态接口尽量补 Redis 校验
尤其是高频查询接口、状态类接口、热点数据接口。

### 12.3 不要只查单一数据源
如果系统同时用了 MySQL 和 Redis，最好能做交叉验证，而不是只查其中一个。

### 12.4 校验要抓关键字段
不一定要把整张表所有字段都看一遍，但关键业务字段一定要重点核对。

### 12.5 自动化中尽量沉淀数据库 / 缓存校验能力
这样能让接口自动化从“浅层断言”升级为“结果闭环断言”。

---

## 13. 小结

MySQL / Redis 数据校验是接口测试中非常重要的一部分，它帮助测试人员从接口表面返回进一步深入到系统真实数据层，确认接口处理结果是否真的正确。

它通常重点关注：

- MySQL 是否真实写入、更新、删除
- Redis 是否正确写缓存、更新缓存、删除缓存
- 缓存与数据库是否一致
- 状态流转是否正确
- 是否出现重复数据、脏数据或旧数据残留

一个真正可靠的接口测试，不只是“返回看起来没问题”，而是接口返回、数据库状态和缓存结果都能形成一致、可信的闭环。
