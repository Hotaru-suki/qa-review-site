# 数据一致性测试

## 1. 概念

数据一致性测试主要关注接口调用前后，不同存储层、不同服务层以及不同展示层之间的数据是否保持同步和一致。

在接口测试中，很多问题并不是“接口没返回”，而是“接口返回看起来正常，但系统里的真实数据已经不一致了”。  
因此，数据一致性测试的核心问题就是：  
**接口返回的数据，是否与系统内部真实状态一致。**

它通常会涉及接口层、数据库层、缓存层、消息层以及前端展示层之间的交叉验证。

---

## 2. 为什么要关注数据一致性测试

在真实项目中，接口往往不是只影响一个地方的数据，而是会同时影响多个层面，例如：

- 接口返回给前端的数据
- MySQL 中的主数据
- Redis 中的缓存数据
- 日志或事件记录
- 下游服务同步后的结果

如果这些地方的数据不一致，就可能出现：

- 页面展示和数据库实际数据不一致
- 数据库已更新，但缓存还是旧值
- 主表更新了，明细表没更新
- 接口返回成功，但下游服务未同步成功
- 前端看起来操作成功，实际业务状态没有真正改变

因此，数据一致性测试的核心意义在于：  
**确保接口不仅返回正确，而且系统内外相关数据都保持统一。**

---

## 3. 常见校验对象

进行数据一致性测试时，常见的校验对象包括以下几类。

### 3.1 接口返回值 vs 数据库实际值
接口返回的数据应与数据库中真实存储的数据一致，尤其是查询类、详情类、状态类接口。

例如：
- 用户信息接口返回的昵称、手机号是否与数据库一致
- 状态查询接口返回的状态值是否与表中状态字段一致

### 3.2 MySQL vs Redis
如果系统使用缓存，则需要重点验证数据库与缓存之间是否一致。

常见问题包括：
- 数据库已更新，但 Redis 仍是旧值
- 删除后数据库没问题，但缓存未清理
- 查询命中了陈旧缓存，导致返回结果错误

### 3.3 主表 vs 明细表
对于有主从结构、订单头明细、业务主记录与流水记录的系统，需要确认多表之间的数据是否匹配。

例如：
- 订单主表金额与明细汇总金额是否一致
- 主状态更新后，关联明细状态是否同步变化

### 3.4 服务 A 返回结果 vs 服务 B 实际处理结果
在微服务或多系统联动场景中，需要校验一个服务返回的结果，是否真正被另一个服务正确接收和处理。

例如：
- 上报服务返回成功，但状态服务是否真的更新
- 创建订单服务返回成功，库存服务是否真的完成扣减

### 3.5 前端展示数据 vs 后端落库数据
接口返回给前端的数据、页面最终展示的数据，以及后端数据库中的数据，三者应保持一致。

这种校验特别适合：
- 列表页
- 详情页
- 状态展示页
- 配置展示页

---

## 4. 常用方法

### 4.1 调接口后查数据库
这是最基础的数据一致性验证方式。

即在调用新增、修改、删除或状态变更接口后，直接查询数据库，确认数据是否真实写入、更新或删除成功。

常见校验内容包括：

- 是否新增成功
- 字段值是否更新正确
- 状态是否已变更
- 更新时间是否刷新
- 是否产生重复记录

### 4.2 调接口后查缓存
当系统使用 Redis 等缓存时，需要进一步确认：

- 缓存是否正确写入
- 缓存是否及时更新
- 缓存是否在删除场景下被正确清理
- 缓存值是否与数据库一致

### 4.3 多接口交叉验证
通过多个相关接口相互印证数据是否一致。

例如：
- 调用创建接口后，再调用查询接口确认结果
- 调用更新接口后，再调用详情接口和列表接口核对数据
- 调用状态变更接口后，再调用统计接口确认是否同步变化

### 4.4 核对事件记录、状态记录、操作日志
对于事件驱动或状态流转明显的系统，可以结合事件表、操作日志、审计记录等信息进行辅助验证。

这种方式适合发现：
- 接口返回成功但未真正执行核心逻辑
- 状态改了但未记录操作轨迹
- 日志和业务数据不匹配

---

## 5. 常见问题

数据一致性测试中经常发现以下问题：

- 接口返回成功，但数据库未写入
- 数据库已更新，但缓存未同步
- 列表页和详情页展示数据不一致
- 主表状态已更新，但明细表未同步
- 接口返回的是旧缓存数据
- 前端显示成功，但实际业务未完成
- 多服务链路中某一步失败，导致局部成功、整体不一致
- 日志显示处理成功，但查询结果仍异常

这些问题通常不会仅靠状态码检查发现，必须结合数据库、缓存和多接口结果一起验证。

---

## 6. 测试示例

以“区域占用状态查询接口”为例，数据一致性测试可以至少覆盖以下内容：

### 6.1 接口返回值与数据库比对
调用查询接口后，检查返回的占用状态是否与 MySQL 中区域状态表一致。

### 6.2 数据库与 Redis 比对
如果该状态同时缓存到 Redis，需要进一步检查 Redis 中保存的状态值是否与数据库相同。

### 6.3 状态变更后的链路验证
先调用“人员进入事件上报接口”，再调用“状态查询接口”，确认：

- 数据库状态已从空闲变为占用
- Redis 状态同步更新
- 查询接口返回最新状态

### 6.4 日志与记录校验
检查事件记录表或日志，确认本次状态变化确实被正确记录，而不是只改了展示结果。

---

## 7. 与其他测试维度的区别

数据一致性测试主要回答的是：  
**系统各层数据是不是统一的。**

它与其他测试维度的区别如下：

- **可用性测试**：关注接口能不能调用
- **正确性测试**：关注接口结果本身对不对
- **数据一致性测试**：关注接口结果、数据库、缓存、日志、前端展示是否一致
- **稳定性测试**：关注长期运行是否稳定
- **性能测试**：关注负载上来后快不快、扛不扛压

所以，数据一致性测试更强调“跨层验证”和“结果闭环”。

---

## 8. 小结

数据一致性测试是接口测试中非常重要的一部分，它不是只看接口返回值，而是要进一步确认接口调用前后，系统中的多层数据是否始终保持统一。

它通常需要重点关注：

- 接口返回值与数据库是否一致
- 数据库与缓存是否一致
- 主表与明细表是否一致
- 上游服务返回结果与下游服务处理结果是否一致
- 前端展示与后端真实数据是否一致

一个真正通过数据一致性测试的接口，不只是“返回看起来没问题”，而是系统中的真实数据也始终处于统一、可信的状态。
