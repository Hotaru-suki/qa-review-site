# 错误码、状态码与异常处理

## 1. 概述

在接口测试中，状态码、业务错误码与异常处理是非常基础但也非常重要的一部分。  
很多时候，接口问题并不只是“有没有返回”，而是“返回得是否规范”“失败时是否清晰”“异常时是否可控”。

因此，这一部分的核心问题就是：  
**当接口成功或失败时，系统是否能用统一、清晰、可理解的方式把结果表达出来。**

通常可以分为三层来理解：

- **HTTP 状态码**：表示请求在协议层面的处理结果
- **业务错误码**：表示业务层面的处理结果
- **异常处理**：表示系统面对错误输入、异常依赖、内部异常时的整体处理方式

---

## 2. 什么是 HTTP 状态码

HTTP 状态码是服务器对客户端请求结果的一种标准化响应标识，主要反映的是请求在协议层面是否被成功接收、理解和处理。

常见状态码包括：

- `200`：请求成功
- `201`：创建成功
- `400`：请求参数错误
- `401`：未认证
- `403`：无权限
- `404`：资源不存在
- `405`：请求方法不允许
- `500`：服务器内部错误

它回答的是：  
**这次 HTTP 请求在协议层面发生了什么。**

---

## 3. 什么是业务错误码

业务错误码是系统在业务层自定义的一套结果标识，用来表达“业务操作是成功还是失败，以及为什么失败”。

例如某些系统会约定：

- `code = 0` 表示成功
- `code = 1001` 表示参数错误
- `code = 2001` 表示用户未登录
- `code = 3002` 表示库存不足

它回答的是：  
**这次业务处理结果到底是什么。**

也就是说，HTTP 状态码更偏协议层，业务错误码更偏业务层。

---

## 4. 为什么不能只看 HTTP 状态码

这是接口测试中非常典型的一个误区。

例如一个接口返回：

- HTTP 状态码是 `200`
- 但业务码是 `1002`
- message 是“订单创建失败”

这说明 HTTP 层请求是成功到达并被处理了，但业务本身并没有成功。

因此，接口测试不能只看状态码，还要继续看：

- 业务错误码
- message 提示
- data 字段
- 数据是否真实变更
- 异常信息是否合理

也就是说：

- **HTTP 200 不等于业务成功**
- **HTTP 500 也不一定说明问题已经表达清楚**

---

## 5. 什么是异常处理

异常处理，是指系统在面对错误输入、非法请求、依赖异常、内部异常等场景时，如何以一种可控、统一、清晰的方式处理并返回结果。

一个好的异常处理，通常应满足以下特征：

- 不崩溃
- 不返回混乱结果
- 不暴露内部实现细节
- 能返回明确错误信息
- 便于前端处理
- 便于测试和开发定位问题

因此，异常处理本质上是接口质量的一种体现。

---

## 6. 为什么要关注状态码、错误码和异常处理

如果这部分设计不清晰，系统就很容易出现以下问题：

- 返回状态码与业务结果不匹配
- 同类错误返回不一致
- 前端无法准确判断失败原因
- 测试人员难以定位问题
- 开发排查效率低
- 错误响应中泄露堆栈、SQL 等敏感信息
- 参数错误和系统错误混在一起，语义混乱

因此，关注这部分的核心意义在于：  
**让接口在成功时表达清楚，在失败时也表达清楚。**

---

## 7. 常见 HTTP 状态码及测试关注点

### 7.1 200 OK
表示请求成功处理。  
测试时仍要继续确认：

- 业务码是否成功
- 返回数据是否正确
- 数据是否真实生效

### 7.2 201 Created
常用于创建资源成功，例如新增用户、创建订单。  
测试时可进一步确认是否确实新增成功。

### 7.3 400 Bad Request
通常表示请求参数错误或格式不合法。  
测试时应关注：

- 是否真正是参数问题
- 错误提示是否清晰
- 是否能准确指出哪个参数有问题

### 7.4 401 Unauthorized
通常表示未认证或 token 无效。  
测试时应关注：

- 不带 token 是否正确返回 401
- 伪造 token、过期 token 是否也能正确返回

### 7.5 403 Forbidden
通常表示用户身份有效，但没有权限执行当前操作。  
测试时适合结合越权场景验证。

### 7.6 404 Not Found
表示接口或资源不存在。  
测试时可验证：

- 错路径是否正确返回 404
- 查询不存在资源时系统如何设计返回方式

### 7.7 405 Method Not Allowed
表示请求方法错误，例如该接口只支持 `POST`，却用了 `GET`。  
测试时可主动构造错误请求方法。

### 7.8 500 Internal Server Error
表示服务端内部异常。  
测试时要关注的不只是“报 500 了”，而是：

- 是否真的是内部异常
- 返回信息是否过于模糊或过于暴露细节
- 是否有日志支撑问题定位

---

## 8. 常见业务错误码设计思路

虽然不同公司和项目会有不同规范，但一个较合理的业务错误码体系通常应具备以下特点：

### 8.1 成功与失败有清晰区分
例如：

- `0` 表示成功
- 非 `0` 表示失败

### 8.2 不同类型错误尽量分类
例如可区分为：

- 参数类错误
- 认证类错误
- 权限类错误
- 资源类错误
- 状态类错误
- 系统类错误

### 8.3 同类错误返回保持一致
例如“参数缺失”不要一会返回 `1001`，一会返回 `2003`，否则前端和测试都难以维护。

### 8.4 错误码与 message 应相互匹配
错误码应该能唯一表达某类错误，message 应该便于人理解。

---

## 9. 接口测试中重点关注什么

### 9.1 状态码是否与场景匹配
例如：

- 参数错误不应返回 200 + 成功提示
- 未登录不应返回 500
- 越权不应返回模糊错误

### 9.2 业务错误码是否清晰
要看失败是否能被明确区分，而不是全部返回同一个失败码。

### 9.3 message 是否可理解
message 不一定要给用户看，但至少要让前端、测试和开发明白问题大概是什么。

### 9.4 错误返回结构是否统一
一个系统里的错误响应最好保持统一字段格式，例如：

- `code`
- `message`
- `data`
- `traceId`（如有）

### 9.5 异常时是否暴露内部实现细节
例如是否直接返回：

- 堆栈信息
- SQL 语句
- 物理路径
- 框架异常原文

这些都属于不合理异常返回。

---

## 10. 常见测试方法

### 10.1 正常场景验证
先验证正常请求下：

- 状态码是否正确
- 业务码是否成功
- message 是否合理
- 返回结构是否完整

### 10.2 参数异常场景验证
例如：

- 缺参
- 参数类型错误
- 非法枚举值
- 越界值

确认系统是否能返回正确状态码和业务码。

### 10.3 认证异常场景验证
例如：

- 不带 token
- 过期 token
- 无效 token

确认是否区分未认证与无权限场景。

### 10.4 权限异常场景验证
例如：

- 普通用户访问管理员接口
- 用户 A 访问用户 B 的资源

确认是否返回合理权限错误。

### 10.5 内部异常场景验证
通过构造依赖异常、数据库异常、下游异常等方式，检查系统是否能合理返回系统错误，而不是直接崩溃或泄露内部细节。

---

## 11. 常见问题

在接口测试中，这部分经常暴露的问题包括：

- HTTP 200，但业务失败
- 参数错误却返回 500
- 未登录和无权限返回混乱
- 同类错误返回的错误码不一致
- message 含义模糊，无法定位问题
- 错误返回结构不统一
- 异常时直接暴露堆栈信息
- 业务失败和系统异常无法区分
- 前端无法根据错误码稳定处理逻辑

这些问题不一定会让接口“不能用”，但会明显降低接口的可维护性和可测试性。

## 12. 测试示例

以“创建订单接口”为例，可以这样验证状态码、错误码与异常处理：

### 12.1 正常下单
传入合法参数，确认：

- HTTP 状态码为 `200` 或 `201`
- 业务码为成功码
- message 合理
- 订单真实创建成功

### 12.2 参数缺失
缺少商品 ID 或数量，确认：

- 返回 `400` 或约定的参数错误响应
- 业务码明确表示参数问题
- message 能指出问题原因

### 12.3 未登录请求
不带 token 下单，确认：

- 返回 `401`
- 业务码和 message 能明确表示未登录

### 12.4 库存不足
参数合法但业务条件不满足，确认：

- HTTP 层可仍为 `200`
- 业务码明确表示库存不足
- message 清晰表达失败原因

### 12.5 数据库异常
模拟数据库写入失败，确认：

- 返回系统异常结果
- 不暴露内部 SQL 或堆栈
- 日志中可追踪问题

---

## 13. 与其他测试维度的关系

状态码、错误码与异常处理虽然看起来更偏“返回规范”，但实际上和很多测试维度都有关联。

例如：

- 和**可用性测试**有关，因为接口最起码要能返回合理结果
- 和**正确性测试**有关，因为错误表达本身也要正确
- 和**健壮性测试**有关，因为异常处理是否可控就是健壮性的核心体现
- 和**安全性测试**有关，因为异常返回不能泄露内部实现细节

所以，这部分既是基础内容，也是质量细节。

---

## 14. 面试中如何表达

如果面试官问“状态码 200 是否代表成功”或“接口异常你怎么测”，可以这样回答：

- 先说明 HTTP 状态码和业务错误码不是一回事
- 再说明 200 只能代表 HTTP 层成功，不代表业务一定成功
- 接着补充会检查业务码、message、返回结构和数据落库情况
- 最后说明异常场景下还会关注错误信息是否清晰、返回是否统一、是否泄露内部实现细节

这样回答会显得更完整，也更体现测试思维。

---

## 15. 小结

错误码、状态码与异常处理，是接口测试中非常基础但非常重要的一部分。  
它们关注的不只是“成功怎么返回”，更是“失败时系统是否仍然表达得清楚、控制得住”。

接口测试中通常需要重点关注：

- HTTP 状态码是否合理
- 业务错误码是否清晰
- message 是否可理解
- 错误返回结构是否统一
- 异常时是否可控
- 是否泄露内部实现细节

一个真正设计良好的接口，不只是正常流程能跑通，更应该在失败和异常场景下，也能保持统一、清晰、可靠的表达方式。
