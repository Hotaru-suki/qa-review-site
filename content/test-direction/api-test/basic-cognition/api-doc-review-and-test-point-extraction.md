# 接口文档评审与测试点提炼

## 摘要

很多接口测试做浅，不是因为不会调接口，而是拿到文档后不知道先看什么、哪些地方最值得追问。接口文档评审真正解决的是：在正式测试前把契约、边界、状态变化和风险点先提炼出来，避免后面一边测一边补洞。

---

## 核心问题

- 拿到接口文档后应该优先看哪些信息
- 哪些字段和流程最容易埋 bug
- 怎样从文档里快速提炼出测试点，而不是只是抄参数表

---

## 1. 文档评审到底在评什么

不是只看：

- URL
- 请求方法
- 入参列表

而是要看：

- 这个接口在业务链路里解决什么问题
- 输入、输出和状态变化是否说清楚
- 失败路径和边界条件是否定义明确

---

## 2. 优先看哪些部分

- 必填 / 非必填参数
- 约束范围和默认值
- 状态流转
- 返回字段含义
- 错误码和失败语义
- 是否有幂等、权限、频控要求

这些信息往往直接决定测试点的主骨架。

### 文档里最值得追问的空白

如果文档没有明确写这些内容，通常都值得追问：

- 参数为空时到底怎么处理
- 同一请求是否允许重复提交
- 成功返回后哪些数据会被异步更新
- 权限和频控边界是什么

---

## 3. 一个更实用的提炼方式

可以把测试点快速拆成：

- 输入校验
- 正常业务路径
- 状态变化
- 数据副作用
- 异常与失败表达

这样就不会只写成“参数 A 测一下、参数 B 测一下”。

---

## 4. 一个典型场景

例如退款接口文档里写：

- 订单必须是已支付
- 同一订单只能退款一次
- 退款成功后异步回写账单状态

那你马上就能提炼出：

- 非已支付订单是否被挡住
- 重复退款是否被幂等拦住
- 同步返回成功后账单状态是否最终收敛

这才是从文档到测试点的真正转换。

### 一个更像测试工作的提炼结果

更好的输出通常不是“我看过文档了”，而是：

- 关键前置状态有哪些
- 关键输入边界有哪些
- 关键副作用和异步链路有哪些
- 文档还缺哪些定义

---

## 5. 一个更完整的评审顺序

可以按：

1. 先看业务目标
2. 再看输入约束
3. 再看输出和错误表达
4. 最后看状态变化和依赖链

---

## 6. 常见误区

- 只看参数，不看业务状态前置
- 只看成功响应，不看副作用和异步链路
- 文档没写就默认没有风险
- 把文档评审等同于“接口能否调通”

---

## 小结

接口文档评审的价值，不是为了提前背接口，而是为了在测试开始前把契约和风险看清楚。只要文档评审做得好，后面的用例设计、联调和回归都会更稳。
