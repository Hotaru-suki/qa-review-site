# 接口性能测试

## 1. 概念

性能测试主要关注接口在不同负载条件下的响应速度、吞吐能力、资源消耗和承压上限，用于判断系统是否能够满足业务要求。

如果说可用性测试关注“能不能用”，正确性测试关注“做得对不对”，稳定性测试关注“长时间运行稳不稳”，健壮性测试关注“异常情况下会不会失控”，那么性能测试关注的就是：  
**在请求量上来之后，接口还能不能快速、稳定地处理请求。**

它不仅要看“快不快”，还要看“能承受多少请求”“资源消耗如何”“瓶颈在哪里”。

---

## 2. 为什么要关注性能测试

很多接口在低并发、少量请求时表现正常，但一旦进入真实业务场景，就可能暴露性能问题，例如：

- 响应时间明显变慢
- 并发一上来就开始报错
- 吞吐量达不到业务预期
- 数据库、缓存、线程池成为瓶颈
- 高峰期请求堆积，导致系统雪崩
- 接口虽然返回成功，但耗时过高，用户体验极差

因此，性能测试的核心意义在于：  
**验证接口在实际或接近实际业务负载下，是否仍然具备可接受的响应能力和承载能力。**

---

## 3. 常见性能维度

性能测试通常会从以下几个维度进行分析。

### 3.1 响应时间
即一次请求从发出到收到响应所消耗的时间，是最直观的性能指标之一。

### 3.2 吞吐量
表示系统在单位时间内能够处理的请求数量，用于衡量整体处理能力。

### 3.3 并发能力
指系统在同一时刻可支持多少请求同时处理而不明显退化。

### 3.4 成功率
性能测试不只关心快慢，还要确认在负载下请求是否依然大部分成功。

### 3.5 资源利用率
需要观察 CPU、内存、磁盘、网络、数据库连接池等资源在压测下的消耗情况。

### 3.6 峰值承载能力
即系统在短时间高峰流量冲击下，最多能承受到什么程度。

### 3.7 系统瓶颈位置
通过性能测试识别瓶颈究竟在应用层、数据库、缓存、网络还是外部依赖。

---

## 4. 常看指标

性能测试通常会重点观察以下指标。

### 4.1 响应时间类指标

#### 平均响应时间
反映整体请求的平均耗时情况，是常见基础指标。

#### 最大响应时间
用于观察最慢请求耗时，但容易被个别极端值影响。

#### 最小响应时间
用于了解系统在最理想条件下的响应能力。

#### P90 / P95 / P99 响应时间
用于衡量大部分请求的真实体验，尤其能反映尾部慢请求问题。

例如：
- P95 表示 95% 的请求响应时间不超过某个值
- P99 更适合观察高并发场景下的抖动和长尾问题

### 4.2 吞吐类指标

#### TPS（Transactions Per Second）
每秒事务数，常用于衡量系统整体处理事务的能力。

#### QPS（Queries Per Second）
每秒查询数，常用于查询类接口或数据库查询能力分析。

#### RPS（Requests Per Second）
每秒请求数，更偏 HTTP 请求层面的吞吐量指标。

### 4.3 结果类指标

#### 成功率
在总请求中成功响应的比例。

#### 错误率
在总请求中失败请求的比例，如 4xx、5xx、超时、连接失败等。

#### 超时率
请求未在规定时间内完成的比例，适合用于判断系统高负载下是否出现明显阻塞。

### 4.4 资源类指标

#### CPU 使用率
观察应用服务、数据库服务、缓存服务在压测时的 CPU 占用情况。

#### 内存使用率
需要关注是否持续上涨、是否存在明显泄漏、是否出现频繁 GC 等现象。

#### 磁盘 IO
适合排查日志写入、数据库落盘、大量文件操作等场景下的瓶颈。

#### 网络带宽
用于判断高并发下是否存在网络层拥塞。

#### 数据库连接池占用
判断数据库连接是否成为瓶颈，是否存在连接耗尽风险。

#### Redis 连接数
观察缓存层在高并发场景下的承受情况。

#### 线程池活跃线程数
用于判断应用线程资源是否被打满或任务是否堆积。

## 5. 常见性能测试类型

### 5.1 基准测试
在较低、较稳定的负载下观察接口基础性能表现，作为后续对比基线。

适合回答：
- 单接口正常耗时大概多少
- 当前服务在轻负载下表现如何

### 5.2 负载测试
在预期业务负载下运行，验证系统是否能够达到目标性能要求。

例如：
- 预计平时并发 50
- 峰值并发 100
- 目标响应时间小于 500ms

### 5.3 压力测试
逐步提高并发或请求强度，观察系统在压力增大时的性能变化和瓶颈拐点。

适合回答：
- 系统能扛到多少并发
- 从什么时候开始变慢
- 从什么时候开始报错

### 5.4 峰值测试
模拟短时间流量激增场景，验证系统面对瞬时高峰是否具备抗冲击能力。

常见于：
- 活动开始瞬间
- 秒杀抢购
- 开服登录高峰
- 热门接口瞬时访问暴涨

### 5.5 稳定性测试
在一定负载下长时间运行，观察系统是否持续稳定，是否出现资源泄漏、性能衰退或错误积累。

这类测试与“稳定性测试”关系较近，但在性能专项中通常更强调“带负载持续运行”。

---

## 6. 常用方法

### 6.1 JMeter 并发压测
使用 JMeter 构造并发请求，对接口进行负载、压力或峰值测试，是最常见的方法之一。

### 6.2 阶梯式加压
逐步提高并发数，例如：

- 10 并发
- 30 并发
- 50 并发
- 100 并发

通过分阶段观察性能变化，识别性能拐点。

### 6.3 固定并发持续运行
在固定负载下持续压测一段时间，观察接口是否会随着时间推移出现性能下降或错误增多。

### 6.4 核心接口单独压测
优先对登录、查询、下单、状态查询等核心接口进行专项压测，而不是一开始就把所有接口混合测试。

### 6.5 压测同时监控数据库、缓存、服务资源
性能测试不能只看压测工具报告，还应同步观察：

- 服务 CPU / 内存
- MySQL 连接数与慢查询
- Redis 使用情况
- 日志异常
- 线程池状态

### 6.6 压测后检查数据是否异常
对于写操作接口，压测结束后还应检查：

- 是否产生脏数据
- 是否出现重复写入
- 状态是否正确
- 数据库与缓存是否一致

---

## 7. 注意事项

性能测试不能只看“接口快不快”，还要综合关注以下问题。

### 7.1 压测时是否报错
如果接口很快，但错误率很高，那么这种“快”没有意义。

### 7.2 数据是否正确
尤其是写接口，在高并发下可能出现数据错乱、重复提交、状态异常等问题。

### 7.3 服务是否出现资源泄漏
长时间压测后，如果内存、连接、线程持续不回落，就可能存在资源管理问题。

### 7.4 下游依赖是否成为瓶颈
有时瓶颈并不在当前服务，而是在：

- MySQL
- Redis
- 消息队列
- 第三方服务
- 网关或负载均衡

### 7.5 缓存、数据库是否出现异常波动
例如：

- Redis 命中率下降
- 数据库慢查询增多
- 连接池耗尽
- 锁竞争严重

---

## 8. 常见问题

性能测试中常见暴露的问题包括：

- 平均响应时间过高
- P95 / P99 抖动明显
- 吞吐量上不去
- 并发稍高就开始报错
- 数据库成为系统瓶颈
- Redis 连接数过多
- 服务 CPU 打满
- 内存持续升高
- 高峰时请求超时严重
- 压测后出现脏数据或重复数据

这些问题很多在普通功能测试中难以发现，但在真实业务高峰时会直接影响系统可用性和用户体验。

---

## 9. 测试示例

以“状态查询接口”为例，性能测试可以至少覆盖以下内容：

### 9.1 基准测试
在低并发条件下测试接口平均响应时间，作为基线。

### 9.2 负载测试
模拟日常业务负载，例如 50 并发持续运行，观察：

- 平均响应时间
- P95 响应时间
- 成功率
- CPU / 内存变化

### 9.3 压力测试
逐步提升到 100、200、300 并发，观察系统从哪里开始明显退化。

### 9.4 峰值测试
短时间拉高并发，模拟突发流量，验证系统是否出现明显超时或错误。

### 9.5 压测后数据验证
如果接口涉及状态变更或写入逻辑，还要确认压测后数据库、缓存、日志中的数据仍然正确。

---

## 10. 与其他测试维度的区别

性能测试主要回答的是：  
**接口在负载上来之后还能不能快、还能不能扛。**

它与其他测试维度的区别如下：

- **可用性测试**：关注接口正常情况下能不能调用
- **正确性测试**：关注接口结果对不对
- **稳定性测试**：关注接口长时间运行稳不稳
- **健壮性测试**：关注异常输入或异常依赖下会不会失控
- **性能测试**：关注负载增加后快不快、扛不扛压、资源消耗如何

所以，性能测试更偏向容量、速度、资源和瓶颈分析。

---

## 11. 小结

性能测试是接口测试中非常重要的一环，它关注的不是接口“能不能返回”，而是接口在真实业务负载甚至高峰流量下，是否仍然具备：

- 可接受的响应速度
- 足够的吞吐能力
- 较高的成功率
- 合理的资源消耗
- 清晰可定位的性能瓶颈

一个真正性能达标的接口，不只是“低负载下很快”，而是能够在业务压力到来时依然稳定、高效地提供服务。
