# 安全性测试（接口视角）

## 摘要

接口安全性测试关注的是：接口作为系统能力入口时，是否只允许正确的人，以正确的方式，访问正确的数据和能力。它不等同于渗透测试，也不只是扫漏洞，而是从认证、鉴权、越权、数据暴露、请求可信度和审计留痕这些维度，验证接口边界是否足够可靠。

---

## 核心问题

- 为什么接口功能完全正确，仍然可能是严重安全问题
- 认证、鉴权、越权、脱敏和防重放分别在保护什么
- 测试时怎样从“普通调用视角”切到“攻击者视角”

---

## 1. 为什么接口天然是高风险暴露面

页面只是交互入口，真正执行业务动作的通常是后端接口。

只要接口暴露在网络上，就必须回答：

- 这个请求是不是合法用户发起的
- 这个用户有没有权限访问这份数据
- 这个请求有没有被篡改、伪造或重放
- 返回结果里有没有暴露不该公开的信息

如果这些问题回答不清，即使业务流程完全正确，系统仍然是不安全的。

所以接口安全测试本质上是在验证：

- 边界有没有被正确收紧
- 风险动作有没有被正确约束
- 敏感信息有没有被正确裁剪

---

## 2. 接口安全最常见的风险怎么分

### 2.1 认证缺失

本应要求登录或携带 token 的接口，未登录也能访问，或者过期 token 仍能调用。

它解决的是：

- 系统有没有确认“你是谁”

### 2.2 鉴权缺失

系统知道你是谁，但没有校验“你能做什么”。

例如：

- 普通用户能访问管理员接口
- 用户 A 改参数后就能查用户 B 的数据

它解决的是：

- 这个身份能不能执行当前动作

### 2.3 数据暴露过度

接口返回了不该返回的字段，例如：

- 明文手机号、身份证
- token、密钥、签名字段
- 内部主键、内部路径、SQL、堆栈

它解决的是：

- 即使功能成功，也不能顺手泄露系统内部信息

### 2.4 请求可信度不足

接口缺少签名、时间戳、防重放或参数完整性保护，攻击者可以：

- 伪造请求
- 篡改关键参数
- 重放历史请求

它解决的是：

- “这个请求是不是可信的”

---

## 3. 认证和鉴权为什么必须拆开测

很多项目最容易混淆的就是：

- 认证
- 鉴权

最简单的区分是：

- 认证解决“你是谁”
- 鉴权解决“你能做什么”

所以：

- 带 token 了，不代表安全
- 登录成功了，也不代表权限正确

最典型的例子就是水平越权：

- 用户 A 已登录
- 但只要改 `userId` 或 `orderId`
- 就能查或改用户 B 的资源

这种问题说明认证成立了，鉴权却失败了。

---

## 4. 安全测试到底怎么做

### 4.1 先识别高风险接口

优先看：

- 用户资料、订单详情、资产类查询接口
- 管理后台、审批、配置修改接口
- 支付、充值、发券、积分变更接口
- 第三方回调、开放平台、内部服务接口

这些接口的共同点是：

- 一旦越权或被伪造，业务损失直接可见

### 4.2 再列攻击视角测试点

最常见的一组安全测试点可以按下面顺序展开：

- 未登录访问
- 过期 token / 伪造 token
- 不同角色 token 交叉调用
- 参数篡改
- 资源标识越权
- 重放请求
- 敏感字段检查
- 错误信息暴露检查

这样思路会比“我随便改几个参数试试”稳定很多。

---

## 5. 一个用户详情接口怎么测安全

假设接口是：

- 查询用户详情

更完整的安全测试至少包括：

### 认证验证

- 不带 token 是否被拦截
- 过期 token 是否失效
- 随机伪造 token 是否被拒绝

### 水平越权验证

- 用户 A 的 token 查询用户 B 的详情
- 修改路径参数、query 参数或请求体中的用户标识

### 垂直越权验证

- 普通用户是否能调用仅管理员可访问的详情字段或操作

### 数据暴露验证

- 是否返回明文手机号、身份证、token、内部 ID
- 错误响应里是否暴露 SQL、堆栈、内部路径

### 审计验证

- 越权访问和失败请求是否在日志或审计记录里留痕

这个例子说明：  
接口安全测试不是看“接口能不能成功”，而是看“它会不会在不该成功的时候成功”。

---

## 6. 参数篡改和恶意输入要怎么测

很多接口安全问题并不是“未登录就能访问”，而是：

- 改了关键参数就越权
- 改了金额、资源 ID、状态值就绕过流程

常见测试动作包括：

- 修改 `userId`、`orderId`、`roleId`
- 修改金额、数量、折扣、状态字段
- 构造超长值、空值、特殊字符、异常格式
- 修改签名字段、时间戳、nonce

你要观察的不是“报不报错”这么简单，而是：

- 系统有没有识别非法请求
- 非法请求有没有真正被拦截
- 是否留下了安全审计痕迹

---

## 7. 重放风险为什么要单独看

对于支付、回调、发券、状态推进类接口，重放风险非常典型。

场景例如：

- 第三方回调包被重复发送
- 历史请求被抓包后再次重放
- 同一个签名在有效期内被重复提交

这时测试重点包括：

- 是否有时间戳、nonce、签名校验
- 是否有幂等控制
- 重放后是否会重复推进状态或产生副作用

所以接口安全和幂等性在这类接口上往往是相连的。

---

## 8. 安全测试不能只看返回码

很多安全问题在返回码上看起来完全正常。

例如：

- 仍然返回 200
- 但其实返回了别人的数据
- 或者返回了超出权限范围的字段

所以安全测试至少要同时看：

- 调用是否被允许
- 资源范围是否正确
- 返回字段是否最小化
- 敏感字段是否脱敏
- 日志 / 审计是否留痕

也就是说，安全测试是：

```text
能不能调
调到了什么
看到了什么
系统有没有记录
```

---

## 9. 常见误区

### 9.1 带 token 就以为安全

认证通过不代表鉴权正确。

### 9.2 只测未登录，不测越权

真实系统里越权问题往往比未登录直通更常见。

### 9.3 只看功能返回，不看字段暴露

很多安全问题是系统自己把敏感信息吐出来了。

### 9.4 不看审计和日志

高风险接口除了拦截本身，还应具备可追踪能力。

---

## 小结

接口安全性测试关注的不是业务流程本身，而是边界控制是否足够可靠。它真正要回答的是：谁可以访问接口、能访问哪些资源、能执行哪些动作、请求能否被伪造或重放，以及系统是否把不该暴露的数据和能力严格收住。只有从这些角度看，接口才算真正安全。
