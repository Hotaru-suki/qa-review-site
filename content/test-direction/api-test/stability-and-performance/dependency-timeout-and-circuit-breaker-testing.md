# 依赖超时与熔断降级测试

## 这个知识点在解决什么问题
很多接口本身逻辑不复杂，但一旦依赖数据库、缓存、第三方服务变慢，主接口也会被拖垮。依赖超时与熔断降级测试，解决的是：当下游异常时，主链路是否能及时止损，而不是把故障放大。

## 典型场景
- 第三方支付接口超时
- 推荐服务变慢
- Redis 短时不可用
- 下游 RPC 错误率飙升

## 测试时要验证什么
- 超时是否及时返回
- 是否有兜底或降级结果
- 是否触发熔断而不是持续堆积
- 故障恢复后能否自动回到正常路径

## 一个典型例子
### 商品详情接口依赖推荐服务
如果推荐服务突然变慢：
- 商品详情主信息是否还能返回
- 推荐位是否降级为空或默认值
- 主接口整体 RT 是否仍可接受

这类测试验证的不是“推荐服务有没有问题”，而是主链路有没有被不必要地拖死。

## 怎么设计实验
1. 让依赖服务超时
2. 让依赖服务报错
3. 让依赖服务恢复

分别观察：
- 主接口 RT
- 错误率
- 线程池和连接池占用
- 熔断状态是否打开和关闭

## 为什么这和稳定性强相关
没有超时、熔断和降级时：
- 线程会被长时间占住
- 请求堆积
- 上游一起变慢

所以依赖异常测试本质上是在验证系统的故障隔离能力。

## 常见误区
- 只压正常流量，不测依赖异常
- 只看主接口超时，不看线程池和连接池是否被拖垮
- 故障恢复后不验证熔断状态是否正确恢复
- 只测“失败”，不测降级结果是否符合业务预期
