# 可靠性、吞吐与延迟为什么要一起看

## 这个知识点在解决什么问题
接口性能讨论里最容易出现的误区，是只盯一个数字。例如 QPS 很高，就以为系统表现很好；平均耗时不高，就以为用户体验没问题。实际上，可靠性、吞吐和延迟是一起描述系统行为的三个面。

## 三个指标分别代表什么
### 可靠性
系统在持续运行和异常波动下，是否还能稳定给出正确结果。常见观察量：
- 错误率
- 超时率
- 成功率
- 重试后成功率

### 吞吐
单位时间内系统处理了多少请求或事务。常见口径：
- QPS：每秒查询或请求数
- TPS：每秒事务数，更偏“业务事务完成数”

### 延迟
单次请求从发起到收到结果的耗时。常看：
- 平均值
- 中位数
- P95 / P99
- 最大值

## 为什么不能只看一个指标
```text
高吞吐 + 高错误率 = 系统扛不住
低平均延迟 + 高 P99 = 少量用户体验很差
高成功率 + 低吞吐 = 扩展能力不足
```

所以接口性能测试的结论不能写成“QPS 5000，性能良好”，而要写成“在 QPS 5000 时错误率、P95、CPU、连接池和下游依赖是否仍处于可接受范围”。

## 典型场景
### 查询接口
- 关注 QPS、P95、缓存命中率、下游数据库负载
- 风险是热点 key、缓存穿透、慢 SQL

### 写接口
- 关注 TPS、事务耗时、锁等待、消息堆积
- 风险是重复提交、写放大、串行瓶颈

### 弱网或重试场景
- 关注超时率、重试成功率、重复写入风险
- 风险是吞吐看起来高，但业务上其实出现重复副作用

## 测试时怎么一起看
1. 先定义业务可接受标准
- 成功率要到多少
- P95 不能超过多少
- 峰值吞吐目标是多少

2. 升压时同步观察
- 应用日志
- APM/trace
- CPU、内存、GC
- 数据库连接池、线程池、下游依赖

3. 结论按组合来写
- 在目标吞吐下是否还能保持可靠性
- 为了保住延迟是否牺牲了吞吐
- 为了保住吞吐是否产生了错误和排队

## 一个接口压测场景
假设你在压“库存扣减接口”：
- 1000 TPS 时成功率 99.99%，P95 120ms
- 1500 TPS 时成功率仍高，但 P99 开始显著升高
- 1800 TPS 时错误率升到 1.2%，数据库锁等待明显增加

这时结论不能只写“最高支持 1800 TPS”，而应该写：
- 1500 TPS 以内是稳定区间
- 1500 到 1800 TPS 进入风险区间
- 1800 TPS 以上主要失效模式是锁竞争和重试放大

## 为什么读接口和写接口关注不同
### 读接口
- 更容易先出现缓存、索引、长尾问题

### 写接口
- 更容易先出现锁冲突、事务堆积、重复副作用

所以当你写“吞吐提升”的时候，也要同步确认写接口有没有开始牺牲正确性和一致性。

## 常见误区
- 把 QPS 和 TPS 混着用
- 只看平均耗时，不看长尾
- 只看接口层，不看下游链路
- 不区分读接口和写接口的关注重点
