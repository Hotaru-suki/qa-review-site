# 可用性测试

## 1. 概念

可用性测试主要关注接口“能不能正常用”，即在合法请求条件下，接口是否能够被正常访问、正常处理并返回符合预期的结果。

它强调的是接口的基础可调用能力与主流程可执行性，通常是接口测试中最先验证的一层内容。  
如果一个接口连最基本的访问、请求处理、正常返回都无法保证，那么后续的正确性、健壮性、性能等测试就失去了前提。

---

## 2. 为什么要关注可用性测试

接口可用性是最基础的质量门槛。

在实际项目中，一个接口即使业务逻辑设计得再完整，如果出现以下问题，依然会直接影响使用：

- 接口地址错误，无法访问
- 请求方法不匹配
- 必填参数传入后仍无法正常返回
- 接口返回状态异常
- 联调环境中频繁报错或超时
- 上线后外部系统无法正常调用

因此，可用性测试的核心意义在于：  
**优先确认接口是否具备“可访问、可调用、可返回”的基本能力。**

---

## 3. 重点关注内容

进行接口可用性测试时，通常重点关注以下方面：

### 3.1 接口地址是否可访问
需要确认接口 URL 是否正确，服务是否已正常启动，请求是否能够真正到达目标服务。

### 3.2 请求方法是否正确
例如本应使用 `POST` 的接口，如果误用 `GET`，系统是否能正确拦截或返回合理提示。

常见请求方法包括：

- `GET`
- `POST`
- `PUT`
- `DELETE`

### 3.3 必填参数齐全时是否能正常返回
在参数合法且完整的前提下，接口是否能正常处理请求，而不是出现无意义报错或内部异常。

### 3.4 返回状态码是否符合预期
例如成功请求通常应返回 `200`、`201`，而不是出现 `404`、`405`、`500` 等错误状态。

### 3.5 返回字段结构是否完整
即使接口能返回数据，也需要检查返回报文的基本结构是否正确，例如：

- 是否包含 `code`
- 是否包含 `message`
- 是否包含 `data`
- 字段层级是否与接口文档一致

### 3.6 业务流程是否可闭环执行
对于处在业务链路中的接口，不能只单独看当前接口是否返回成功，还要确认它是否能支撑主流程正常跑通。

例如：
- 登录接口成功后是否能继续访问鉴权接口
- 创建接口成功后是否能在查询接口中查到对应数据

### 3.7 联调环境下接口是否稳定可调用
在开发联调阶段，接口可能存在环境配置错误、服务未启动、依赖未就绪等问题，可用性测试需要尽早识别这些阻塞项。

### 3.8 接口上线后是否可被外部系统正常访问
除了本地和测试环境，还需要关注部署后是否因网关、权限、跨域、路由配置等问题导致接口实际不可用。

---

## 4. 常看指标

可用性测试虽然偏基础，但也有一些非常关键的观察指标。

### 4.1 HTTP 状态码
常见需要关注的状态码包括：

- `200`：请求成功
- `201`：创建成功
- `400`：请求参数错误
- `401`：未认证
- `403`：无权限
- `404`：接口不存在
- `500`：服务内部异常

### 4.2 业务状态码
很多系统除了 HTTP 状态码，还会定义业务码，例如：

- `code = 0`
- `success = true`
- `msg = "ok"`

接口是否真正可用，不能只看 HTTP 层，还要结合业务码判断。

### 4.3 返回时间
可用性测试不要求像性能测试那样深入分析响应耗时，但至少需要确认接口不会出现明显超时、长时间无响应等问题。

### 4.4 返回字段完整性
要确认响应中核心字段是否存在，字段类型和层级是否符合文档预期。

### 4.5 核心数据是否正确落库或更新
对于新增、修改、状态变更类接口，仅仅“返回成功”还不够，还需要确认数据库或缓存中的数据确实发生了对应变化。

---

## 5. 常用方法

### 5.1 正向功能验证
构造一组合法、完整、符合业务规则的请求参数，验证接口能否正常返回。

这是最基础也是最常见的可用性测试方法。

### 5.2 主业务流串联测试
将多个相关接口按照真实业务流程串起来验证，而不是孤立测试单个接口。

例如：
- 登录
- 获取 token
- 调用用户信息接口
- 执行业务操作接口

这样能更真实地验证接口在实际使用链路中是否可用。

### 5.3 最小可用集测试
优先验证系统最核心、最关键的一组接口，确认主要业务流程至少可以跑通。

这种方式常用于：
- 冒烟测试
- 版本提测验证
- 上线前快速确认

### 5.4 冒烟测试
在新版本部署后，用少量核心用例快速判断系统接口是否具备继续深入测试的前提。

如果冒烟阶段就发现大量接口不可用，通常应优先阻塞后续测试，推动环境或服务问题先修复。

### 5.5 联调验证
在前后端联调、服务间联调、第三方接口对接时，通过实际请求确认接口是否已经达到“可接入、可调用、可返回”的状态。

---

## 6. 常见问题

在实际测试中，可用性测试经常暴露以下问题：

- 接口路径写错
- 请求方法使用错误
- 服务未启动或部署失败
- 环境配置错误导致接口不可达
- 鉴权信息缺失导致请求被拦截
- 参数齐全但接口仍报错
- 返回结构与文档不一致
- 接口表面返回成功，但数据库未正确写入
- 网关、代理、跨域配置导致上线后不可访问

这些问题往往不是“复杂逻辑 bug”，但会直接阻塞整个测试流程，因此必须优先解决。

---

## 7. 测试示例

以“用户登录接口”为例，可用性测试可以至少覆盖以下内容：

### 7.1 基础可访问
确认接口地址正确，请求可以正常发出并到达服务端。

### 7.2 正常请求验证
传入正确的用户名和密码，检查：

- 是否返回成功状态码
- 是否返回业务成功标识
- 是否返回 token 或用户信息
- 返回结构是否完整

### 7.3 请求方法验证
如果登录接口规定为 `POST`，则尝试使用 `GET` 请求，确认系统是否合理拦截。

### 7.4 参数完整性验证
传入所有必填字段后，确认接口能够正常处理，而不是出现无意义的空指针、500 异常等问题。

### 7.5 业务闭环验证
登录成功后，继续调用需要鉴权的接口，确认登录接口返回结果确实可用于后续业务流程。

---

## 8. 与其他测试维度的区别

可用性测试关注的是：  
**接口能不能用。**

但它不等于以下维度：

- **正确性测试**：关注接口返回得对不对
- **稳定性测试**：关注接口长时间运行稳不稳
- **健壮性测试**：关注异常输入和异常依赖下抗不抗问题
- **性能测试**：关注高负载下快不快、扛不扛压

也就是说，可用性测试更像是接口质量验证的第一层门槛。

---

## 9. 小结

可用性测试是接口测试中最基础的一层，核心目标是确认接口具备最基本的：

- 可访问性
- 可调用性
- 可返回性
- 可支撑主流程执行的能力

它虽然基础，但非常关键。  
很多项目中的接口问题，最开始暴露出来的并不是复杂逻辑错误，而恰恰是“接口压根不能正常用”。

因此，在接口测试实践中，通常应优先完成可用性验证，再逐步深入到正确性、健壮性、稳定性和性能等更高层次的测试内容。
