# 异常输入与风险防线怎么设计

## 这个知识点在解决什么问题
接口最怕的不是正常流量，而是异常输入、错误顺序、越界值、恶意重放、并发重复提交。风险防线的设计，本质是在问：系统面对“不理想请求”时，能否安全地拒绝、限制、兜底和审计。

## 常见风险输入有哪些
- 参数缺失、类型错误、长度越界
- 枚举值非法、状态流转顺序错误
- 重复提交、并发提交、乱序提交
- 越权访问、伪造用户身份
- 签名篡改、时间戳过期、重放请求
- 大报文、超高频请求、批量试探

## 风险防线通常分几层
```text
客户端校验
  -> 网关/接入层限制
  -> 业务服务参数校验
  -> 权限校验
  -> 幂等/限流/熔断
  -> 审计日志与告警
```

测试时不要只盯某一层是否拦住，而要看多层防线是否协同。例如前端做了必填校验，不代表后端可以不校验。

## 典型测试场景
### 创建订单接口
- 价格字段被篡改为负数
- 同一支付请求快速连续提交两次
- 用户 A 的 token 请求用户 B 的订单详情
- 状态已经“已发货”却继续调用“取消订单”

对应要看的不是只有报错，而是：
- 是否被正确拒绝
- 是否产生脏数据
- 是否留下可定位日志
- 是否触发风控或告警

## 具体怎么设计测试
### 参数校验
- 缺字段、空值、超长、乱码、错误类型
- 看接口返回是否明确、错误码是否稳定

### 状态校验
- 把对象推进到不同业务阶段，再执行不允许的操作
- 看服务端是否按状态机拦截

### 权限校验
- 同角色越权、跨角色越权、游客访问
- 看数据是否被越权读取或修改

### 幂等与重放
- 重复请求同一个业务动作
- 修改时间戳或重复使用签名
- 看是否重复入账、重复发券、重复下单

## 一个完整例子
以“创建提现申请”接口为例，可以专门设计这几类风险输入：
- 金额为负数或超过账户余额
- 同一提现请求连续点击两次
- 使用其他用户 token 提交本人的银行卡 ID
- 在已提现处理中再次发起提现

这时要看的不只是接口是否返回错误，还要继续确认：
- 账户余额有没有被错误冻结
- 提现流水有没有生成重复记录
- 审计日志里是否记录了操作者和失败原因
- 风控或限流是否触发

## 风险防线有没有真正生效，怎么判断
- 输入被拦住后，数据层没有残留半成品
- 错误码稳定，调用方知道如何处理
- 日志中有业务键、用户 ID、失败原因
- 监控能看到异常请求的数量变化

如果只在接口层报错，但数据库已经写了一半，这种防线其实是不合格的。

## 什么叫防线设计得好
- 能拦住错误输入
- 拦截口径稳定，调用方知道如何处理
- 不把内部异常直接暴露给外部
- 出问题后能通过日志和业务键回放现场

## 常见误区
- 只测参数校验，不测业务状态校验
- 只看接口报错，不看数据是否已经被污染
- 只测单请求，不测并发和重放
- 只验证“有没有拦住”，不验证“拦错了会不会误伤正常流量”
