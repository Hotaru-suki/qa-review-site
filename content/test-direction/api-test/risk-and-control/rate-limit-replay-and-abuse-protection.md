# 限流、重放与滥用防护测试

## 这个知识点在解决什么问题
接口风险不只来自错误输入，还来自高频调用、恶意重放和接口滥用。限流、重放与滥用防护测试，解决的是：系统在被高频试探、重复请求或恶意调用时，是否既能保护资源，又不会误伤正常流量。

## 典型风险
- 短时间高频刷接口
- 重复使用旧签名或旧 token
- 批量猜测验证码、券码、资源 ID
- 短时并发请求把某类接口打爆

## 测试时要看什么
- 是否有限流阈值
- 超限后的返回是否稳定
- 重放请求是否被识别
- 防护策略是否影响正常用户

## 一个典型场景
### 验证码接口
如果同一手机号、同一 IP、同一设备短时间反复请求验证码：
- 是否有限流
- 是否会进入风控
- 超限后提示是否稳定
- 正常用户稍后恢复时是否还能正常请求

## 重放测试怎么做
- 重复使用同一签名
- 修改时间窗口外的请求时间
- 复制历史请求多次提交

重点不是“系统有没有报错”，而是：
- 能不能识别重复请求
- 是否阻止了重复副作用
- 是否留下了审计证据

## 为什么这很重要
很多系统不是被“打穿”才算有问题。  
只要滥用流量让正常用户变慢、让资源被异常占满，风险就已经出现了。

## 常见误区
- 只测参数校验，不测调用频率风险
- 只验证拦截，不验证是否误伤正常请求
- 没有区分单用户限流、单 IP 限流和全局限流
- 看到 429 或报错就结束，不继续看副作用是否真的被阻止
