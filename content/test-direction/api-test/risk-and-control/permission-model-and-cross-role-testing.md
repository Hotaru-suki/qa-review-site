# 权限模型与跨角色测试怎么落

## 这个知识点在解决什么问题
很多接口权限问题，不是“有没有登录”这么简单，而是角色边界、对象边界、组织边界没有真正收住。跨角色测试解决的是：一个用户是否能看到不该看的数据、执行不该执行的动作。

## 权限模型通常有哪些层次
- 未登录与已登录
- 普通用户与管理员
- 同租户与跨租户
- 本人资源与他人资源
- 前台角色与后台角色

测试时不能只按接口文档上的“需要登录”去理解，而要先搞清楚系统实际的权限模型。

## 典型场景
### 订单详情接口
- 用户 A 查询自己的订单，应成功
- 用户 A 查询用户 B 的订单，应失败
- 运营角色查询指定用户订单，可能成功，但返回字段可能不同

### 后台操作接口
- 普通客服能否执行审核
- 只读角色能否修改配置
- 某部门管理员能否访问其他部门数据

## 怎么设计跨角色测试
1. 列角色
- 至少列出最小权限角色、普通角色、管理角色

2. 列资源边界
- 自己的数据
- 同团队数据
- 跨团队数据

3. 列动作
- 查看
- 新增
- 修改
- 删除
- 导出

这样可以形成一个“角色 x 资源 x 动作”的测试矩阵。

## 一个测试矩阵长什么样
以“订单后台”举例：

```text
角色              资源范围         动作
普通用户          本人订单         查看
普通用户          他人订单         查看
客服              本组用户订单     查看 / 备注
运营管理员        全量订单         查看 / 导出 / 审核
只读审计角色      全量订单         仅查看
```

把这张矩阵列出来之后，很多隐性风险会马上暴露：
- 导出接口是否比详情接口权限更宽
- 批量操作是否漏做对象边界校验
- 列表页做了权限过滤，详情页是否仍可直接越权访问

## 具体怎么测
### 查看类接口
- 换不同角色 token 访问同一个资源 ID
- 验证返回码、返回字段和数据内容

### 修改类接口
- 用低权限角色提交高权限动作
- 不只看接口是否报错，还要查数据有没有真的被改

### 列表和导出类接口
- 看是否能通过筛选条件或分页参数拿到越权数据
- 导出文件里是否包含敏感字段

### 批量接口
- 混入一部分无权限资源，验证整体处理策略
- 是整批失败、部分成功，还是错误放过

## 看什么证据
- 接口是否直接返回了不该有的数据
- 是否只是前端隐藏按钮，但后端没拦截
- 日志里是否记录了操作人和目标对象
- 越权失败时是否给出稳定的错误口径

更完整一点，还要看：
- 审计日志里有没有角色、用户 ID、目标对象 ID
- 导出、批量处理、异步任务这些旁路接口是否也有权限校验
- 下游数据层是否按租户或组织边界过滤

## 常见误区
- 只测游客和管理员，不测同级越权
- 只测列表接口，不测详情、导出、批量操作
- 只看接口是否报错，不看是否已经泄露敏感字段
- 只改前端菜单和按钮，不验证服务端对象级权限
