# 错误码、状态码与异常处理

## 摘要

状态码、业务错误码和异常处理共同决定了接口在成功或失败时是否表达清楚。它们解决的不是“有没有返回”，而是“返回得是否规范、失败时是否可理解、异常时是否可控”。接口设计成熟与否，很多时候就体现在：客户端能不能明确判断结果、测试能不能稳定断言、开发能不能快速定位问题。

---

## 核心问题

- HTTP 状态码和业务错误码分别表达什么
- 为什么 HTTP 200 不等于业务成功
- 异常处理怎样才算统一、清晰、可追踪

---

## 1. 先把三层含义分开

### HTTP 状态码

它表达的是协议层结果，例如：

- 请求格式是否合理
- 资源是否存在
- 权限是否允许
- 服务端是否发生内部错误

### 业务错误码

它表达的是业务层结果，例如：

- 库存不足
- 用户余额不够
- 状态不允许当前操作

### 异常处理

它表达的是系统在遇到错误输入、未预期异常或内部失败时，是否还能以统一方式返回结果并留下证据。

所以这三层不能混为一谈。

---

## 2. 为什么不能只看 HTTP 状态码

HTTP 200 只能说明：

- 请求在协议层被成功接收并处理到了某个返回点

它不代表业务一定成功。

例如：

```json
HTTP 200
{
  "code": 3001,
  "message": "库存不足",
  "data": null
}
```

这时候协议层成功了，但业务层失败了。

所以接口测试必须同时看：

- HTTP 状态码
- 业务错误码
- message
- data
- 真实数据变化

否则很容易把“业务失败但协议成功”误判成接口正常。

---

## 3. 常见状态码到底该怎么测

### `200`

重点不是“返回了 200 就结束”，而是：

- 业务码是否真的成功
- 返回数据是否完整
- 数据库或下游状态是否正确

### `400`

看的是参数错误是否被正确识别，提示是否清楚。

### `401`

看的是未认证、Token 无效、Token 过期是否被正确拦截。

### `403`

看的是身份已确认，但权限不足时是否正确拒绝。

### `404`

看的是接口路径不存在或资源不存在是否符合设计。

### `405`

看的是请求方法错误是否被识别。

### `500`

看的是系统异常时是否留下足够排查信息，同时又不向外暴露敏感内部细节。

---

## 4. 一个好的错误码体系应该具备什么

一个更成熟的错误码体系通常有几个特点：

- 成功和失败有清晰区分
- 同类问题尽量稳定归类
- 前端、测试、开发都能据此快速判断问题类型
- 参数错误、权限错误、系统错误不要混在一起

如果所有错误都返回：

- `code = 1`
- `message = 操作失败`

那接口虽然“有返回”，但几乎不具备排查价值。

---

## 5. 异常处理到底该达到什么标准

一个好的异常处理至少应满足：

- 系统不崩
- 返回结构统一
- 提示信息可理解
- 不泄露堆栈、SQL、路径等敏感细节
- 日志足以支撑排查

这意味着：

- 对外要克制
- 对内要留证据

如果对外把完整堆栈、数据库错误、服务器路径全部吐出来，就是明显风险；如果对外只说“失败”，对内也没有日志，那排查又会非常困难。

---

## 6. 一个具体测试场景

以“创建订单接口”为例：

### 参数错误

- 数量为空
- 金额格式错误
- 商品 ID 不合法

应看到：

- 合理的参数错误状态码或业务码
- 明确指出哪个字段有问题

### 权限错误

- 未登录下创建订单
- 普通角色访问管理员接口

应看到：

- 401 或 403 类语义正确的结果

### 业务错误

- 库存不足
- 优惠券失效

应看到：

- 协议层可能仍是 200
- 但业务码明确失败原因
- 数据不应误写成功状态

### 系统异常

- 数据库异常
- 下游超时

应看到：

- 返回结果可控
- 外部不暴露内部实现细节
- 日志能支撑后续定位

---

## 7. 测试时怎样看得更完整

做这类测试时，不应只看响应体，还应继续看：

- 数据是否真的变化
- 日志是否记录了关键异常
- 不同错误类型是否被稳定归类
- 前端是否能根据错误码做正确处理

因为错误表达是否清楚，不只是接口自己的问题，也会影响前端交互和后续排障。

---

## 8. 常见误区

- 只看 HTTP 状态码
- 参数错误和系统错误共用一个模糊错误码
- 对外暴露内部堆栈和实现细节
- 错误信息前后不一致，导致前端和测试无法判断

这些问题会让接口“看起来能用”，但实际上不好测、不好用、也不好排查。

---

## 小结

接口不仅要在成功时讲清楚，也要在失败时讲清楚。HTTP 状态码解决协议层表达，业务错误码解决业务层表达，异常处理解决系统在失败场景下是否可控。只有这三层都清楚，接口才真正算得上成熟。
