# 鉴权与 Token 机制

## 摘要

鉴权与 Token 机制解决的是接口如何确认“你是谁”和“你能做什么”。它既是接口调试中的高频问题，也是安全性测试、越权测试和登录态测试里的核心基础。真正要测透这部分，不能只停留在“带不带 token”，而要把认证、鉴权、token 生命周期、越权边界和失效逻辑放到一条完整身份链路里看。

---

## 核心问题

- 认证、鉴权和 Token 分别承担什么职责
- Token 在前后端分离接口里通常怎么工作
- 做接口测试时，未登录、过期、篡改、越权这些场景该怎么拆

---

## 1. 认证、鉴权、Token 分别是什么

最容易记混的是这三个词不在同一层。

### 认证

认证解决的是：

> 当前请求者到底是谁

例如账号密码登录成功，说明系统确认了身份。

### 鉴权

鉴权解决的是：

> 这个身份能做什么

例如普通用户能看自己的资料，但不能看管理员后台。

### Token

Token 是认证成功后发给客户端的一种访问凭证，用来在后续请求里持续带上“我已经登录过”这件事。

所以可以记成：

- 认证解决身份问题
- 鉴权解决权限问题
- Token 解决登录态在请求之间怎么传递

---

## 2. Token 典型工作方式是什么

一个常见流程通常是：

```text
用户登录
  ↓
服务端验证账号密码
  ↓
签发 token
  ↓
客户端保存 token
  ↓
后续请求把 token 放到 Header 或 Cookie
  ↓
服务端校验 token
  ↓
决定是否放行请求
```

最常见的传递方式是：

```text
Authorization: Bearer <token>
```

测试时要注意，真正重要的不是 Header 名字，而是：

- token 是否被正确签发
- 是否被正确携带
- 服务端是否真的按它做校验

---

## 3. 为什么接口测试必须关注 Token

很多接口问题直接和 Token 有关：

- 登录成功后拿不到 Token
- Token 格式错误导致请求失败
- Token 过期后系统仍然放行
- 用户 A 的 Token 访问了用户 B 的数据
- 退出登录后旧 Token 仍可继续使用

这说明 Token 不只是登录功能的一部分，它直接影响：

- 登录态
- 权限边界
- 接口安全
- 数据隔离

---

## 4. 测试时到底该怎么拆

### 正常链路

- 登录成功后正确返回 Token
- 带上 Token 能正常访问受保护接口

### 未登录链路

- 不带 Token 是否被正确拦截
- 空 Token 或格式错误 Token 是否被拒绝

### 失效链路

- 过期 Token 是否失效
- 退出登录后旧 Token 是否失效
- 修改密码或权限变化后旧 Token 是否仍被错误放行

### 越权链路

- 用户 A 的 Token 是否能访问用户 B 的资源
- 普通角色是否能访问管理员接口

这几类场景必须同时看，不能只测“登录成功后能访问接口”。

---

## 5. 一个具体场景

以“查询个人订单接口”为例：

### 正常测试

- 用户 A 登录，拿到 Token
- 用这个 Token 查询自己的订单
- 应返回自己的订单列表

### 水平越权测试

- 用户 A 登录后，把请求中的 `user_id` 改成用户 B
- 如果系统仅按参数查数据，没有基于 Token 的身份做约束，就可能把 B 的订单返回给 A

### 失效测试

- 用过期 Token 再请求
- 应返回未登录或 Token 失效，而不是继续成功

这类测试的核心，是验证系统是否真正把“身份”和“资源边界”绑住了。

---

## 6. refresh token 和 access token 要看什么

如果系统存在双 token 机制，通常要区分：

- access token：短期访问凭证
- refresh token：刷新 access token 的凭证

测试时要关注：

- access token 过期后是否正确拒绝
- refresh token 是否能正确换新
- refresh token 本身是否也有失效机制
- 刷新后旧 access token 是否按设计处理

这一块如果不测，很容易出现“看似有失效机制，实际旧 token 还能继续用”的问题。

---

## 7. 常见误区

- 把认证和鉴权混为一谈
- 只验证能不能登录，不验证权限边界
- 只测主流程，不测过期、篡改和越权场景
- 认为只要有 Token 就一定安全

Token 只是凭证形式，安全性取决于整个校验链路是否真正做对。

---

## 小结

Token 机制的核心，不在于“会不会带请求头”，而在于系统是否真正用它把身份和权限边界管住了。做接口测试时，必须把签发、携带、校验、失效和越权这几条线都测清楚，才能说这套鉴权机制是真的可靠。
