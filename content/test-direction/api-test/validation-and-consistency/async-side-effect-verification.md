# 异步副作用与最终一致性怎么验

## 这个知识点在解决什么问题
很多接口返回成功，只说明同步处理完成了，但真实业务结果可能还在异步链路里继续传播。异步副作用与最终一致性验证，解决的是：当消息队列、延迟任务、异步回调参与链路时，测试怎么证明结果最终真的成立。

## 为什么这类场景容易误判
- 接口响应已成功
- 但消息可能没发出去
- 下游可能没消费
- 数据可能延迟写入

如果测试只看立即结果，就容易把“还没完成”误判成“已经正确完成”。

## 典型场景
- 下单后异步发券
- 支付成功后异步履约
- 提交后异步生成报表
- 修改资料后异步刷新缓存或搜索索引

## 怎么验
1. 先确认同步响应正确
2. 再确认异步任务已创建
3. 继续确认下游消费和最终状态
4. 给出合理等待窗口，而不是无限等

## 一个更完整的例子
### 下单后异步发券
你至少要验证：
- 下单接口返回成功
- 券发放消息已进入队列
- 发券服务实际消费成功
- 用户账户里能看到券
- 券库存和发放次数同步变化

如果只看接口响应和主订单状态，仍可能漏掉“订单成功但券没发”的问题。

## 测试时最关键的两个点
- 要有业务键贯穿同步和异步链路
- 要定义“多久内完成算正常”

否则测试结果就会陷入：
- 不知道该等多久
- 不知道是否真的在查同一笔业务

## 常见误区
- 响应成功就结束测试
- 没有业务键，异步链路追不下去
- 等待时间拍脑袋，导致偶发误判
- 只查消息是否发出，不查最终业务结果是否真的落地
