# 响应契约和业务结果为什么要分开看

## 这个知识点在解决什么问题
很多接口“HTTP 200 且 `code=0`”看起来成功，但业务结果未必真的成立。接口测试如果只盯响应体字段，很容易放过库存未扣减、状态未流转、缓存未刷新这类真实问题。把“响应契约正确”和“业务结果正确”拆开看，才能知道到底是在测接口格式，还是在测系统行为。

## 响应契约在验证什么
响应契约关注的是调用方能否按约定解析结果，通常包括：
- HTTP 状态码是否符合协议语义
- 响应体字段是否齐全、类型是否稳定
- `code`、`message`、`data` 结构是否符合接口文档
- 字段含义和可选项是否与版本约定一致

这部分解决的是“调用方能不能接得住”，并不直接证明业务已完成。

## 业务结果在验证什么
业务结果关注的是接口调用后，系统状态有没有按预期变化，常见检查点有：
- 订单状态是否从“待支付”变成“已支付”
- 库存、余额、优惠券次数是否正确扣减
- 关联表、缓存、消息队列、搜索索引是否同步更新
- 幂等场景下第二次请求是否保持正确结果

这部分解决的是“系统到底有没有做对”。

## 典型测试场景
以“用户下单”接口为例：

```text
请求成功返回
  -> 前端拿到订单号
  -> 订单表写入一条记录
  -> 库存表减少库存
  -> 营销表记录优惠使用
  -> 异步消息通知履约系统
```

如果只看响应：
- 订单号返回了
- message = success

如果继续看业务结果，才会发现：
- 订单表有记录，但库存没扣
- 订单创建了两次
- 订单状态正确，但履约消息没发出

## 为什么这两层要分开写用例
因为失败位置不同，对应定位路径也不同。

契约失败通常落在：
- 接口定义变更
- 序列化/反序列化问题
- 字段兼容性问题
- 网关或协议层处理错误

业务结果失败通常落在：
- 服务内部逻辑错误
- 数据库事务不完整
- 异步链路未消费
- 缓存与数据库不一致

## 实际执行时怎么做
一个接口知识页里的验证动作，通常至少分三层：

1. 响应层
- 看状态码、响应时间、字段结构、错误码

2. 数据层
- 查数据库、缓存、搜索索引、中间件消息

3. 业务层
- 从用户视角或下游系统视角确认状态是否真的生效

## 一个实际例子
例如“领取优惠券”接口：
- 响应层要看领取是否成功、返回的券信息是否完整
- 数据层要看券领取表、券库存表、用户账户券列表是否变化
- 业务层要看用户后续下单时是否真的能用这张券

如果前两层都对，但用户下单仍然不可用，那问题就不在响应契约，而在业务结果和后续链路。

## 常见误区
- `200` 就代表成功：只能说明 HTTP 处理成功，不代表业务完成。
- `code=0` 就不用查数据：很多问题恰恰藏在数据层和异步链路。
- 数据查到了就行：还要确认关联对象是否同步变化，而不是只看单表。

## 应该形成的测试习惯
写接口用例时，最好显式区分：
- 响应契约校验
- 数据结果校验
- 跨系统结果校验

这样后面写自动化时也更容易拆成断言层次，不会所有断言都挤在响应体里。
