# 业务键、状态机与结果校验怎么配合

## 这个知识点在解决什么问题
接口结果校验如果只看“字段值对不对”，很容易漏掉链路问题。真实业务里，很多正确性依赖业务键能串起整条链路，状态机能说明对象是否走在合法路径上。把业务键和状态机一起用，才能把“这次请求到底造成了什么结果”看清楚。

## 什么是业务键
业务键不是数据库自增主键，而是能跨系统定位同一笔业务的标识，例如：
- 订单号
- 支付流水号
- 用户 ID + 活动 ID
- 请求幂等号

测试时，业务键的价值是把接口返回、数据库记录、日志、消息、下游状态串成同一个事件。

## 状态机在校验什么
很多对象不是简单的“存在 / 不存在”，而是会经历一系列合法状态：

```text
待创建 -> 已创建 -> 已支付 -> 已发货 -> 已完成
```

状态机校验的核心不是“当前值是不是某个枚举”，而是：
- 这个状态是否合理
- 这次请求是否允许从上一个状态流转到当前状态
- 是否跳过了中间状态
- 是否出现了回退、乱序或重复流转

## 典型测试场景
### 支付回调
- 先用订单号定位订单主记录
- 再看支付流水表是否写入
- 再看订单状态是否从待支付变为已支付
- 最后看履约消息是否投递

如果状态已变，但流水没写，说明结果并不完整。  
如果流水写了两次，但状态只变一次，可能说明幂等保护不完整。

## 实际校验时怎么做
1. 先拿业务键
- 请求里带了什么唯一标识
- 响应里返回了什么业务对象标识

2. 画状态路径
- 这次调用前对象在哪个状态
- 这次调用后应该走到哪个状态

3. 查上下游结果
- 主表
- 关联表
- 缓存
- 消息
- 日志

## 一个完整链路怎样检查
以“发券接口”举例：

```text
请求发券
  -> 券领取表新增记录
  -> 用户账户下可见券数量变化
  -> 券库存减少
  -> 营销流水记录发放事件
  -> 可能有异步消息通知下游系统
```

如果只看到响应里返回“发放成功”，还远远不够。你至少要继续确认：
- 券领取表里是否真的写入
- 券状态是否从“待领取”变成“已发放”
- 同一用户、同一活动是否出现重复发券
- 券库存和发放流水是否同步变化

## 状态机为什么特别重要
因为很多错误不是“值错了”，而是“流转错了”。

### 典型错误
- 从“待支付”直接跳到“已完成”
- 已取消订单还能再次支付
- 已退款订单又被履约

这些问题只看最终状态字段可能发现不了，必须带着“上一状态是什么、当前动作允许哪些后继状态”去检查。

## 实战里的校验顺序
### 第一步：先校验接口层
- 状态码
- 错误码
- 核心返回字段

### 第二步：再校验主业务对象
- 主表状态
- 金额、数量、版本号

### 第三步：再校验副作用
- 库存、余额、积分、券次数
- 缓存刷新
- 异步消息

### 第四步：最后看证据是否串得起来
- 日志里的业务键是否一致
- traceId 是否贯通
- 同一业务动作有没有出现多次写入

## 自动化怎么落
这类场景适合把断言拆成三层：
- 响应断言
- 数据断言
- 状态流转断言

这样自动化失败时，能快速判断是契约问题、数据问题还是状态流转问题。

## 常见误区
- 只查一张主表，不查关联表和下游状态
- 把数据库主键当业务键，结果跨服务对不上
- 只看最终状态，不看状态迁移是否合法
- 业务键没有记录进日志，导致出了问题只能模糊排查
