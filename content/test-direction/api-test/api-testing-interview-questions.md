# 接口测试常见面试题

## 1. 概述

接口测试是测试岗位面试中的高频考点之一。  
面试官通常不会只问“你会不会用 Postman”，而更关注你是否真正理解：

- 什么是接口测试
- 接口测试主要测什么
- 如何设计接口测试用例
- 如何做接口自动化
- 如何做数据校验
- 如何处理幂等性、性能、安全等问题

因此，整理接口测试常见面试题的目的，不只是背答案，而是帮助建立一套更完整的接口测试表达框架。

---

## 2. 什么是接口测试

### 2.1 常见问法
- 什么是接口测试？
- 你怎么理解接口测试？
- 接口测试和功能测试有什么关系？

### 2.2 回答思路
可以从“定义 + 测试对象 + 核心目标”三个层面回答。

### 2.3 示例回答
接口测试是对系统模块之间数据交互接口进行验证的一类测试活动，主要关注接口在不同输入、不同场景下，是否能够正确完成请求接收、业务处理、结果返回以及异常处理。  
它通常不依赖前端页面，而是直接验证请求参数、响应结果、状态码、业务逻辑、数据落库、缓存变化等内容，是功能测试和系统质量保障中非常重要的一部分。

---

## 3. 接口测试主要测什么

### 3.1 常见问法
- 接口测试主要测试哪些内容？
- 一个接口你一般从哪些方面去测？
- 你做接口测试时关注哪些点？

### 3.2 回答思路
不要只说“测状态码、测返回值”，最好分维度回答。

### 3.3 示例回答
接口测试通常会从以下几个方面展开：

- 可用性：接口能不能正常访问和调用
- 正确性：返回结果、业务逻辑、状态流转是否正确
- 参数校验：必填参数、类型、范围、边界是否合理
- 数据一致性：接口返回、数据库、缓存是否一致
- 健壮性：异常输入和依赖异常下是否可控
- 幂等性：重复请求是否会产生重复处理
- 性能：高并发下响应时间和吞吐量是否达标
- 安全性：是否存在未授权访问、越权、敏感信息泄露等风险

---

## 4. 接口测试和页面测试有什么区别

### 4.1 常见问法
- 接口测试和 UI 测试有什么区别？
- 为什么接口测试更适合自动化？

### 4.2 回答思路
从测试层次、关注重点、执行效率、问题定位几个角度回答。

### 4.3 示例回答
页面测试更偏用户视角，主要关注界面展示、交互逻辑和用户体验；接口测试更偏系统交互视角，主要关注请求响应、业务逻辑和数据流转。  
接口测试不依赖前端页面，因此执行效率更高、定位问题更直接，也更适合自动化回归。很多页面问题，最终都需要通过接口测试来进一步判断是前端问题还是后端问题。

---

## 5. 状态码 200 是否代表接口没有问题

### 5.1 常见问法
- 接口返回 200 就算通过吗？
- 你怎么看 HTTP 200？

### 5.2 回答思路
这是非常经典的问题，重点是说明“200 只代表 HTTP 层成功，不代表业务一定正确”。

### 5.3 示例回答
不代表。  
HTTP 200 只能说明请求在 HTTP 层面被成功处理，不代表业务逻辑一定正确。  
实际测试中还要继续检查：

- 业务码是否正确
- 返回字段是否完整
- 字段值是否符合预期
- 数据是否真正写入数据库
- 缓存是否同步更新
- 业务状态是否真的发生变化

所以 200 只是基础判断，不能作为测试通过的唯一依据。

## 6. 你如何设计一个接口测试用例

### 6.1 常见问法
- 一个新接口给你，你怎么设计测试用例？
- 你写接口用例一般怎么考虑？

### 6.2 回答思路
回答时最好体现结构化思维，而不是只说“测正常、测异常”。

### 6.3 示例回答
我一般会先看接口文档和需求，先明确接口的业务用途、请求方式、参数规则、返回结构以及它对数据库或缓存的影响。  
然后从几个维度设计用例：

- 正常场景：合法参数下是否正确返回
- 参数场景：必填、选填、类型、范围、边界值
- 业务场景：状态流转、角色权限、前置条件
- 数据校验：数据库、缓存是否正确变化
- 异常场景：缺参、错参、重复请求、越权、依赖异常
- 非功能场景：性能、幂等性、安全性等

这样可以保证既覆盖基础功能，也覆盖风险点。

## 7. 接口测试常用工具有哪些

### 7.1 常见问法
- 你平时做接口测试用什么工具？
- Postman、Apifox、pytest、JMeter 分别是干什么的？

### 7.2 示例回答
我平时做接口测试常用的工具主要有：

- Postman / Apifox：用于接口调试、联调验证、环境变量管理
- pytest + requests：用于接口自动化测试
- JMeter：用于接口性能测试和并发压测
- MySQL / Redis 工具：用于校验数据库和缓存数据
- Mock 工具：用于模拟第三方服务或异常场景

这些工具分别对应接口测试中的调试、自动化、性能和数据验证几个不同层面。

---

## 8. 你是如何做接口自动化的

### 8.1 常见问法
- 你做过接口自动化吗？
- 用 pytest + requests 怎么做接口自动化？
- 接口自动化框架一般怎么设计？

### 8.2 回答思路
从“为什么做 + 怎么做 + 框架结构”三个层面回答会更完整。

### 8.3 示例回答
我做接口自动化时，一般会优先选择核心业务链路、高频回归场景和规则相对稳定的接口。  
技术上通常会基于 `pytest + requests` 来实现，请求层负责发送接口请求，pytest 负责组织用例、断言结果和批量执行。  
一个基础框架通常会包括：

- 配置管理：环境地址、账号、数据库配置
- 请求封装：统一处理 baseUrl、header、token
- 用例层：按业务模块组织测试用例
- 断言层：状态码、业务码、字段、响应时间断言
- 数据校验层：校验 MySQL、Redis 数据
- 报告层：结合 Allure 输出测试报告

这样既方便回归，也方便后续接 Jenkins 做持续集成。

## 9. 你如何验证数据库和缓存

### 9.1 常见问法
- 你做接口测试时会查数据库吗？
- Redis 你一般怎么校验？
- 为什么接口测试要查 MySQL / Redis？

### 9.2 回答思路
重点强调“不能只信接口返回”。

### 9.3 示例回答
会。  
对于新增、修改、删除、状态变更类接口，我通常不会只看接口返回，而是会进一步查数据库和缓存。  
MySQL 主要看数据是否真实写入、字段值是否正确、状态是否正确流转；Redis 主要看缓存是否写入、是否及时更新、是否与数据库保持一致。  
因为很多问题是“接口返回成功，但系统内部数据并没有真正处理正确”，所以查库和查缓存是接口测试形成结果闭环的重要环节。

---

## 10. 什么是幂等性？你怎么测

### 10.1 常见问法
- 什么是幂等性？
- 幂等性接口怎么测试？
- 哪些场景需要关注幂等性？

### 10.2 示例回答
幂等性是指同一个请求执行一次和执行多次时，系统结果应保持一致，至少不能因为重复调用而产生重复数据或业务错乱。  
常见场景包括订单提交、支付回调、库存扣减、消息消费等。  
测试时我一般会通过：

- 连续发送相同请求
- 并发发送相同请求
- 模拟超时后重试
- 校验数据库是否出现重复记录
- 校验状态是否重复流转

来验证接口是否真正具备幂等性。

---

## 11. 你如何做接口性能测试

### 11.1 常见问法
- 接口性能测试怎么做？
- JMeter 你一般怎么用？
- 性能测试看哪些指标？

### 11.2 回答思路
说明目标、工具、指标和分析方式。

### 11.3 示例回答
接口性能测试我一般会先明确性能目标，比如并发量、响应时间、成功率等要求，然后使用 JMeter 对核心接口做并发压测。  
过程中重点关注：

- 平均响应时间
- P95 / P99 响应时间
- 吞吐量
- 成功率 / 错误率
- CPU、内存、数据库连接数等资源指标

如果是写接口，压测后还会继续校验数据库和缓存，确认没有出现脏数据或重复数据。  
性能测试不能只看 JMeter 自己的结果，还要结合服务监控一起分析瓶颈。

---

## 12. 你如何做安全性测试

### 12.1 常见问法
- 接口安全性主要测什么？
- 你怎么测越权？
- token 相关问题你一般怎么考虑？

### 12.2 示例回答
接口安全性测试我主要关注：

- 未登录是否可访问受限接口
- 用户 A 是否能访问用户 B 的数据
- 是否存在水平越权和垂直越权
- 敏感字段是否脱敏
- token 失效后是否还能继续访问
- 错误信息是否泄露内部实现细节

测试时通常会通过改 token、改 userId / orderId 等资源标识、构造非法参数等方式验证接口是否存在安全风险。

---

## 13. 你如何做异常场景测试

### 13.1 常见问法
- 除了正常流程，你还会怎么测？
- 健壮性测试一般怎么做？

### 13.2 示例回答
除了正常流程外，我会重点补充异常和边界场景，比如：

- 必填参数缺失
- 参数类型错误
- 非法枚举值
- 超长字符串
- 重复请求
- token 失效
- 越权访问
- 下游服务异常
- 数据库或 Redis 异常

我关注的不只是接口会不会报错，还会进一步看：

- 报错是否清晰
- 状态码是否合理
- 系统是否会崩
- 数据是否被写乱
- 日志是否便于排查

---

## 14. 如何理解 Mock

### 14.1 常见问法
- Mock 是什么？
- 为什么接口测试要用 Mock？
- Mock 能替代真实联调吗？

### 14.2 示例回答
Mock 是通过模拟接口、模拟依赖服务返回结果来替代真实下游系统的一种测试手段。  
它主要用于依赖未完成、依赖不稳定、异常场景难构造等情况，可以帮助测试更早开始，也能提高异常场景覆盖率。  
但 Mock 不能完全替代真实联调，因为它解决的是“可控测试条件”问题，而不是真实链路打通问题。实际项目中，Mock 和真实联调通常是配合使用的。

---

## 15. 如果接口文档不完整，你怎么办

### 15.1 常见问法
- 接口文档不清晰时你怎么测？
- 需求和接口文档不一致怎么办？

### 15.2 示例回答
如果接口文档不完整，我一般不会直接凭感觉测，而是先结合需求说明、前端交互、数据库设计和开发沟通，把接口的业务规则尽量补齐。  
如果文档和实际返回不一致，我会先确认是文档没更新，还是接口实现有问题，再决定是提缺陷还是推动文档修正。  
我觉得接口测试不只是执行测试，也包括对接口契约清晰度的关注。

---

## 16. 接口测试中最容易犯的错误有哪些

### 16.1 常见问法
- 接口测试有哪些常见误区？
- 新手最容易忽略什么？

### 16.2 示例回答
我觉得常见错误主要有：

- 只看状态码，不看业务结果
- 只测正常流程，不测异常流程
- 不查数据库和缓存
- 忽略幂等性、权限和数据一致性
- 只会用工具发请求，但不会分析业务逻辑
- 自动化覆盖很多接口，但核心链路反而没覆盖好

本质上，接口测试不能停留在“会调接口”这个层面，更重要的是有没有系统化验证接口质量的思维。

---

## 17. 回答接口测试面试题的建议

### 17.1 不要只说工具
面试官更看重的是测试思路，而不是单纯会不会用某个工具。

### 17.2 尽量结构化回答
可以使用“定义 -> 关注点 -> 方法 -> 示例”的表达方式，让回答更清晰。

### 17.3 尽量结合项目经验
比如说：
- 你测过什么接口
- 你怎么查 MySQL / Redis
- 你怎么做自动化
- 你怎么做并发压测

这样回答会比纯理论更有说服力。

### 17.4 不要把接口测试说得太浅
如果只回答“测状态码、测返回值”，通常会显得理解不够深入。

---

## 18. 小结

接口测试常见面试题，本质上围绕几个核心能力展开：

- 你是否理解接口测试本身
- 你是否知道接口主要测什么
- 你是否具备用例设计和问题分析能力
- 你是否了解自动化、性能、数据校验、安全和幂等性这些进阶内容
- 你是否能把工具使用和项目实践真正结合起来

真正有竞争力的回答，不只是“背概念”，而是能够让面试官感受到你对接口测试的理解是成体系的，并且能落到真实项目实践里。
