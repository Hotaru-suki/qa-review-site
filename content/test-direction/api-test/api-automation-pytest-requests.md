# pytest + requests 接口自动化

## 1. 概述

在接口测试实践中，`pytest + requests` 是非常常见的一套自动化方案，适合用来编写接口测试脚本、组织测试用例、执行批量回归，并与测试报告、数据库校验、持续集成流程结合。

其中：

- `requests` 主要负责发送 HTTP 请求
- `pytest` 主要负责组织用例、执行测试、断言结果和输出测试结论

这套组合的核心价值在于：  
**把原本依赖手工调试的接口测试，逐步沉淀为可重复执行、可持续维护的自动化测试体系。**

---

## 2. 为什么要做接口自动化

当项目中的接口越来越多、版本迭代越来越频繁时，如果每次都依赖手工用 Postman 或 Apifox 逐个回归，会出现很多问题：

- 重复劳动多
- 回归效率低
- 容易漏测
- 测试结果不稳定
- 难以和 Jenkins、CI/CD 集成

因此，接口自动化的核心意义在于：

- 提高回归效率
- 降低重复人工成本
- 提升执行一致性
- 让核心业务链路可以高频验证
- 支撑持续集成和持续交付

而 `pytest + requests` 正是非常适合落地这件事的一套技术组合。

---

## 3. 为什么常用 pytest + requests

### 3.1 requests 足够轻量直接
`requests` 语法清晰，发送 GET、POST、PUT、DELETE 等请求都很方便，适合接口测试场景快速上手。

### 3.2 pytest 组织能力强
`pytest` 不只是“跑测试”，它还支持：

- 用例发现
- 参数化
- fixture
- 前后置处理
- 断言
- 报告插件
- 与 CI 集成

### 3.3 适合扩展
相比单纯使用接口调试工具，代码型自动化更容易接入：

- MySQL 校验
- Redis 校验
- 日志校验
- 配置管理
- 数据驱动
- 测试报告
- Jenkins 流水线

### 3.4 学习成本相对可控
对于已经有一点 Python 基础的测试人员来说，这套方案的学习门槛并不算高，而且很贴近实际项目需求。

---

## 4. 这套方案主要能做什么

### 4.1 批量执行接口测试用例
可以一次性执行多个模块、多组场景、多种环境下的接口用例。

### 4.2 实现自动断言
除了判断状态码，还可以进一步断言：

- 业务码
- 返回字段
- 字段值
- 响应时间
- 数据库结果
- Redis 缓存结果

### 4.3 支持数据驱动
可以让同一套测试逻辑跑多组测试数据，提高复用率。

### 4.4 支持前后置处理
例如：

- 登录获取 token
- 创建前置测试数据
- 清理脏数据
- 初始化数据库状态

### 4.5 支持生成测试报告
通常可结合：

- Allure
- pytest-html

输出更清晰的执行结果和失败信息。

### 4.6 支持持续集成
可以接入 Jenkins、GitHub Actions 等工具，实现自动回归。

---

## 5. 常见适用场景

`pytest + requests` 更适合以下场景：

- 核心接口回归测试
- 高频迭代项目的接口验证
- 需要做数据库 / Redis 联动校验的场景
- 需要做多环境切换的场景
- 需要集成 Jenkins 的项目
- 需要沉淀长期维护测试体系的项目

它尤其适合“核心接口比较稳定，但回归频率很高”的业务模块。

---

## 6. 一个基础的自动化框架通常包含什么

在项目中，基于 `pytest + requests` 的接口自动化框架，通常不会只是一堆零散脚本，而会有基本结构。

### 6.1 配置管理
用于管理：

- 测试环境地址
- 数据库连接信息
- Redis 连接信息
- 默认请求头
- 测试账号
- 超时时间

### 6.2 请求封装
把常用请求方法、公共请求头、日志打印、异常处理等做统一封装，避免每个测试文件重复写。

### 6.3 测试用例层
负责组织具体接口测试场景，例如：

- 登录接口测试
- 状态查询接口测试
- 事件上报接口测试

### 6.4 公共断言层
用于统一校验状态码、业务码、字段、响应时间等内容。

### 6.5 数据校验层
负责查询 MySQL、Redis 等，验证接口处理后的真实数据。

### 6.6 报告输出层
结合 Allure 或其他报告工具，输出测试执行结果。

## 7. 常见设计思路

### 7.1 先从核心接口开始自动化
并不是所有接口都值得第一时间自动化，通常应优先选择：

- 核心业务链路接口
- 高频回归接口
- 稳定且规则明确的接口
- 容易出问题的高风险接口

### 7.2 请求与断言尽量解耦
发送请求和检查结果最好分开组织，这样更方便复用和维护。

### 7.3 环境与数据尽量配置化
不要把 URL、账号、token、数据库地址硬编码在测试代码里，应尽量通过配置文件或环境变量管理。

### 7.4 用例要可重复执行
自动化测试要尽量避免强依赖历史环境状态，否则很容易出现“第一次能跑，第二次就失败”。

### 7.5 尽量形成结果闭环
接口自动化不应只断言返回值，最好还能结合数据库、Redis 等一起验证。

## 8. 常见能力点

### 8.1 参数化测试
使用 `pytest.mark.parametrize` 可以让同一个测试逻辑跑多组输入数据。

适合：
- 参数校验类测试
- 边界值测试
- 错误码覆盖测试

### 8.2 fixture 前后置处理
`fixture` 常用于：

- 初始化 token
- 准备数据库数据
- 清理测试数据
- 构造公共依赖对象

### 8.3 公共请求封装
通常会封装统一的请求入口，处理：

- baseUrl 拼接
- Header 合并
- Token 注入
- 超时处理
- 请求日志打印

### 8.4 公共断言封装
例如封装：

- 状态码断言
- 业务码断言
- 字段存在性断言
- 字段值断言
- 响应时间断言

### 8.5 数据库与缓存校验
可通过 Python 连接 MySQL、Redis，在接口执行后直接核对真实数据状态。

## 9. 与 Postman / Apifox 的区别

`pytest + requests` 和 Postman / Apifox 并不是竞争关系，但定位不同。

### 9.1 Postman / Apifox 更适合
- 手工调试
- 联调验证
- 快速排查问题
- 小规模回归

### 9.2 pytest + requests 更适合
- 大规模自动化回归
- 复杂逻辑校验
- 多环境运行
- 数据库 / Redis 联动验证
- Jenkins 集成
- 长期维护

所以，实际项目中很常见的方式是：

**先用 Postman / Apifox 调通接口，再把高价值场景沉淀到 pytest + requests 自动化中。**

---

## 10. 常见问题

在基于 `pytest + requests` 做接口自动化时，经常会遇到以下问题：

- 脚本能跑，但结构混乱，难以维护
- URL、token、环境信息硬编码
- 只断言状态码，不断言业务结果
- 测试数据依赖严重，导致回归不稳定
- 脚本彼此耦合，执行顺序混乱
- 没有数据库 / Redis 校验，自动化覆盖很浅
- 报告信息不足，失败后定位困难
- 所有接口都自动化，反而维护成本过高

这些问题说明接口自动化不是“写脚本”这么简单，而是需要有一定的工程化思路。

---

## 11. 学习建议

如果要从零开始学习 `pytest + requests` 接口自动化，比较推荐的顺序是：

### 11.1 先学会 requests 基础请求
先掌握如何发送：

- GET 请求
- POST 请求
- 传 query 参数
- 传 JSON body
- 设置 header
- 获取响应结果

### 11.2 再学 pytest 基本组织方式
例如：

- 测试函数命名
- 执行测试
- 使用断言
- 参数化
- fixture

### 11.3 再学公共封装
逐步把请求、断言、配置、数据处理抽出来，减少重复代码。

### 11.4 最后补数据库校验和报告能力
这样才能从“能跑脚本”走向“能维护测试框架”。

---

## 12. 小结

`pytest + requests` 是接口自动化测试中非常经典且实用的一套方案，它既适合初步入门，也足以支撑中小型项目的自动化回归需求。

它的核心价值在于：

- 让接口测试从手工验证走向自动化执行
- 让高频回归场景可以重复、稳定地运行
- 让测试结果更可沉淀、更可维护
- 让接口测试更容易与数据库校验、报告系统和 CI/CD 流程结合

一个真正成熟的 `pytest + requests` 接口自动化体系，不只是“会发请求”，更在于能围绕核心业务链路，建立稳定、清晰、可持续演进的自动化测试能力。
