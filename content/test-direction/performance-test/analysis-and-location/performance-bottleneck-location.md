# 性能瓶颈如何分层定位

## 摘要

性能定位不是盯着一条慢日志反复猜，而是把问题按层拆开，再用指标、日志、监控和 Profiling 逐步缩小范围。真正有效的定位过程，应该能把“感觉很卡”“接口变慢了”落到某个具体环节。

---

## 核心问题

- 为什么性能问题必须按层拆
- 哪些现象更像 CPU、IO、网络或数据库瓶颈
- 如何从现象一步步走到根因

---

## 1. 为什么必须分层定位

因为“慢”只是表象，根因可能完全不同。

例如同样是“接口超时”，可能是：

- 服务本身 CPU 打满
- 下游数据库锁等待
- Redis 抖动
- 网络重传明显
- 上游把无效大流量打了进来

如果不分层，定位就容易停留在“接口慢”这种无效结论上。

---

## 2. 一个稳定的定位顺序

较稳的定位顺序通常是：

1. 先看现象和主要指标
2. 再区分是客户端、网络还是服务端问题
3. 然后用监控和 Profiling 落到具体模块

也就是先回答三个问题：

- 慢在哪里表现出来
- 慢发生在哪一层
- 这一层里是谁在消耗时间或资源

---

## 3. 客户端性能问题怎么拆

如果是客户端卡顿，先区分：

- 常驻性低帧率
- 偶发性 spike
- 加载时卡
- 输入后卡
- 切场景或开 UI 时卡

然后再看证据：

- `stat unit` 里是 Game 高、Draw 高还是 GPU 高
- `stat unitgraph` 是否有明显尖刺
- `stat gpu` 是否有某类渲染开销偏高
- 内存、显存是否接近上限
- 卡顿是否伴随资源加载、GC、对象创建激增

典型判断思路：

- Game 高：优先看逻辑、AI、脚本、物理、对象管理
- GPU 高：优先看渲染、材质、阴影、后处理、特效、分辨率
- Draw 高：优先看渲染线程压力和场景复杂度
- 只在切图或开 UI 卡：优先看资源加载和同步阻塞

---

## 4. 服务端性能问题怎么拆

如果是服务端接口变慢，先区分：

- 是所有接口一起变慢，还是某个接口独慢
- 是延迟上升，还是错误率先升
- 是平均值变差，还是只有 P99 恶化
- 是持续慢，还是只在流量高峰慢

然后继续拆：

- 应用层：线程池、协程、GC、业务逻辑、序列化反序列化
- 依赖层：数据库、缓存、消息队列、第三方服务
- 系统层：CPU、内存、磁盘 IO、网络带宽、连接数

例如：

- CPU 高、热点集中在 JSON 解析，可能是序列化开销过大
- CPU 不高，但数据库锁等待高，可能是事务和锁竞争问题
- 本机资源正常，但下游 RPC 超时明显，问题可能在依赖链路

---

## 5. 网络层问题怎么识别

网络问题常见但容易被误判成“服务慢”。

常见信号包括：

- 延迟抖动明显
- 超时和重试多
- 丢包、重传、连接重建频繁
- 客户端和服务端各自看都不慢，但端到端慢

这时应该结合：

- 客户端埋点时间
- 网关或接入层日志
- 服务端收到请求和返回响应的时间
- 丢包率、重传率、连接建立耗时

如果服务端业务处理只花了 30ms，但客户端感知到 500ms，问题很可能在网络链路或重试机制。

---

## 6. 一个服务端定位示例

现象：

- 压测到 600 QPS 后，P99 从 300ms 跳到 2s
- 错误率升到 3%

排查过程：

1. 先看应用监控，CPU 只有 55%，内存平稳
2. 看数据库监控，连接池接近耗尽，慢 SQL 明显增多
3. 进一步查 SQL，发现某个库存更新语句走了低效执行计划
4. 同时锁等待上升，说明高并发写入时竞争严重

最终结论：

- 主要瓶颈不在应用 CPU，而在数据库执行计划和锁竞争

---

## 7. 一个客户端定位示例

现象：

- 团战时玩家反馈“技能一多就卡”
- 平均 FPS 还可以，但体感明显不好

排查过程：

1. 先看 `stat unitgraph`，确认团战瞬间存在高 spike
2. 看 `stat unit`，发现 GPU 时间明显高于 Game
3. 用 `stat gpu` 或 GPU Profiler 看具体通道，发现特效和后处理耗时高
4. 对比特效开关前后结果，确认问题和粒子数量、屏幕覆盖率有关

最终结论：

- 问题主要在 GPU 渲染负载，不是逻辑线程

---

## 8. 为什么一定要结合监控、日志和调用链

因为单一维度只能看到局部：

- 压测工具只知道请求结果
- 应用日志只知道代码路径
- 系统监控只知道资源状态
- 调用链才知道时间耗在哪一段链路上

只有把这些证据串起来，定位才不会停留在“感觉像数据库慢”这种猜测层面。

---

## 小结

性能定位的关键，是把“慢”拆层，再把证据落到具体环节。先分客户端、网络、服务端，再分线程、模块、依赖、资源，最终才能从现象走到根因。
