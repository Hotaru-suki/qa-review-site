# 瓶颈假设与验证实验怎么做

## 这个知识点在解决什么问题
性能排查如果一上来就凭感觉改配置，很容易越改越乱。瓶颈假设与验证实验解决的是：先提出“可能慢在哪”，再通过对照实验验证，而不是盲目猜。

## 为什么要先有假设
性能瓶颈可能在：
- CPU
- 内存
- 磁盘 IO
- 网络
- 数据库
- 锁竞争

不先收敛范围，就会陷入“大海捞针”。

## 一个简单流程
1. 根据监控提出候选瓶颈
2. 设计最小验证实验
3. 比较前后变化
4. 再决定下一步

## 为什么一定要做对照
如果你同时改了：
- 线程池参数
- SQL 索引
- JVM 配置

最后性能变好了，你其实很难知道到底是哪一个起作用。  
所以验证实验最好一次只改一个关键变量。

## 典型例子
怀疑数据库慢：
- 单独压数据库依赖路径
- 减少查询量或改成 mock 对照
- 看 RT 和吞吐是否明显变化

如果变化很大，说明数据库假设更可能成立。

## 常见误区
- 一发现慢就同时改很多参数
- 没有对照组，导致结论不可信
- 只凭单次结果就下最终判断
- 把“感觉像瓶颈”当成证据，不做验证实验
