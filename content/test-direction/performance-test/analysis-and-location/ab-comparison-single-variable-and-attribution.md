# A/B 对照、单变量实验与归因

## 摘要

性能定位里最危险的做法，是看到问题后直接猜根因。A/B 对照和单变量实验的价值在于：用最少变量变化去验证假设，让“我觉得是这个问题”变成“证据显示更可能是这个问题”。

---

## 核心问题

- 为什么性能归因必须靠实验
- 什么叫单变量实验
- 怎样设计 A/B 对照才能避免结论被噪声污染

---

## 1. 为什么性能问题不能靠猜

因为同一个现象“慢”，可能来自：

- SQL 变慢
- 下游超时
- 线程池竞争
- GC 抖动
- 网络抖动

如果不做实验，就很容易把相关性当成因果。

---

## 2. 什么叫单变量实验

单变量实验就是：

- 一次只改一个关键因素
- 观察结果是否随之稳定变化

例如：

- 只放大数据库连接池
- 只关掉某段日志
- 只替换某个下游 mock

这样才能更有把握地判断哪个因素在起主要作用。

### 为什么一次改很多东西最危险

如果你同时：

- 改 SQL
- 改缓存策略
- 改线程池参数

最后性能变好了，也很难知道真正起决定作用的是哪一项。

---

## 3. 一个典型场景

如果怀疑某条 SQL 是瓶颈，可以做：

- 原版本 A
- 优化 SQL 后版本 B

其他条件尽量保持一致，再比较：

- 吞吐
- P95 / P99
- DB 耗时

如果只有这项变化带来了显著改善，归因才更站得住。

### 一个更像真实定位的例子

如果怀疑下游服务导致长尾，可以做：

- A：真实下游
- B：mock 下游或降低下游延迟

再比较：

- 主服务 RT
- 下游调用耗时
- 错误率

如果替换下游后长尾明显消失，归因就更有把握。

---

## 4. 做实验时要控制什么

除了变量本身，还要尽量控制：

- 流量模型
- 数据量
- 环境配置
- 下游负载

否则你看到的差异可能来自别的噪声。

---

## 5. 常见误区

- 一次改很多东西，再说“效果变好了”
- 环境和流量模型都变了，还拿结果硬比较
- 不设对照组，只看单次实验结果
- 没有重复验证，就直接下根因结论

---

## 小结

性能归因的关键，不是提出多少怀疑，而是用实验把怀疑一层层排掉。A/B 对照和单变量实验，就是把定位从“猜测”变成“可验证判断”的核心方法。
