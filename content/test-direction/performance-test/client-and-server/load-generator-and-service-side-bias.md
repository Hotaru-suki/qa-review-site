# 压测机瓶颈与服务端瓶颈怎么区分

## 这个知识点在解决什么问题

压测里最容易出现的一类误判，是把“压测端先打满”当成“服务端极限”。这会直接导致错误结论，比如明明是发压机 CPU、连接数、网络栈或压测脚本先顶住了，却把结论写成“服务端 QPS 上限只有某个值”。

这个知识点解决的是：

- 怎么判断吞吐上不去到底是谁的锅
- 为什么单机压测很容易测出假的瓶颈
- 怎么用更可信的证据证明服务端真的扛不住了

## 为什么会误判

压测时如果：

- 压测机 CPU 打满
- 网络带宽打满
- 连接数或文件句柄受限
- 客户端线程模型、连接复用或压测脚本写法本身有问题
- 压测工具统计口径和服务端指标没对齐

那你看到的吞吐上不去，不一定是服务端的真实上限。

## 一个典型场景

假设你压一个下单接口，预期服务端至少能扛 3000 QPS，但单台压测机跑到 1200 QPS 就上不去了。

如果这时：

- 压测机 CPU 95% 以上
- 网卡出口带宽接近打满
- 发压线程数继续增加，RT 反而主要涨在客户端
- 服务端 CPU 只有 40%，线程池和连接池也没满

那这更像是压测端先成了瓶颈，而不是服务端极限。

## 怎么判断

### 1. 先看压测端是不是先满了

- 压测机 CPU、负载、上下文切换是否先明显升高
- 网卡吞吐是否先到上限
- 客户端进程是否出现大量超时、排队或连接创建失败
- 压测工具本身是否提示 socket、fd、连接池不足

### 2. 再看服务端有没有同步出现极限信号

- 服务端 CPU、内存、线程池、连接池是否一起逼近上限
- 服务端 RT、队列积压、错误率是否同步恶化
- 应用日志、监控、trace 是否显示请求已经进入服务端但处理不过来

如果服务端指标很轻，但吞吐上不去，结论通常不能直接写成服务端瓶颈。

### 3. 用扩压和对照实验验证

- 把单台压测改成多台发压，看吞吐是否明显上升
- 压测端升级机器规格，看上限是否跟着升
- 改压测模型，比如长连接改连接复用、同步阻塞改更合理并发模型

如果只要增强压测端能力，上限就上去了，原来的瓶颈多半在压测端。

## 一个排查思路

1. 先看压测机 CPU、内存、带宽
2. 再看服务端 CPU、线程池、连接池
3. 再看客户端和服务端的 RT、错误率、请求量是否对得上
4. 然后试着增加压测机数量或更换压测方式
5. 最后再判断瓶颈是真在服务端，还是压测端先失真

如果增加压测机后吞吐明显提升，说明原来更可能是压测端瓶颈。

## 哪些证据最关键

### 压测端证据

- 发压机 CPU、load、上下文切换
- 网卡流量和丢包
- 客户端连接建立耗时、socket 错误、文件句柄使用
- 压测工具侧的发送 QPS、成功率、超时率

### 服务端证据

- 接收请求数、完成请求数、错误率
- 应用线程池、连接池、消息堆积
- 服务端 RT 分位数
- DB、缓存、下游依赖的耗时和错误

### 对齐关系

最关键的不是各看一边，而是要对齐：

- 客户端发出去多少
- 服务端收到多少
- 服务端成功处理多少
- 客户端统计到多少成功 / 超时 / 失败

如果客户端发起量就已经起不来，服务端自然不可能暴露真实上限。

## 为什么单机压测很容易出偏差

- 单机网络出口固定
- 单进程线程模型有限
- 本机时间统计、序列化、日志打印都会吃掉 CPU
- 如果脚本里还有签名、加密、构造复杂 payload，压测端本身就会先烧 CPU

所以测试报告里如果只有“工具压到 X QPS”，但没有交代压测端资源情况，这个结论是不完整的。

## 一个更可靠的验证方法

可以用下面的顺序：

1. 先做小流量基线，确认压测脚本和统计口径正确
2. 再逐步升压，看客户端和服务端哪边先出现拐点
3. 把压测端和服务端指标放在同一时间轴上
4. 做一次多机发压对照
5. 再写出“当前观测下的瓶颈更像在哪一侧”

这样比直接盯一张服务端 CPU 图更可信。

## 常见误区

- 只看服务端监控，不看压测端资源
- 单机压测结果直接当服务端极限
- 压测工具本身参数不合理却不自知
- 压测端和服务端同时有瓶颈时，只盯其中一边
- 压测端统计到超时，就默认是服务端超时
- 不说明压测模型、连接模型和发压机规格，就直接下结论

## 小结

压测结果是否可信，第一步不是看服务端够不够忙，而是先确认压测端有没有先失真。只有把压测端能力、服务端能力、双侧指标和对照实验一起看，才能区分“压测机打不动”和“服务端扛不住”。
