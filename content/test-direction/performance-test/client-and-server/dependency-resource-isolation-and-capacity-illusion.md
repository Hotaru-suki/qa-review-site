# 下游依赖、资源隔离与容量错觉

## 摘要

很多服务端压测结论之所以不可靠，不是主服务逻辑测错了，而是把下游依赖和共享资源的限制误当成了主服务极限。数据库、缓存、消息队列、线程池、连接池、共享网关都可能先成为上限来源。这个知识点就是把“主服务能扛多少”和“整条依赖链能扛多少”拆开。

---

## 核心问题

- 为什么服务端容量经常受下游和共享资源影响
- 什么叫资源隔离，为什么没有隔离就会出现容量错觉
- 测试时怎样证明瓶颈在主服务还是在依赖链

---

## 1. 什么叫容量错觉

假设压测某接口只能跑到 2000 QPS，就直接写“服务上限 2000 QPS”。

但真实情况可能是：

- 主服务 CPU 还很空
- 数据库连接池先满了
- 缓存被共享流量打爆了
- 网关限流先触发了

这时你测到的不是主服务极限，而是整条链路里最弱一环的极限。

---

## 2. 为什么资源隔离重要

如果：

- 测试流量和线上其他流量共用数据库
- 多个接口共用线程池 / 连接池
- 缓存热点互相影响

那压测结果就会混入很多外部噪声。

资源隔离的价值在于：

- 让你更清楚地看到瓶颈到底属于谁

### 没隔离时最容易出现什么误判

- 共享数据库抖了，却以为主服务扛不住
- 共享缓存热点打满，却以为是本接口计算逻辑慢
- 网关或连接池先到阈值，却误写成应用极限

---

## 3. 一个典型场景

例如下单接口压测时：

- 应用 CPU 只有 50%
- 但数据库连接池已满
- 大量请求都在等连接

这时如果直接说“下单服务顶不住”，就不够准确。更准确的说法是：

- 当前链路容量受数据库连接资源限制

### 这类结论为什么更重要

因为“服务端上限 2000 QPS”和“当前链路受 DB 连接池限制在 2000 QPS”是完全不同的结论：

- 前者会推动去优化主服务
- 后者会推动去看下游资源和隔离策略

---

## 4. 测试时怎么拆

更稳的做法通常是：

1. 同时采集主服务和关键下游资源
2. 看拐点出现时哪边先打满
3. 尽量隔离共享流量和共享资源
4. 用不同下游规格做对照实验

如果下游放大后上限明显上升，就说明原瓶颈不在主服务。

### 一个更完整的拆分思路

除了看主服务，还应该同步采：

- 数据库 CPU / 连接 / 锁等待
- 缓存命中率 / RT / 热点 key
- MQ 堆积
- 网关限流和连接数

---

## 5. 常见误区

- 只盯主服务 CPU
- 没有采下游指标就写容量结论
- 把“整链路上限”说成“单服务上限”
- 共享资源没隔离就拿结果做容量规划

---

## 6. 一个更适合测试报告的表达

更好的报告结论通常会写成：

- 当前压测上限是多少
- 这个上限是在什么资源条件下测到的
- 拐点时先满的是哪类资源
- 若放大下游或做资源隔离，上限是否还能继续抬高

---

## 小结

服务端性能测试里最常见的误判之一，就是把依赖链的限制当成主服务本身的限制。只要系统依赖数据库、缓存、消息或共享网关，容量测试就必须把资源隔离和下游瓶颈一起看。
