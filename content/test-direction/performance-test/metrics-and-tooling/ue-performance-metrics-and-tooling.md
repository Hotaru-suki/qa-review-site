# 性能测试关注指标与工具

## 摘要

这一篇不只是罗列工具名，而是把客户端和服务端性能里最常用的指标真正讲清楚，并说明工具在测试流程里分别扮演什么角色。你需要知道一个数字在描述什么风险，也要知道该用什么工具拿到证据。

---

## 核心问题

- FPS、帧时、分位值、QPS、TPS、错误率分别在说什么
- 为什么工具只能帮助采集和定位，不能代替分析
- UE 项目里的客户端性能测试通常怎样落地执行

---

## 1. 先建立一个工具观

工具在性能测试里主要做三件事：

- 采集指标
- 记录时序
- 帮助定位热点

工具不能自动替你回答“为什么慢”“先优化哪里”“这个结果是否达标”。结论仍然来自测试目标、基线和证据链。

---

## 2. 客户端性能里最常见的指标是什么

### FPS

FPS 是每秒渲染多少帧，适合快速判断“现在大概顺不顺”，但它有两个局限：

- 它是结果，不是原因
- 它容易掩盖短时间尖刺

所以 FPS 更像体表温度，不像化验结果。

### 帧时

帧时表示一帧总共花了多少毫秒，是分析客户端性能的主指标。

直觉参考：

- 16.67ms 约等于 60 FPS
- 33.33ms 约等于 30 FPS

如果一段场景平均帧时是 16ms，但偶尔飙到 80ms，玩家依然会明显感到卡顿。

### P95 / P99 帧时

它们用来看长尾卡顿。平均帧时正常，不代表极端帧时正常。

### Game / Draw / GPU

- Game：逻辑线程时间，偏 CPU 逻辑
- Draw / Render：渲染线程时间
- GPU：显卡完成渲染的时间

它们的高低关系，会直接影响你先查逻辑、渲染还是 GPU。

### 内存 / 显存

决定系统能否持续运行，也影响切图、加载、贴图流送和后台切回等场景。

### 加载时间 / 输入延迟

这两个指标直接面向体感：

- 加载时间影响等待成本
- 输入延迟影响操控手感

---

## 3. 服务端性能里最常见的指标是什么

### QPS

QPS 指每秒请求数，表示系统每秒处理了多少次请求。

例如：

- 网关总流量 2000 QPS
- 角色查询接口稳定 600 QPS

它适合描述接口处理量，但不能单独代表体验和业务成功。

### TPS

TPS 指每秒事务数，更偏完整业务动作完成量。

例如：

- 每秒成功创建 120 个订单
- 每秒完成 80 次支付事务

这里一定要注意事务口径。不同团队对 TPS 的定义可能不同，报告里必须说明。

### 响应时间和分位值

平均值适合看总体水平，P95 / P99 适合看长尾风险。

一个常见误区是只看平均 100ms，不看 P99 已经飙到 2s。

### 错误率 / 超时率

它们在告诉你系统有没有失控。吞吐高但错误率升高，不等于性能好。

### 资源指标

包括：

- CPU
- 内存
- 网络
- 线程池
- 数据库连接池
- 缓存命中率

资源指标解释的是“为什么会变慢”。

## 4. UE 客户端性能测试具体怎么做

以 UE 项目为例，一个实用流程通常是：

### 4.1 固定测试条件

- 地图或场景
- 摄像机路径或角色行为
- 分辨率
- 画质档位
- 是否开启 VSync、动态分辨率、Lumen、Nanite
- 测试构建类型
- 测试机型和驱动版本

### 4.2 快速初筛

先用：

```text
stat fps
stat unit
stat unitgraph
```

目的不是直接下结论，而是先判断方向：

- 是整体常驻慢，还是偶发 spike
- 是 Game 高，还是 GPU 高

### 4.3 深度定位

根据初筛结果再补工具：

- `stat gpu`：看 GPU 渲染阶段耗时
- Unreal Insights：看线程时间线、热点函数、spike 前后时序
- Memory Insights / LLM：看内存增长、分配释放和对象类别

### 4.4 长时间稳定性验证

不能只测几分钟，要加入：

- 长时间挂机
- 反复切图
- 高频开关 UI
- 连续战斗
- 热更或资源刷新后继续跑

这样才能看出内存上涨、资源释放不干净或热衰减问题。

---

## 5. 一个 UE 客户端测试场景示例

假设你要测一段团战场景：

1. 固定地图、角色数量、技能释放节奏
2. 记录普通阶段和团战爆发阶段的数据
3. 采集平均帧时、P95、P99、Game / Draw / GPU
4. 如果团战明显掉帧，再用 `stat gpu` 或 Unreal Insights 深查
5. 对卡顿时刻做截图、录像或 trace 留证

最后你应该输出的不是“团战比较卡”，而是：

- 团战阶段 GPU 时间从 12ms 升到 28ms
- P99 帧时出现 90ms 尖刺
- 问题主要与高覆盖率特效和后处理叠加有关

这才叫能落地的性能结论。

## 6. 工具分别适合解决什么问题

### `stat fps`

适合快速看 FPS 水平，但只能做表层观察。

### `stat unit`

适合快速判断 Frame、Game、Draw、GPU 哪一项更可疑，是客户端初筛最常用的入口。

### `stat unitgraph`

适合观察波动和 spike，比单纯看 FPS 更能反映体感问题。

### `stat gpu`

适合确认 GPU 侧是否是主要瓶颈，以及渲染负载大致落在哪类通道。

### Unreal Insights

适合深度分析时序和热点，是把“卡顿”落到具体线程或模块的重要工具。

### Memory Insights / LLM

适合分析长期运行后的内存增长、对象分配和释放不平衡问题。

---

## 7. UE 指标和系统硬件指标有什么区别

这是一个特别容易混淆的点。

### UE 指标

UE 指标看的是引擎内部某一帧在不同线程和阶段上花了多少时间。

例如：

- `Game = 18ms`
- `GPU = 22ms`

这表示这一帧里逻辑线程和 GPU 分别消耗了多少时间。

### 系统硬件指标

系统指标看的是整个设备资源状态，例如：

- CPU 总体使用率
- GPU 占用率
- 内存占用
- 温度
- 频率

两者不能互相替代。一个看的是帧级耗时，一个看的是设备级资源状态。

---

## 8. 配合系统监控时应该看什么

### CPU

- 总体占用
- 单核分布
- 当前频率
- 温度
- 是否发生热降频

意义是判断逻辑线程是否受限，以及温度是否放大了问题。

### GPU

- GPU 占用率
- 温度
- 频率
- 显存

意义是辅助判断渲染压力是否真的落在显卡侧，以及是否存在显存紧张或热衰减。

### 内存

- 常驻内存
- 峰值内存
- 场景切换前后变化

意义是判断是否存在资源累计和长时间运行退化。

### 磁盘和网络

- IO 抖动
- 带宽
- 丢包和重传

意义是判断加载卡顿或联网场景卡顿是否和外部链路有关。

---

## 小结

指标和工具的价值，不在于“看到了多少数字”，而在于你能否借助这些数字解释系统行为。客户端要把流畅度和帧时讲清楚，服务端要把 QPS、TPS、延迟、错误率和资源压力连起来，工具只是帮助你把这条证据链补完整。

### 8.2 GPU 相关
- GPU 利用率
- 显存占用
- 当前频率
- 温度
- 功耗

意义：

- 如果 UE 中 `GPU` 时间高，同时 GPU 利用率也高，通常是 GPU bound
- 显存接近打满时，可能伴随贴图流送抖动和突发掉帧

---

### 8.3 内存相关
- 系统内存占用
- 峰值内存
- 长时间运行后的趋势

意义：

- 判断是否存在内存泄漏、缓存膨胀、对象池不缩等问题

---

## 9. 如何组合看指标

### 场景 1：Game 高
应该同时看：

- 单核 CPU 占用
- CPU 温度
- CPU 频率
- Unreal Insights 中的 Game Thread 热点

---

### 场景 2：GPU 高
应该同时看：

- GPU 利用率
- 显存占用
- GPU 温度
- GPU 频率
- `stat gpu` / GPU profiler

---

### 场景 3：长时间运行后越来越卡
应该同时看：

- Memory Insights / LLM
- 系统内存
- 显存
- 温度
- 频率
- 是否发生场景切换后资源回不去

---

## 10. 表达骨架

### UE5 性能测试怎么做
> 在 UE5 项目里，我会先用 `stat fps`、`stat unit`、`stat unitgraph` 做快速实时判断，区分是整体帧率问题、CPU 线程瓶颈还是 GPU 瓶颈，并观察是否存在明显 spike。更完整的方式是固定测试场景、路线、分辨率、画质和构建类型，建立可重复的性能基线；然后对 CPU 问题用 Unreal Insights 抓 trace，看 Game Thread 热点，对 GPU 问题使用 `stat gpu` 或 GPU Profiler 分析渲染阶段耗时；最后结合长时间运行、反复切图、UI 高频开关和高压战斗场景做回归测试，并记录平均 FPS、帧时分布、Game/Draw/GPU 时间和内存趋势。

### 主要看哪些指标
> 客户端主要关注平均 FPS、帧时、P95/P99、卡顿 spike、Game/Draw/GPU 时间、内存与显存趋势以及加载时长；服务端则主要关注 QPS、TPS、并发数、平均响应时间、P95/P99 延迟、错误率和 CPU/内存/带宽等资源指标。

### UE 工具和系统监控分别看什么
> UE 工具主要看引擎内部的帧级、线程级和资源级耗时，例如 Frame、Game、Draw、GPU、trace 和内存分配；系统监控则看硬件资源使用状态，如 CPU/GPU 占用、温度、频率、显存和功耗，两者结合后才能更完整地判断瓶颈是在逻辑、渲染、内存还是设备散热。
