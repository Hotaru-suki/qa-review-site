# 性能监控、埋点与证据链怎么建

## 这个知识点在解决什么问题
性能测试不是把压测工具跑起来就结束了。没有监控和证据链，测试只能得到“慢了”这种结论，无法回答“慢在哪里、为什么慢、是客户端还是服务端慢、是系统瓶颈还是测试方法有问题”。

## 什么叫性能证据链
```text
压测流量
  -> 接口耗时/成功率
  -> 应用内部指标
  -> 系统资源指标
  -> 下游依赖指标
  -> 日志/trace 佐证
```

能把这条链路连起来，结论才有说服力。

## 常见监控对象
### 接口层
- QPS / TPS
- 平均耗时、P95、P99
- 错误率、超时率

### 应用层
- 线程池活跃数、队列长度
- GC 次数与停顿
- 连接池使用率
- 缓存命中率

### 主机层
- CPU、内存、磁盘 IO、网络带宽

### 依赖层
- 数据库慢查询、锁等待
- Redis 命中率和耗时
- MQ 堆积
- 第三方接口耗时

## 客户端性能也需要证据
以客户端性能测试为例，不能只写“卡顿”：
- 要看帧率、帧时间分布
- 要看 CPU/GPU 占用
- 要看内存增长与回收
- 要看加载阶段和交互阶段是否不同

## 埋点和日志的关系
- 埋点适合做结构化统计，回答“多少、多久、发生在什么阶段”
- 日志适合看单次请求细节，回答“具体这次为什么慢”
- trace 适合串联分布式调用链，回答“时间耗在哪一跳”

三者结合起来，才能从现象走到根因。

## 实际执行怎么做
1. 测前确认所有关键指标可见
2. 给压测批次打上版本、环境、流量模型标记
3. 测中实时看长尾和错误率，不只看平均值
4. 测后把时间段和日志、trace、监控图对齐

## 一个具体场景怎么落
### 接口压测场景
假设你要压“下单接口”：
- 压测工具负责产生并发请求
- 应用监控负责看接口 RT、错误率、线程池、连接池
- 数据库监控负责看慢 SQL、锁等待、活跃连接
- 业务日志负责根据订单号定位失败链路
- trace 负责把网关、订单服务、库存服务、支付服务串起来

如果压测结论是“P99 飙高”，但没有这几层证据，你根本无法判断到底是：
- 订单服务逻辑慢
- 库存服务偶发超时
- 数据库锁冲突
- 客户端重试把流量放大

### 客户端性能场景
假设你要测“进入战斗场景时的卡顿”：
- 客户端埋点记录进入场景、开始加载、首帧展示、可交互时间
- 运行时采集 CPU/GPU、内存、帧时间分布
- 资源系统记录纹理、模型、特效加载耗时
- 服务端只需要提供登录或房间匹配接口耗时

这时证据链的重点就不是 QPS，而是“用户看到卡顿的那几秒，CPU/GPU/资源加载/网络请求分别在做什么”。

## 指标应该怎样对应到问题
- `QPS/TPS` 高但 `P99` 差：说明系统能处理总量，但长尾请求明显异常
- `错误率` 升高且 `CPU` 很低：更可能是下游、配置或线程池阻塞，不一定是算力瓶颈
- `CPU` 高但 `QPS` 没上去：可能有锁竞争、单线程热点或同步等待
- `客户端总耗时` 高但服务端 RT 正常：问题可能在网络、渲染、资源加载或客户端逻辑

## 日志、埋点、trace 各自该记录什么
### 日志
- 请求参数摘要
- 业务键，如订单号、用户 ID、traceId
- 关键阶段耗时
- 错误堆栈和失败原因

### 埋点
- 阶段开始和结束时间
- 用户操作和页面状态变化
- 聚合后需要统计的结构化字段

### trace
- 跨服务调用链
- 每一跳的耗时
- 哪一跳出现重试、超时、异常返回

## 测试报告里怎么写才有说服力
不要只写“QPS 3000，平均响应 120ms”。至少要写成：
- 在 3000 QPS 下，成功率 99.95%，P95 210ms，P99 480ms
- 订单服务 CPU 65%，数据库连接池峰值 72%，库存服务出现 3 次 1s 以上长尾
- trace 显示长尾主要集中在库存扣减链路
- 建议优先检查库存服务锁竞争和热点商品行更新

## 常见误区
- 只保存压测报告，不保存监控截图和时间段
- 客户端只看 FPS，不看 CPU/GPU/内存
- 服务端只看 QPS，不看下游和线程池
- 测后才发现没有业务键、traceId 或时间对齐信息
- 埋点只报总耗时，不拆阶段，导致后续无法定位
