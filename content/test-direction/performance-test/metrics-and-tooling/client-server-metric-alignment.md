# 客户端和服务端指标怎么对齐看

## 这个知识点在解决什么问题
性能测试里经常出现“客户端说卡，服务端说没问题”或者“服务端 QPS 很稳，用户却感觉慢”的情况。原因通常不是谁看错了，而是双方看的指标不在一个时间轴、同一个请求范围或同一个业务阶段上。

## 为什么会对不上
- 客户端关注的是用户可感知耗时
- 服务端关注的是接口处理耗时
- 两者中间还隔着网络、重试、序列化、渲染和资源加载

所以服务端 80ms 不代表用户只等了 80ms。

## 怎么对齐
### 先统一业务动作
例如“点击领取奖励到页面展示奖励结果”是一个完整动作，而不是单独一个 HTTP 请求。

### 再拆阶段
```text
用户点击
  -> 客户端发请求
  -> 网络传输
  -> 服务端处理
  -> 响应返回
  -> 客户端解析与渲染
```

### 最后映射指标
- 客户端：点击响应时延、页面可交互时间、帧率抖动
- 服务端：接口耗时、QPS、错误率、下游耗时

## 一个完整例子
以“点击领取奖励后界面卡顿”举例：

### 客户端看到的现象
- 点击按钮后 1.8 秒才看到结果
- 期间动画掉帧

### 服务端看到的现象
- 领奖接口平均 120ms
- P99 380ms
- 错误率接近 0

这时不能直接说“服务端没问题”，因为客户端的 1.8 秒可能包含：
- 按钮点击后的本地动画逻辑
- 网络传输
- 服务端处理
- 结果解析
- 奖励弹窗资源加载和渲染

真正对齐后，才能知道 1.8 秒里哪一段最贵。

## 实际怎么采集
### 客户端侧
- 点击时间
- 请求发起时间
- 响应收到时间
- UI 可交互时间
- 首帧或关键动画耗时

### 服务端侧
- 网关接收时间
- 应用处理开始/结束时间
- 下游 RPC/SQL 时间
- traceId 和业务键

把这些阶段对齐后，才能把“用户体感慢”拆成可定位的问题。

## 典型场景
### 页面打开慢
可能不是服务端主接口慢，而是：
- 静态资源多
- 客户端首帧渲染慢
- 串行请求太多

### 抽奖按钮卡顿
可能是：
- 接口实际很快，但动画、渲染或资源加载堵住了
- 也可能是服务端长尾请求拉高了等待时间

## 实际执行时的证据链
- 用业务键或 traceId 对齐一次操作
- 对齐客户端埋点时间和服务端日志时间
- 同时看长尾，不只看平均值

还要注意两件事：
- 客户端和服务端机器时间是否校准
- 重试、预请求、批量请求是否会让一次用户动作对应多次服务端调用

## 最终想得到什么结论
不是简单判断“谁背锅”，而是回答：
- 这次用户体感慢主要耗在哪一段
- 是客户端渲染、网络传输还是服务端处理
- 哪一段最值得优先优化

## 常见误区
- 用客户端总耗时直接指责服务端
- 用服务端接口耗时直接否定客户端问题
- 两边没有统一业务动作定义
- 客户端埋点只上报总耗时，不拆阶段，导致无法和服务端指标对齐
