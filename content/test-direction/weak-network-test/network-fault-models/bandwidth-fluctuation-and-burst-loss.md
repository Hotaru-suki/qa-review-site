# 带宽波动与突发丢包场景

## 这个知识点在解决什么问题
弱网不只有固定延迟和固定丢包。真实移动网络里更常见的是带宽忽高忽低、瞬时突发丢包、短时间抖动集中过来。带宽波动和突发丢包场景，解决的是系统在“网络不稳定但又没彻底断”的情况下是否还能保持合理体验。

## 和固定弱网参数有什么区别
- 固定限速更像平稳差网络
- 带宽波动更像地铁、电梯、切小区时的真实场景
- 突发丢包更像某一小段时间突然收不到包

所以很多线上“偶发卡死”在实验室固定 10% 丢包里未必能复现。

## 典型影响
- 资源下载速度忽快忽慢
- 长连接心跳偶发超时
- 视频流、语音流短时中断
- 请求重试次数突然上升

## 一个更真实的时序
很多线上弱网不是“持续 200ms 延迟、持续 10% 丢包”，而是像这样：

```text
0s - 10s    网络正常
10s - 15s   带宽骤降
15s - 18s   突发丢包
18s - 25s   网络逐步恢复
```

这类模型更容易暴露：
- 下载进度卡住但 UI 没有提示
- 请求超时后同时触发多路重试
- 长连接短时断开后恢复逻辑不完整

## 怎么测
- 先给一个基础弱网模型
- 再叠加短时带宽下降
- 再插入几秒钟的高丢包突发段

观察重点：
- UI 是否持续可用
- 请求是否出现雪崩重试
- 长连接是否异常断开
- 恢复后是否自动回稳

## 测试场景怎么拆
### 下载型场景
- 安装包更新
- 大资源包加载
- 图片、音视频拉流

重点看：
- 进度条是否可信
- 是否支持断点续拉
- 恢复后是否从上次进度继续

### 交互型场景
- 点击领奖
- 发消息
- 提交表单

重点看：
- 用户是否会重复点击
- 客户端是否提前把操作标成成功
- 恢复后是否出现重复提交

### 长连接场景
- IM、对战、语音房间、心跳保活

重点看：
- 心跳超时阈值是否过于敏感
- 短时波动是否被误判成彻底掉线
- 恢复后房间状态、成员状态是否正确

## 证据链该怎么看
- 客户端埋点：重试次数、恢复耗时、操作耗时
- 服务端日志：是否收到重复请求、是否产生重复副作用
- 网络模拟参数：哪一秒开始降速、哪一段丢包
- 长连接日志：断开原因、重连次数、恢复结果

## 常见误判
- 看到请求最终成功，就忽略中间体验已经非常差
- 只看单次请求，不看波动期间是否产生了大量重试流量
- 只测下载，不测交互和长连接

## 常见误区
- 只测恒定弱网，不测波动型弱网
- 只看请求是否成功，不看恢复后是否残留中间态
- 波动持续时间设计太短，系统还没暴露问题就结束了
- 弱网参数虽然复杂，但没有和真实用户场景对应起来
