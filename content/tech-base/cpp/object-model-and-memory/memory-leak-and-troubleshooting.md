# 内存泄露 / 未释放及排查
## 摘要

内存泄露 / 未释放及排查 关注 C++ 中对象生命周期、资源管理和内存行为，是理解代码执行结果和排查问题的关键基础。

---

## 核心问题

- 理解对象在创建、拷贝、销毁过程中的行为
- 掌握常见内存问题的成因
- 把知识点和排查思路关联起来

---


## 1. 基本定义

### 1.1 内存未释放
指的是：

> 本该释放的内存当前还没有释放

但未必已经构成严格意义上的泄漏。

例如：

```cpp
int* p = new int(10);
// 暂时还没 delete
```

如果后续还能：

```cpp
delete p;
```

那它只是“当前未释放”。

---

### 1.2 内存泄漏
更严格的定义是：

> 动态申请的内存仍然占着，但程序已经失去对它的有效访问路径，后续无法再正常释放

例如：

```cpp
int* p = new int(10);
p = nullptr;
```

这里原来那块内存还在，但地址丢了，再也无法 `delete`。

---

## 2. 程序关闭后内存会怎样

这是一个常见误区。

### 正确结论
- 程序运行期间发生的内存泄漏会一直占着内存
- 但**进程结束后，操作系统会回收该进程占用的内存**

所以：

- 泄漏不会永久占住电脑 RAM
- 关闭程序后，系统会兜底回收
- 删除程序文件和内存泄漏不是一回事

### 但为什么泄漏仍然严重
因为很多程序并不是短时间运行，而是：

- 游戏客户端
- 浏览器
- 编辑器
- 后台服务
- 服务器

这类程序长时间运行时，泄漏会不断累积，导致：

- 占用越来越大
- 性能下降
- 最终崩溃或被系统杀掉

---

## 3. 造成内存泄漏 / 未释放的常见原因

### 3.1 申请后忘记释放

```cpp
void func() {
    int* p = new int(10);
}
```

函数结束后：

- 指针变量没了
- 堆内存没删
- 形成泄漏

---

### 3.2 指针被覆盖，导致原地址丢失

```cpp
int* p = new int(10);
p = new int(20);
```

第一块内存地址丢失，无法释放。

---

### 3.3 提前 return / 分支遗漏

```cpp
void func(bool flag) {
    int* p = new int(10);
    if (flag) return;
    delete p;
}
```

如果提前返回，释放语句不会执行。

---

### 3.4 异常导致释放逻辑跳过

```cpp
void func() {
    int* p = new int(10);
    throw 1;
    delete p;
}
```

异常抛出后，`delete` 根本执行不到。

---

### 3.5 所有权不清晰

多个模块都拿着裸指针，但：

- 没人确定谁负责释放
- 大家都以为别人会释放

最终容易造成泄漏；反过来也可能重复释放。

---

### 3.6 类自己管理资源，但拷贝/赋值写错

例如：

```cpp
A& operator=(const A& other) {
    p = new int(*other.p);  // 原来的 p 丢了
    return *this;
}
```

没有先释放旧资源，就会造成泄漏。

---

### 3.7 new/delete、malloc/free 混用

```cpp
int* p = new int;
free(p);   // 错
```

```cpp
int* p = (int*)malloc(sizeof(int));
delete p;  // 错
```

这类情况属于未定义行为，可能导致：

- 没释放
- 堆损坏
- 崩溃

---

### 3.8 shared_ptr 循环引用

```cpp
class A { public: shared_ptr<B> pb; };
class B { public: shared_ptr<A> pa; };
```

A 和 B 互相持有，引用计数永远不归零，逻辑上形成泄漏。

解决方式：

- 其中一边改为 `weak_ptr`

---

## 4. 如何避免内存泄漏

### 4.1 减少裸 new/delete
优先使用：

- 局部对象
- 标准容器
- 智能指针

---

### 4.2 使用 RAII
资源跟对象生命周期绑定：

- 构造时拿资源
- 析构时释放资源

例如：

```cpp
auto p = std::make_unique<int>(10);
```

---

### 4.3 所有权设计清晰
必须明确：

- 谁创建
- 谁拥有
- 谁释放

---

### 4.4 自定义资源类时遵守 Rule of Three / Five
如果类自己管理堆资源，要同步考虑：

- 析构
- 拷贝
- 赋值
- 移动

---

### 4.5 shared_ptr 场景注意 weak_ptr
涉及双向关系时，不要双方都用 `shared_ptr`。

---

## 5. 游戏运行时内存占用越来越大，底层原因是什么

结论是：

> 不一定全是严格意义上的内存泄漏，但本质都和“申请、持有、释放失衡”有关

常见来源包括：

### 5.1 真实的内存泄漏
- 对象创建后没销毁
- 资源句柄丢失
- 脚本层和原生层绑定关系断裂

### 5.2 缓存策略失控
- 纹理
- 音频
- UI 图集
- 预制体
- Shader 变体

缓存只进不出，会导致内存持续上涨。

### 5.3 对象池 / 内存池只扩不缩
高峰场景撑大池子后，空闲对象仍保留，内存长期维持高位。

### 5.4 引用链没断干净
例如：

- 回调没解绑
- 闭包抓住对象
- shared_ptr 循环引用
- 场景切换后旧资源仍被某个系统持有

### 5.5 内存碎片化
频繁分配/释放不同大小内存块，会导致碎片化，表现为：

- 总占用偏高
- 大块连续空间不足
- 分配效率下降

### 5.6 统计口径混淆
有时用户口中的“内存变大”混着：

- 系统内存
- 显存
- 引擎缓存
- 驱动缓存

不能一概称为泄漏。

---

## 6. 排查思路

### 6.1 先做稳定复现
设计高风险路径：

- 反复切场景
- 高频开关 UI
- 长时间挂机
- 高压战斗循环
- 热更新后反复进入功能页

---

### 6.2 看增长曲线
判断是：

- 线性涨
- 阶梯涨
- 峰值后不回
- 某操作瞬时突增

---

### 6.3 对比关键对象和资源数量
例如：

- UI 节点数
- 特效实例数
- 纹理数
- Actor 数
- 脚本对象数
- 音频实例数

---

### 6.4 做快照对比
查看：

- 哪类对象在增长
- 哪些资源引用链还没断
- 哪些对象本应释放却没释放

---

### 6.5 区分泄漏、缓存、池化、碎片
不要一看到占用上涨就直接下结论说“泄漏”。

---

## 7. 表达骨架

### 定义
> 内存泄漏的本质是动态申请的内存失去了有效管理，导致在程序运行期间无法被正常释放。

### 常见原因
> 常见原因包括：申请后忘记释放；指针被覆盖导致原地址丢失；提前 return、异常或分支遗漏使释放逻辑未执行；多个模块共享裸指针但所有权不清晰；类中手动管理资源时析构、拷贝和赋值实现不当；以及 shared_ptr 循环引用。

### 预防手段
> 预防上应尽量采用 RAII、标准容器和智能指针，减少裸 new/delete；在共享场景中谨慎使用 shared_ptr，并在需要时用 weak_ptr 打破循环引用。

### 游戏场景补充
> 游戏运行时内存占用越来越大不一定都是严格意义上的内存泄漏，还可能来自缓存滞留、对象池膨胀、引用链未断、碎片化和统计口径差异，排查时需要结合复现路径、内存快照和对象计数进行分类分析。
