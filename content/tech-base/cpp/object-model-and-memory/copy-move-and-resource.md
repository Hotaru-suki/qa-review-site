# 拷贝、移动与资源所有权

## 摘要

C++ 里一旦对象开始持有资源，拷贝和移动就不再只是“复制几个字段”。它们决定的是资源到底归谁管理、对象销毁时会不会重复释放、性能会不会因为不必要的复制而急剧下降。很多悬空指针、双重释放、无意义深拷贝问题，本质上都和所有权设计不清有关。

---

## 核心问题

- 为什么资源型对象最怕默认拷贝语义
- 移动语义到底解决了什么问题
- 什么情况下该拷贝，什么情况下只该移动或禁止复制

---

## 1. 为什么这件事在 C++ 里特别重要

普通值对象的拷贝通常只是值复制，例如 `int`、`double` 或只保存简单字段的结构体，复制成本和语义都比较直观。

但资源型对象不一样。它们内部可能持有：

- 堆内存
- 文件句柄
- socket
- 互斥锁
- 数据库连接
- GPU 或系统资源句柄

这时“拷贝一个对象”其实是在处理“资源归谁管”。如果语义没设计好，问题就会从“复制对象”直接演变成：

- 两个对象指向同一块资源，却都认为自己该释放
- 对象被移动后还继续被使用
- 一次本可转移的资源被深拷贝多次，性能急剧下降

所以拷贝和移动表面上是语言特性，底层其实在解决所有权问题。

---

## 2. 默认拷贝为什么经常有风险

如果一个类内部直接保存裸指针，而你没有显式实现拷贝行为，编译器生成的默认拷贝通常只是“按成员逐个复制”。

例如：

```cpp
class Buffer {
public:
    Buffer(size_t n) : data(new int[n]), size(n) {}
    ~Buffer() { delete[] data; }

private:
    int* data;
    size_t size;
};
```

如果这样写：

```cpp
Buffer a(10);
Buffer b = a;
```

那么 `a.data` 和 `b.data` 很可能会指向同一块内存。最后两个对象析构时都执行 `delete[]`，就会产生双重释放。

这就是为什么资源型对象不能简单依赖默认拷贝。

---

## 3. 深拷贝、浅拷贝和所有权分别是什么意思

### 浅拷贝

浅拷贝只复制指针或句柄本身，不复制底层资源。

特点是：

- 速度快
- 语义危险
- 容易造成共享但未声明共享关系

### 深拷贝

深拷贝会重新创建一份独立资源，再把内容复制过去。

特点是：

- 两个对象互不影响
- 释放安全
- 可能成本很高

### 所有权

所有权描述的是“谁有责任在生命周期结束时释放资源”。

只要所有权不清，拷贝和移动就容易出问题。C++ 中很多设计选择，本质上都在回答：

- 资源是否允许共享
- 共享时怎么管理生命周期
- 不共享时是否允许转移

---

## 4. 移动语义到底解决了什么

移动语义的核心不是“更快一点”，而是允许把资源所有权从一个对象安全转移到另一个对象，而不是重新复制底层资源。

典型场景：

- 函数返回大型对象
- 容器扩容时搬运元素
- 临时对象传递给新对象
- 独占资源需要交接给主人

例如：

```cpp
class Buffer {
public:
    Buffer(size_t n) : data(new int[n]), size(n) {}

    Buffer(Buffer&& other) noexcept
        : data(other.data), size(other.size) {
        other.data = nullptr;
        other.size = 0;
    }

    ~Buffer() { delete[] data; }

private:

    int* data;
    size_t size;
};
```

这里移动构造做的事不是复制数组内容，而是把指针接管过来，并把旧对象置于“可析构但不再拥有资源”的状态。

这正是移动语义的本质：转移所有权。

---

## 5. 什么时候应该支持拷贝，什么时候只该支持移动

这取决于资源语义。

### 应该支持拷贝

当资源本身天然允许复制，而且复制后两个对象各自独立时，可以实现深拷贝。

例如：

- 自定义字符串缓冲区
- 图片或数据块对象

### 应该只支持移动

当资源具有独占性，复制没有意义或代价过高时，应只允许移动。

例如：

- `std::unique_ptr`
- 文件句柄封装
- socket 连接对象
- 锁守卫对象

### 应该显式禁止

如果资源既不能合理复制，也不应被随意转移，就要显式删除对应操作。

例如：

```cpp
class NonCopyableResource {
public:
    NonCopyableResource(const NonCopyableResource&) = delete;
    NonCopyableResource& operator=(const NonCopyableResource&) = delete;
};
```

禁止本身也是设计的一部分。

## 6. 这和 Rule of Three / Five 有什么关系

如果类需要自己管理资源，通常就不能只写一个析构函数然后放任其他默认行为。

传统上有 Rule of Three：

- 析构函数
- 拷贝构造
- 拷贝赋值

因为只要你自定义了资源释放逻辑，就往往也要定义资源复制逻辑。

C++11 之后又扩展成 Rule of Five：

- 析构函数
- 拷贝构造
- 拷贝赋值
- 移动构造
- 移动赋值

意思不是每个类都必须手写五个函数，而是如果你在管理资源，就必须明确这些语义，而不能默认假设编译器生成的一定符合你的所有权设计。

## 7. 一个真实工程场景

假设你写了一个日志文件封装类，内部持有文件句柄。

如果你允许默认拷贝：

- 两个对象可能同时认为自己拥有同一个文件句柄
- 其中一个析构后，另一个继续写文件就会失败
- 问题表现出来可能不是立即崩溃，而是“偶发写日志失败”

更合理的设计通常是：

- 禁止拷贝
- 允许移动
- 用 RAII 在析构时自动关闭句柄

这时候类的语义就很清楚：资源只能有一个真正的拥有者，但可以安全地交接。

---

## 8. 常见错误写法

- 内部持有裸指针，却依赖默认拷贝
- 写了移动构造，却忘记把源对象置空
- 资源独占对象既允许拷贝又允许移动，语义混乱
- 不需要手写资源管理时仍然手写，反而增加错误概率

现代 C++ 更推荐优先使用标准库类型表达所有权，例如：

- `std::unique_ptr`
- `std::shared_ptr`
- `std::vector`
- `std::string`

能用现成正确语义的类型时，不要为了“练习”去自己管理裸资源。

---

## 小结

拷贝和移动的本质，不是语法技巧，而是资源所有权设计。只要对象持有真实资源，你就必须明确回答三个问题：资源是否能复制、谁负责释放、能否安全转移。把这三个问题想清楚，很多 C++ 生命周期和内存问题就会自然收敛。
