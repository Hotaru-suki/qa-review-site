# 对象生命周期管理

## 摘要

对象生命周期管理关注的是：一个对象什么时候被创建、什么时候被销毁、资源什么时候被拿到、什么时候被释放、在拷贝和异常过程中所有权怎么变化。如果这条线不清楚，C++ 里的拷贝构造、析构、内存泄漏、悬空指针、重复释放这些问题就会全部混在一起。

---

## 核心问题

- 生命周期和作用域为什么不是一回事
- 对象创建、拷贝、移动、销毁时到底发生了什么
- 为什么几乎所有资源管理问题最后都会回到生命周期

---

## 1. 先分清作用域和生命周期

这两个词经常一起出现，但不是同一个概念。

### 作用域

名字在哪些位置可以被访问。

### 生命周期

对象从什么时候开始存在，到什么时候结束。

例如：

```cpp
void func() {
    int n = 0;
}
```

这里 `n` 的作用域和生命周期都在这个代码块里。  
但很多对象场景下，两者并不完全重合。

例如局部 `static`：

```cpp
void counter() {
    static int n = 0;
    ++n;
}
```

这里 `n` 的作用域在函数内部，但生命周期从第一次执行到定义位置开始，一直到程序结束。

理解生命周期，重点不是“这个名字在哪能用”，而是“这份对象和资源什么时候真的活着”。

---

## 2. 不同对象的生命周期差异在哪里

### 自动对象

函数内普通局部变量，进入作用域时创建，离开作用域时自动销毁。

这是最常见、也最稳定的一类对象管理方式。

### 全局对象

程序启动时构造，程序结束时销毁。

它生命周期很长，但也容易引出：

- 初始化顺序问题
- 全局依赖问题

### static 局部对象

第一次执行到定义位置时初始化，直到程序结束才销毁。

它的作用域在函数里，但生命周期远长于普通局部变量。

### 动态对象

通过 `new` 创建，生命周期不再由作用域自动管理，而取决于什么时候 `delete`。

这也是 C++ 内存问题最容易开始出现的地方。

---

## 3. 创建对象时到底发生了什么

对一个类对象来说，创建并不只是“拿到一块内存”，还包括：

1. 分配对象所需内存
2. 调用构造函数初始化成员
3. 让对象进入可用状态

例如：

```cpp
struct FileHolder {
    FILE* fp;

    FileHolder(const char* path) {
        fp = fopen(path, "r");
    }

    ~FileHolder() {
        if (fp) fclose(fp);
    }
};
```

这里对象一创建，就不仅仅是“一个结构体存在了”，而是：

- 打开文件
- 持有文件句柄

这说明对象生命周期和资源生命周期已经绑定在一起。

---

## 4. 为什么 `new` 和 `malloc` 不能混着理解

它们都可能申请内存，但语义完全不同。

### `new`

- 分配内存
- 按类型构造对象

### `malloc`

- 只分配原始内存
- 不负责对象构造

所以对于类对象来说，`new` 和 `malloc` 的差异不是“写法不同”，而是：

- 一个在创建对象
- 一个只是在拿一块裸内存

释放也必须匹配：

- `new` 对 `delete`
- `new[]` 对 `delete[]`
- `malloc` 对 `free`

混用通常会直接进入未定义行为。

---

## 5. 生命周期为什么会和拷贝绑定在一起

一旦对象里持有资源，比如：

- 堆内存
- 文件句柄
- socket
- 锁

“对象被拷贝”就不再是简单复制几个值，而变成了资源所有权问题。

最核心的问题是：

- 两个对象是不是共享同一份资源
- 销毁时到底谁来释放
- 一个对象销毁后，另一个对象还能不能安全使用

这就是为什么 C++ 里会不断讨论：

- 深拷贝
- 浅拷贝
- 拷贝构造
- 赋值运算符
- 析构函数

它们本质上都在讨论“资源跟着对象怎么活、怎么死”。

---

## 6. 一个典型错误：默认拷贝带来的双重释放

假设有这样一个类：

```cpp
struct Buffer {
    int* data;

    Buffer(int n) {
        data = new int[n];
    }

    ~Buffer() {
        delete[] data;
    }
};
```

表面上看没问题，但如果这样写：

```cpp
Buffer a(10);
Buffer b = a;
```

默认拷贝只会把指针值复制过去，结果是：

```text
a.data ----\
            -> 同一块堆内存
b.data ----/
```

当 `a` 和 `b` 分别析构时，就会对同一块内存 `delete[]` 两次，直接形成未定义行为。

这就是为什么资源对象一旦支持拷贝，就必须认真定义：

- 是否允许拷贝
- 拷贝后资源怎么管理

---

## 7. 一个更稳的资源管理方式：RAII

RAII 的核心思想是：

- 资源在对象构造时获得
- 资源在对象析构时释放

这样资源生命周期就跟对象生命周期绑定在一起。

例如：

```cpp
#include <memory>

void func() {
    auto p = std::make_unique<int[]>(10);
    // 使用 p
} // 离开作用域自动释放
```

它的优势在于：

- 作用域结束自动释放
- 异常中途退出也不会泄漏
- 不需要人工记住每个 `delete`

这也是现代 C++ 更推荐智能指针和标准容器的原因。

---

## 8. 异常为什么会把生命周期问题放大

很多资源问题在正常流程里看不出来，但一旦中途抛异常就暴露了。

例如：

```cpp
void work() {
    FileHolder f("a.txt");
    // 中间抛异常
}
```

如果资源绑定在对象析构里，那么异常离开作用域时析构仍会执行，资源能被正确释放。

如果资源靠手动 `close()`、`delete` 管理，就很容易在异常路径里漏掉。

所以异常安全本质上也是生命周期管理问题。

---

## 9. 一个稳定的理解顺序

学习对象生命周期时，建议按这个顺序思考：

1. 对象什么时候创建
2. 对象什么时候销毁
3. 它有没有持有额外资源
4. 如果被拷贝，这份资源归谁管
5. 如果异常退出，资源会不会丢

把这五个问题想清楚，后面的：

- 析构函数
- RAII
- 智能指针
- Rule of Three / Five

都会自然很多。

---

## 10. 常见误区

### 10.1 把作用域和生命周期当成同一件事

它们经常相关，但不等价。

### 10.2 觉得 `new` 和 `malloc` 只是两种写法

一个创建对象，一个只分配裸内存。

### 10.3 资源对象照样使用默认拷贝

这很容易带来双重释放或悬空指针。

### 10.4 把资源管理理解成“最后记得 delete 就行”

真正稳定的管理必须让资源和对象生命周期绑定。

---

## 小结

对象生命周期管理的核心，不是记构造函数和析构函数什么时候调用，而是保证对象存在期间资源状态始终正确，对象结束时资源也能跟着正确结束。真正理解这一点后，C++ 里的拷贝、析构、泄漏、悬空指针和异常安全，本质上都会变成同一个问题：对象怎么活，资源就该怎么活；对象怎么死，资源也必须怎么死。
