# C++ STL 容器、迭代器与算法

## 摘要

STL 的核心不是背接口，而是理解一套统一模型：用容器存数据、用迭代器描述范围、用算法处理范围。真正掌握 STL，关键不在于记住多少成员函数，而在于能根据访问模式、复杂度、迭代器能力和失效规则做出稳定选型。

---

## 核心问题

- `vector`、`deque`、`list`、`set`、`map`、`unordered_map` 的选择逻辑到底是什么
- 迭代器为什么是 STL 的中枢，而不是顺手的遍历工具
- 算法为什么依赖“范围”而不依赖具体容器

---

## 1. STL 到底统一了什么

STL 的设计思想可以概括成三层：

- 容器：负责存什么、怎么组织
- 迭代器：负责从哪里开始、到哪里结束、怎样访问
- 算法：负责对一段范围做排序、查找、变换、统计

这意味着很多算法根本不关心你底层是什么容器，它只关心：

- 你能不能给我一段合法范围
- 这段范围的迭代器能力够不够

例如：

```cpp
std::sort(v.begin(), v.end());
```

`sort` 不关心 `v` 叫不叫 `vector`，它关心的是 `begin()` 和 `end()` 提供的是随机访问迭代器。

---

## 2. 容器选择的第一原则：先看访问模式

很多 STL 选型问题，本质上都不是“哪个更高级”，而是“你主要在做什么操作”。

### 2.1 `vector`

本质：连续内存的可扩容数组。

适合：

- 高频随机访问
- 尾部插入删除
- 需要良好缓存局部性的场景
- 要配合排序、二分、双指针等算法

典型复杂度：

- 下标访问：`O(1)`
- 尾插：均摊 `O(1)`
- 中间插入删除：`O(n)`

为什么常作为默认容器：

- 数据连续
- 访问快
- 和算法兼容性最好
- 心智成本低

### 2.2 `deque`

本质：分段连续结构，支持两端高效插入删除。

适合：

- 头尾都经常操作
- 滑动窗口
- 单调队列

典型复杂度：

- 头尾插入删除：`O(1)` 均摊
- 随机访问：`O(1)`

### 2.3 `list`

本质：双向链表。

适合：

- 已知位置附近频繁插入删除
- 需要稳定迭代器

不适合：

- 高频随机访问
- 追求缓存局部性和实际常数性能

典型复杂度：

- 已知迭代器位置插删：`O(1)`
- 查找某个位置：`O(n)`

很多初学者高估 `list`，因为“中间插删快”是理论优势，但实际工程里，找位置本身往往更贵。

## 3. 关联容器和无序容器怎么选

### 3.1 `set` / `map`

通常基于平衡树。

特点：

- 元素有序
- 支持范围查询
- 查找、插入、删除通常 `O(log n)`

适合：

- 需要有序遍历
- 需要前驱后继、区间、范围查找
- 需要复杂度更稳定

### 3.2 `unordered_set` / `unordered_map`

通常基于哈希表。

特点：

- 无序
- 查找、插入、删除均摊 `O(1)`
- 最坏情况可能退化

适合：

- 主要做 key 到 value 的快速映射
- 不关心顺序

判断标准不是“哈希是不是比树快”，而是：

- 要不要顺序
- 要不要范围能力
- 能不能接受哈希冲突和迭代顺序不稳定

## 4. 一张常用容器选型表

```text
需求                          常见首选
按下标随机访问                vector
头尾高频插删                  deque
已知位置频繁插删              list
需要有序 key 和范围查找       map / set
只做快速映射，不关心顺序      unordered_map / unordered_set
需要优先级最值                priority_queue
```

这张表不是死规则，但能帮你先建立第一反应。

## 5. 迭代器到底是什么

迭代器不是“for 循环的另一种写法”，它本质上是：

- 对容器元素位置的抽象
- 对一段范围的统一描述

可以把它想成“指向序列某个位置的泛化指针”。

例如：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    std::cout << *it << '\n';
}
```

这里 `it` 的意义不是语法花样，而是在表达：

- 从 `begin()` 开始
- 一直走到 `end()`
- 每次访问当前位置元素

STL 算法之所以能复用，就是因为它们都在操作这种“范围”。

---

## 6. 迭代器能力为什么重要

不同算法对迭代器能力要求不同。

常见能力从弱到强可以粗略理解为：

- 输入迭代器
- 输出迭代器
- 前向迭代器
- 双向迭代器
- 随机访问迭代器

为什么这会影响算法能不能用：

- `sort` 需要随机访问迭代器
- `list` 只有双向迭代器，所以不能直接用 `std::sort`
- `advance`、`distance` 在不同迭代器上的代价也不同

这说明 STL 不是“所有算法都能套所有容器”，而是要看容器提供的迭代器能力。

---

## 7. 算法为什么依赖范围而不是依赖容器

STL 算法的设计思想是：

“只要你能给我一段满足要求的范围，我就可以工作。”

例如：

```cpp
std::find(v.begin(), v.end(), 42);
std::accumulate(v.begin(), v.end(), 0);
std::lower_bound(v.begin(), v.end(), 42);
```

这几个算法表达的问题分别是：

- 找元素
- 求累计
- 在有序范围中做二分定位

真正重要的是问题本身，而不是你手写了多少循环。

## 8. 迭代器失效为什么是工程高频坑

STL 里最隐蔽的问题之一就是迭代器失效。

例如 `vector` 扩容后，旧内存可能被搬走，原来的迭代器、引用、指针就失效了。

示例：

```cpp
std::vector<int> v = {1, 2, 3};
auto it = v.begin();
v.push_back(4);  // 可能触发扩容
// 此时 it 可能已经失效
```

类似地：

- `erase` 后被删位置后的迭代器可能失效
- `unordered_map` rehash 后迭代器可能失效

这类问题在面试和工程里都很常见，因为代码表面看起来“逻辑没问题”。

## 9. 一个常见选型例子

假设你要统计单词出现次数。

第一反应通常是：

```cpp
std::unordered_map<std::string, int> cnt;
for (const auto& word : words) {
    ++cnt[word];
}
```

为什么这通常合理：

- 核心需求是 key -> count 映射
- 不关心按字典序遍历
- 查找和更新频率高

如果题目后来改成：

- 需要按字典序输出
- 需要查某个范围内的 key

这时可能就该换成 `map`。

真正成熟的 STL 使用，不是固定背一个容器，而是随着需求变化调整结构。

## 10. 常见误区

### 10.1 会背接口就以为会 STL

真正难的是选型、复杂度和语义，不是成员函数名字。

### 10.2 只看理论复杂度，不看实际常数

理论上链表中间插删快，不代表实际工程就比 `vector` 好。

### 10.3 忽略迭代器失效

这是很多隐藏 bug 的来源。

### 10.4 不看算法要求

算法不是对所有容器都通用，关键在于迭代器能力是否匹配。

---

## 小结

STL 真正要掌握的不是零散接口，而是三件事：用什么容器组织数据、用什么迭代器表达范围、用什么算法处理问题。当你开始从访问模式、复杂度、范围能力和失效规则去思考，而不是只记“这里该调哪个函数”时，STL 才算真正入门。
