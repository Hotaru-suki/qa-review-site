# 迭代器失效与常见 STL 陷阱

## 这个知识点在解决什么问题
STL 容器很好用，但很多 bug 不是出在算法本身，而是出在容器修改之后，原来的迭代器、引用、指针还被继续使用。迭代器失效问题，是 C++ 开发和面试里的高频坑点。

## 什么叫迭代器失效
容器内部结构变化后，原有迭代器不再合法，再继续用就可能出错。

常见触发点：
- `vector` 扩容
- `erase` 删除元素
- `insert` 插入元素
- 某些容器重排内部结构

## 为什么不同容器表现不同
因为底层存储结构不一样：
- `vector` 连续存储，扩容时可能整体搬家
- `list` 节点分散，局部插删对其他节点影响较小
- `unordered_map` rehash 时桶结构会重建

这也是为什么“同样是插入元素”，不同容器的失效规则完全不同。

## 一个典型坑
在遍历 `vector` 时删除元素：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it == 0) {
        v.erase(it);
    }
}
```

这里 `erase` 后原迭代器已经失效，继续 `++it` 就危险了。

更安全的思路通常是接住 `erase` 的返回值，再继续遍历。

例如：

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it == 0) {
        it = v.erase(it);
    } else {
        ++it;
    }
}
```

## 不同容器要特别注意什么
### `vector`
- 扩容会让旧迭代器、引用、指针整体失效
- 中间插删还会让后面元素位置变化

### `unordered_map`
- rehash 之后桶结构变化，相关迭代器可能失效

### `list`
- 局部插删通常更稳定，但不能因此忽略节点被删后的失效问题

## 为什么这不只是“面试知识点”
真实工程里很常见：
- 缓存容器扩容后，另一个模块还拿着旧引用
- 遍历过程中删元素，偶发崩溃
- 日志里打印正常，换成高并发环境才出现数据错乱

所以它本质是工程稳定性问题，不只是 STL 语法点。

## 什么时候最容易踩坑
- 边遍历边删
- 容器扩容后继续保存旧元素地址
- `unordered_map` 插入很多元素后触发 rehash
- 把一个容器元素的引用长期传给别的模块保存

## 为什么这对测试开发也重要
如果你在写 C++ 工具、客户端自动化或引擎扩展代码：
- 偶发崩溃
- 数据遍历错乱
- 某些元素突然读不到

很可能不是业务逻辑错，而是容器修改后旧引用还在被使用。

## 测试和排障里为什么要懂它
很多线上偶发崩溃、数据错乱、遍历异常，背后其实是：
- 容器改了
- 旧引用还在用
- 问题表现却在很后面才出现

## 常见误区
- 记住了几个结论，但不知道为什么失效
- 只关注 `vector`，忽略 `unordered_map` rehash
- 删除元素后继续使用旧迭代器
- 以为“当前运行没崩”就说明写法没问题
