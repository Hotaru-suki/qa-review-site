# 05 数据库基础概念

## 1. 为什么要学数据库基础概念

前面的 SQL 更偏“怎么写查询”。  
这一部分更偏“数据库为什么这样设计、为什么会快、为什么会出问题”。

在测试、测开、后端面试里，常见追问包括：

- 主键是什么
- 外键是什么
- 索引是什么
- 联合索引是什么
- 最左前缀原则是什么
- 为什么建了索引还可能慢

所以这一部分是 SQL 基础向数据库基础过渡的关键。

---

## 2. 主键是什么

主键（Primary Key）是**一张表中用来唯一标识一条记录的字段或字段组合**。

### 2.1 主键的特点

- 唯一：不能重复
- 非空：不能为 `NULL`
- 一张表通常只有一个主键约束
- 很多数据库会自动为主键建立索引

### 2.2 例子

```sql
CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    score INT
);
```

这里：

- `id` 是主键
- 每条学生记录必须有唯一的 `id`
- 不能出现两个学生的 `id` 一样

### 2.3 主键的作用

主键的作用主要有：

- 唯一标识一条数据
- 便于查询、更新、删除
- 便于和其他表建立关联
- 提升数据组织和访问效率

### 2.4 测试场景理解

比如订单表中的 `order_id` 通常就是主键。  
如果测试发现两条订单数据有相同的 `order_id`，那通常就是严重问题，因为订单应该被唯一标识。

---

## 3. 外键是什么

外键（Foreign Key）是**一张表中用来引用另一张表主键的字段**，用于建立表与表之间的关系。

### 3.1 例子

```sql
CREATE TABLE classes (
    class_id INT PRIMARY KEY,
    class_name VARCHAR(50)
);

CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    class_id INT,
    FOREIGN KEY (class_id) REFERENCES classes(class_id)
);
```

这里：

- `classes.class_id` 是主键
- `students.class_id` 是外键

意思是：

> 学生表中的 `class_id` 必须引用班级表中已经存在的 `class_id`

### 3.2 外键的作用

- 建立表与表之间的关联关系
- 保证引用数据的正确性
- 防止出现“引用了不存在对象”的脏数据
- 提升数据完整性

### 3.3 测试场景理解

例如：

- 订单表中的 `user_id` 引用用户表
- 支付表中的 `order_id` 引用订单表
- 成绩表中的 `student_id` 引用学生表

如果支付记录里有一个 `order_id` 根本不存在于订单表里，那通常就是数据链路问题或脏数据问题。

---

## 4. 主键和外键的区别

| 项目 | 主键 | 外键 |
|---|---|---|
| 作用 | 唯一标识本表记录 | 引用其他表记录 |
| 是否唯一 | 必须唯一 | 可以重复 |
| 是否可为空 | 通常不能为空 | 视业务而定 |
| 主要用途 | 标识自己 | 关联别人 |

### 一句话记忆

- 主键：标识自己
- 外键：关联别人

---

## 5. 索引是什么

索引（Index）可以理解为：

> **数据库里的目录或检索加速结构**

它的核心作用是让数据库在查询数据时，不必每次都从头到尾扫描整张表。

### 5.1 类比理解

你要在一本很厚的书里找“事务”这个词：

- 没有目录：只能一页一页翻
- 有目录：直接看目录快速定位

数据库索引就类似这个目录。

### 5.2 为什么需要索引

如果一张表数据很多，而查询条件又很频繁，没有索引时数据库可能只能全表扫描，效率会很差。

有索引后，数据库通常可以更快定位到目标数据。

### 5.3 建立索引示例

```sql
CREATE INDEX idx_students_name
ON students(name);
```

这表示在 `students` 表的 `name` 字段上创建索引。

---

## 6. 索引的优点和代价

### 6.1 索引的优点

- 加快查询速度
- 加快筛选速度
- 加快排序、连接效率
- 某些情况下减少扫描数据量

### 6.2 索引的代价

索引不是免费的，它也有成本：

- 占用额外存储空间
- 插入、更新、删除时需要维护索引
- 索引太多会拖慢写操作
- 索引设计不合理可能起不到预期效果

### 6.3 一句话理解

> 索引提升读性能，但会增加写入维护成本。

---

## 7. 哪些字段适合建索引

通常适合在这些字段上考虑索引：

- 经常出现在 `WHERE` 中的字段
- 经常用于 `JOIN` 的字段
- 经常用于 `ORDER BY` 的字段
- 经常用于分组、筛选的字段
- 区分度高、唯一性高的字段

例如：

- `user_id`
- `order_id`
- `create_time`
- `email`
- `phone`

### 7.1 不适合随便建索引的字段

- 数据量很小的表
- 取值特别少、区分度很低的字段
- 很少查询却频繁修改的字段
- 完全不会作为筛选条件的字段

---

## 8. 索引的常见分类（入门版）

### 8.1 主键索引

主键对应的索引，通常数据库会自动创建。

### 8.2 唯一索引

字段值不能重复。

```sql
CREATE UNIQUE INDEX idx_email
ON users(email);
```

### 8.3 普通索引

最常见的索引类型，用于加快查询。

```sql
CREATE INDEX idx_status
ON orders(status);
```

### 8.4 联合索引

多个字段组成一个索引。

```sql
CREATE INDEX idx_user_status
ON orders(user_id, status);
```

---

## 9. 联合索引是什么

联合索引就是把多个字段按顺序放在同一个索引里。

例如：

```sql
(user_id, status, create_time)
```

它不是 3 个独立索引，而是一个**按字段顺序组织起来的组合索引**。

### 9.1 为什么要用联合索引

因为很多业务查询本来就是多条件组合：

```sql
SELECT *
FROM orders
WHERE user_id = 1001
  AND status = 'paid'
ORDER BY create_time DESC;
```

如果这类查询很常见，联合索引通常比多个单列索引更有效率。

---

## 10. 最左前缀原则是什么

最左前缀原则是联合索引里非常高频的概念。

假设有联合索引：

```text
(user_id, status, create_time)
```

通常更容易支持这些查询：

- `WHERE user_id = ?`
- `WHERE user_id = ? AND status = ?`
- `WHERE user_id = ? AND status = ? AND create_time = ?`

但不一定很好支持：

- `WHERE status = ?`
- `WHERE create_time = ?`

### 10.1 为什么叫最左前缀

因为联合索引是按字段顺序组织的，使用时通常要尽量从最左边开始。

### 10.2 一句话记忆

> 联合索引要尽量从最左边字段开始使用。

---

## 11. 覆盖索引是什么（基础理解）

覆盖索引可以简单理解为：

> 查询需要的字段，索引本身就有，不需要再回表去取。

例如：

```sql
SELECT user_id, status
FROM orders
WHERE user_id = 1001;
```

如果索引里刚好包含这些字段，数据库可能直接从索引中取结果。

### 11.1 当前阶段需要掌握到什么程度

你现在不用深究底层实现，只要知道：

- 覆盖索引通常更快
- 因为减少了额外的回表操作

---

## 12. 什么是全表扫描

全表扫描（Full Table Scan）就是：

> 把表里的每一行都检查一遍，看看符不符合条件。

例如：

```sql
SELECT *
FROM users
WHERE phone = '123456';
```

如果 `phone` 上没有索引，数据库可能只能一行一行地找。

### 12.1 全表扫描一定错吗

不是。  
当表很小的时候，全表扫描未必有明显问题。  
但数据量很大时，全表扫描通常意味着性能风险。

---

## 13. 面试常见问法与标准答法

### 13.1 主键是什么

> 主键是一张表中用来唯一标识一条记录的字段或字段组合，通常要求唯一且不能为空。

### 13.2 外键是什么

> 外键是用于建立表与表之间关联关系的字段，它引用另一张表的主键，用于保证引用完整性。

### 13.3 索引是什么

> 索引是数据库中用于提升查询效率的数据结构，可以理解为目录。它能加快查询、筛选、排序和连接，但会占用空间，并增加写操作的维护成本。

### 13.4 联合索引是什么

> 联合索引是由多个字段组成的组合索引，适合高频多条件查询。使用时通常要注意最左前缀原则。

### 13.5 最左前缀原则是什么

> 最左前缀原则指的是联合索引通常需要从最左边字段开始使用，才能更好地发挥索引效果。

---

## 14. 当前阶段必须记住的核心点

- 主键：唯一标识本表一行
- 外键：引用其他表的主键
- 索引：提高查询速度的目录结构
- 索引不是越多越好
- 高频查询字段才值得重点考虑索引
- 联合索引要注意最左前缀原则
- 覆盖索引通常更快
- 全表扫描在大表里通常需要警惕
