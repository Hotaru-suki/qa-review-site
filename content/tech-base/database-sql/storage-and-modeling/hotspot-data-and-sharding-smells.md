# 热点数据与分库分表前的坏味道

## 这个知识点在解决什么问题
很多系统一慢，就有人马上提分库分表。但真正的问题可能只是热点数据集中、表设计不合理、索引和查询模式有问题。这个知识点解决的是：在走到分库分表之前，哪些现象已经提示你的模型有明显坏味道。

## 什么叫热点数据
热点数据是指少量记录、少量键或少量分区承受了远高于平均水平的访问压力。

典型例子：
- 爆款商品库存行
- 高活跃主播房间
- 全局配置表中的单条记录
- 按时间连续增长导致某个区间特别热

## 为什么热点会先出问题
因为热点不一定让整库都慢，但会让某些行、某些索引、某些分区先成为瓶颈：
- 锁竞争
- 更新串行化
- 缓存击穿
- 单分片流量过大

## 分库分表前的常见坏味道
- 单表主键增长导致写入集中
- 大量查询和更新都围绕少量热点键
- 业务强依赖跨表、跨库事务
- 为了追求方便，把很多不同访问模式都压到一张表

## 为什么测试开发要提前识别
你在压测或问题排查时，可能会看到：
- 总 QPS 还行，但某类请求长尾很差
- 只有某些业务键一压就出问题
- 锁等待集中在少量记录

这些往往已经说明不是“整体不行”，而是热点和模型出了问题。

## 常见误区
- 一出现大表就立刻谈分库分表
- 不分析热点键和访问模式，只看总数据量
- 把分库分表当万能性能方案
