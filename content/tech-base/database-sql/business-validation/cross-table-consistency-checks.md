# 跨表一致性校验思路

## 摘要

很多业务问题不是某一张表错了，而是多张表之间没有一起对齐。跨表一致性校验解决的就是这种问题：主表和明细表是不是一致，状态表和流水表是不是一致，结果表和日志表是不是在描述同一个业务事实。

---

## 核心问题

- 为什么单表正确不等于业务链路正确
- 跨表一致性最常在哪几类场景出问题
- 怎样用 SQL 把“一条业务链是否闭环”查出来

---

## 1. 为什么跨表一致性这么重要

真实业务里，一次动作往往不只改一张表。

例如：

- 下单会改订单主表、订单明细表、库存表
- 支付会改支付流水表、订单状态表、账户资金表
- 发奖会改任务表、发奖记录表、背包表

如果只看其中一张表，常常会误判：

- 这事好像成了

但实际上，真正需要验证的是：

- 这些表是不是共同描述了同一个结果

---

## 2. 跨表一致性常见问题类型

### 主表有记录，从表缺失

例如订单主表有了，但明细表没写进去。

### 状态已推进，副作用没完成

例如支付状态成功了，但库存、积分或通知没同步。

### 多张表状态不一致

例如任务表显示完成，流水表却还停在处理中。

### 重复写入只发生在部分表

例如主表只有一条，流水表却重复写了两条。

---

## 3. 最常见的三类校验关系

### 3.1 主表和明细表

关注：

- 主实体有没有完整展开成从表数据

典型场景：

- 订单和订单明细
- 工单和工单操作记录

### 3.2 状态表和流水表

关注：

- 状态推进时是否留下正确流水

典型场景：

- 支付状态和支付流水
- 审核状态和审核记录

### 3.3 结果表和资源表

关注：

- 最终业务结果是否真的反映到资源变化上

典型场景：

- 发奖记录和背包表
- 扣库存记录和库存余额表

---

## 4. 一个订单场景怎么查

假设你想验证：

- 订单创建是否真正闭环

更完整的链路可能是：

```text
订单主表 -> 订单明细表 -> 支付流水表 -> 状态日志表
```

可以先查：

### 主表有、明细表有没有

```sql
SELECT o.order_id
FROM orders o
LEFT JOIN order_items i
  ON o.order_id = i.order_id
WHERE o.create_time >= '2026-04-01'
  AND i.order_id IS NULL;
```

### 支付成功了、订单状态是不是也成功

```sql
SELECT p.order_id, p.pay_status, o.order_status
FROM payment_record p
JOIN orders o
  ON p.order_id = o.order_id
WHERE p.pay_status = 'SUCCESS'
  AND o.order_status <> 'PAID';
```

这类查询本质上都是在问：

- 一张表上的事实，另一张表是否也承认

---

## 5. 一个发奖场景怎么查

假设支付后会发放道具。

更常见的链路是：

```text
支付流水 -> 发奖任务 -> 发奖记录 -> 背包表
```

要查的就不再只是“有没有支付成功”，而是：

- 支付成功后是否生成发奖任务
- 发奖任务完成后是否生成发奖记录
- 发奖记录生成后背包是否真的增加

这类问题如果只看支付流水，几乎一定不够。

---

## 6. 怎么判断问题落在哪一层

跨表一致性校验最有价值的地方，是能帮你把问题定位到具体断点。

例如：

- 主表有，从表没有
  更像写入逻辑或事务问题
- 状态表成功，流水表缺失
  更像日志 / 事件记录丢失
- 流水表有，资源表没变
  更像副作用执行失败

也就是说，跨表校验不仅是“发现错”，还能帮助回答：

- 错大概发生在哪一步

---

## 7. 常见误区

### 7.1 只查主表

这会错过大量链路断点问题。

### 7.2 只查结果表，不查过程表

状态日志、流水表往往最能解释问题。

### 7.3 不按业务链路组织 SQL

会导致查到数据却解释不了现象。

### 7.4 看到跨表不一致就只怪数据库

很多问题根因其实在业务逻辑、异步消费或补偿链路。

---

## 小结

跨表一致性校验的核心，不是把表都查一遍，而是按业务链路去验证“同一个事实是否被多张表同时正确记录”。只要掌握主表/从表、状态/流水、结果/资源这几类关系，很多复杂业务问题都能更快被拆开和定位。
