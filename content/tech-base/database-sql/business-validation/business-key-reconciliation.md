# 按业务键做链路对账

## 这个知识点在解决什么问题
测试场景里最常见的一类数据校验，不是单表字段对不对，而是同一笔业务在多张表、多套系统里是否一致。按业务键做链路对账，解决的是：如何用订单号、流水号、请求号这类业务键，把主流程和副作用串起来验证。

## 什么叫业务键对账
业务键对账的核心不是“查到一条记录”，而是：
- 用同一个业务标识串起主表、流水表、关联表
- 看每一层状态和金额是否一致
- 看有没有漏写、重复写、错状态

常见业务键有：
- 订单号
- 支付流水号
- 退款单号
- 券发放流水号

## 一个典型场景
### 支付成功链路
你至少要串这些对象：
- 订单表
- 支付流水表
- 营销优惠表
- 库存扣减表

检查点不是“每张表都有数据”，而是：
- 它们是不是同一笔业务
- 时间顺序是否合理
- 金额、状态、次数是否匹配

## SQL 思路怎么拆
1. 先确定主对象
- 例如订单表

2. 再用业务键关联副表
- 支付流水、库存流水、优惠流水

3. 再比关键字段
- 状态
- 金额
- 时间
- 次数

## 一个简单的校验思路
```sql
select o.order_id,
       o.status as order_status,
       p.status as pay_status,
       s.status as stock_status
from orders o
left join pay_flow p on o.order_id = p.order_id
left join stock_flow s on o.order_id = s.order_id
where o.order_id = 'A20260405001';
```

这类查询的价值不是炫 SQL，而是让你快速看出同一业务键在多张表里的状态有没有对齐。

## 再看一个更复杂的例子
### 退款链路
你可能需要同时校验：
- 订单状态是否从已支付变成退款中/已退款
- 支付退款流水是否生成
- 优惠券是否退回
- 营销补贴是否回滚

这时一个业务键往往不够，通常还要组合：
- 订单号
- 退款单号
- 支付渠道流水号

## 对账时最常看的字段
- 主状态和子状态
- 金额
- 次数
- 更新时间
- 错误原因码

如果这些字段之间对不上，就很难说这笔业务真正完成了。

## 测试里怎么落成固定套路
1. 先找主表
2. 再找副作用表
3. 最后做状态、金额、时间一致性对比

很多接口自动化也可以把这套套路沉淀成可复用查询模板。

## 测试开发为什么特别常用
很多接口问题表面上返回成功，但真实业务结果出错：
- 订单创建了，库存没扣
- 退款状态成功，但支付侧没回写
- 券发放成功，但用户账户里没显示

这类问题必须靠业务键对账才能抓出来。

## 常见误区
- 只查一张主表
- 没有固定业务键，查不到整条链
- 只看最终状态，不看多张表是否彼此对应
- 只看“有没有数据”，不看字段值是否真正一致
