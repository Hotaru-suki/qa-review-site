# 数据校验 SQL 模式

## 摘要

测试和排障里最常用的 SQL，不是复杂统计题，而是能快速把业务怀疑转成数据证据的几类固定模式。真正有价值的 SQL 校验能力，不在于会写多少语法，而在于拿到一个问题后，能快速判断这次应该查异常、查重复、查缺失、查最新状态，还是查聚合结果是否异常。

---

## 核心问题

- 测试工作里最常复用的 SQL 校验模式有哪些
- 每种模式分别在验证什么业务问题
- 怎样把“怀疑有问题”迅速转成可执行的查询

---

## 1. 为什么要把数据校验总结成模式

测试场景里最容易低效的地方是：

- 每来一个问题都临时想 SQL

更稳的做法是先把高频问题归类，再按模式下手。  
因为很多问题表面不同，本质却很像，例如：

- “奖励发重复了”
- “订单怎么多了两条”
- “这个用户怎么有两条有效记录”

本质都属于：

- 查重复

同样地：

- “页面显示成功，但数据库没变”
- “主表有了，明细表没了”

本质都属于：

- 查缺失或查链路断点

---

## 2. 模式一：查异常值

这类 SQL 关注的是：

- 字段值本身就不合理

典型场景：

- 金额为负数
- 状态值不在合法枚举内
- 成功状态下关键时间字段为空

示例：

```sql
SELECT order_id, status, pay_time, amount
FROM orders
WHERE status = 'PAID'
  AND (pay_time IS NULL OR amount <= 0);
```

它适合回答：

- “有没有明显错误数据已经落进库里”

---

## 3. 模式二：查重复数据

这类 SQL 关注的是：

- 本应唯一的业务数据是否出现重复

典型场景：

- 订单重复创建
- 奖励重复发放
- 幂等失效导致重复写入

示例：

```sql
SELECT biz_order_no, COUNT(*) AS cnt
FROM payment_record
GROUP BY biz_order_no
HAVING COUNT(*) > 1;
```

这类模式的关键不是对主键计数，而是先找：

- 业务唯一键

因为真正重复的往往不是数据库主键，而是业务语义上的唯一记录。

---

## 4. 模式三：查缺失链路

这类 SQL 关注的是：

- 一条业务链里是否有某一步没落下来

典型场景：

- 订单表有记录，但支付表没有记录
- 支付成功了，但积分表没有加分
- 主表有了，明细表没插成功

示例：

```sql
SELECT o.order_id, o.user_id
FROM orders o
LEFT JOIN payment_record p
  ON o.order_id = p.order_id
WHERE o.status = 'PAID'
  AND p.order_id IS NULL;
```

这类模式最常用于验证：

- 主表和从表
- 上游和下游
- 写操作和后续副作用

是否真的闭环。

---

## 5. 模式四：查最新状态

很多业务表会保留历史记录，但测试时真正关心的是：

- 当前最终状态是什么

典型场景：

- 每个订单的最新状态
- 每个用户最近一次登录记录
- 每个任务当前最新执行结果

示例：

```sql
WITH ranked AS (
  SELECT
    order_id,
    status,
    update_time,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY update_time DESC
    ) AS rn
  FROM order_status_log
)
SELECT order_id, status, update_time
FROM ranked
WHERE rn = 1;
```

这类模式特别适合：

- 状态流转
- 历史日志
- 最终一致性确认

---

## 6. 模式五：查聚合结果异常

这类 SQL 关注的是：

- 单行看没问题，但聚合后暴露异常

典型场景：

- 某用户订单数异常偏高
- 某区服发奖量异常
- 某天错误记录突然激增

示例：

```sql
SELECT user_id, COUNT(*) AS order_cnt
FROM orders
WHERE create_time >= '2026-04-01'
GROUP BY user_id
HAVING COUNT(*) > 100;
```

这类模式特别适合：

- 发现批量异常
- 对账前做数据健康扫描

---

## 7. 一个真实问题该怎么映射到模式

假设你收到一个反馈：

- “用户支付成功了，但没收到道具”

更稳的拆法通常是：

1. 先查支付流水是否成功
2. 再查发奖记录是否缺失
3. 再查背包表是否落库
4. 必要时查最新状态和消息消费记录

对应到模式就是：

- 查异常值：支付状态是否异常
- 查缺失链路：支付成功但发奖记录缺失
- 查最新状态：订单和发奖任务最终停在哪一步

这样问题就不再是模糊怀疑，而是能落成具体数据检查。

---

## 8. 常见误区

### 8.1 一上来就写 SQL，不先定义问题类型

会导致 SQL 能跑，但证明不了什么。

### 8.2 只查单表

很多问题真正出在链路断点和跨表不一致。

### 8.3 不先找业务唯一键

查重复时最容易失真。

### 8.4 只看接口返回，不做数据模式校验

这会漏掉大量“表面成功、内部错误”的问题。

---

## 小结

测试里的数据校验 SQL，真正重要的不是语法花样，而是问题模式。把异常、重复、缺失、最新状态和聚合异常这几类模式用熟之后，大量业务问题都能更快被转成明确、可执行、可复现的数据证据。
