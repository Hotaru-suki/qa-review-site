# 03 SQL 窗口函数与高频题

## 摘要

窗口函数解决的是“我既想保留明细行，又想在每一行上计算排名、累计值、组内编号或相邻差异”这类问题。它是 SQL 从基础查询走向真实业务分析、状态抽取和测试排查的关键能力。

---

## 核心问题

- 窗口函数和 `GROUP BY` 的本质区别是什么
- `ROW_NUMBER()`、`RANK()`、`DENSE_RANK()` 到底分别适合什么题
- 为什么很多“每组最新一条”“连续登录”“Top N”题，本质上都能还原成窗口函数问题

---

## 1. 窗口函数到底在做什么

窗口函数最核心的能力是：

- 不压缩原始行数
- 但允许你在“某个分组内部、某个排序规则下”给每一行增加一个计算结果

你可以把它理解成：

```text
原始明细行
   +
一个组内视角
   =
每一行都带着额外信息
```

例如：

- 这条记录在本用户的第几次登录
- 这条订单在本商家的金额排名第几
- 这条状态记录是不是该业务键下最新的一条

---

## 2. 为什么它和 GROUP BY 不一样

`GROUP BY` 的特点是把多行压成少量汇总结果。

例如：

```sql
SELECT user_id, MAX(login_time)
FROM login_log
GROUP BY user_id;
```

结果里只剩每个用户一行。  
问题是：你虽然得到了最大时间，但原来那一整行的其它字段可能丢了。

窗口函数则是：

```sql
SELECT
  user_id,
  login_time,
  ROW_NUMBER() OVER (
    PARTITION BY user_id
    ORDER BY login_time DESC
  ) AS rn
FROM login_log;
```

它不会把行压掉，而是给每一行加一个 `rn`。

所以两者的区别不是“谁更高级”，而是：

- `GROUP BY` 更适合做汇总
- 窗口函数更适合在保留明细的前提下做组内计算

---

## 3. OVER、PARTITION BY、ORDER BY 怎么理解

窗口函数最常见的结构：

```sql
函数名() OVER (
  PARTITION BY 分组字段
  ORDER BY 排序字段
)
```

可以拆成三部分理解：

### `OVER(...)`

表示“这个函数要在一个窗口里算”。

### `PARTITION BY`

表示先按哪个维度切成多个小组。

例如 `PARTITION BY user_id` 就是每个用户单独算，不会互相影响。

### `ORDER BY`

表示组内按什么顺序排。

如果不排序，很多排名和累计类函数就没有实际意义。

---

## 4. 三个最常见的排名函数

假设有一组成绩：

```text
100, 100, 95, 90
```

### 4.1 `ROW_NUMBER()`

逐行编号，不考虑并列。

```text
100 -> 1
100 -> 2
 95 -> 3
 90 -> 4
```

适合：

- 每组最新一条
- 每组第一条记录
- 明确要求“第 N 条”而不是“第 N 名”

### 4.2 `RANK()`

并列同名次，但后面跳号。

```text
100 -> 1
100 -> 1
 95 -> 3
 90 -> 4
```

适合：

- 真正带“名次”语义的排序题

### 4.3 `DENSE_RANK()`

并列同名次，但后面不跳号。

```text
100 -> 1
100 -> 1
 95 -> 2
 90 -> 3
```

适合：

- 查第 N 个不同值
- 问“第 3 名成绩”且希望并列不占空位

## 5. 高频题型怎么拆

### 5.1 每组最新一条

这是测试和业务查询里最常见的窗口函数题。

例如每个订单的最新状态：

```sql
WITH ranked AS (
  SELECT
    order_id,
    status,
    update_time,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY update_time DESC
    ) AS rn
  FROM order_status_log
)
SELECT order_id, status, update_time
FROM ranked
WHERE rn = 1;
```

为什么用 `ROW_NUMBER()`：

- 你不是在查“第几名”
- 而是在每组内部找排序后的第一条

### 5.2 Top N / 第 N 名

先分清题目到底在问什么：

- 第 N 条记录
- 第 N 个不同值
- 第 N 名且允许并列

这三种题，函数可能分别是：

- `ROW_NUMBER()`
- `DENSE_RANK()`
- `RANK()`

很多人题做错，不是 SQL 不会，而是题意没分清。

### 5.3 连续登录 / 连续日期

这类题的本质是把“连续的数据”识别成同一组。

典型思路：

1. 先按日期排序
2. 给每行一个连续编号 `ROW_NUMBER()`
3. 用“日期 - 编号”构造稳定分组键

示意：

```text
日期        编号   日期-编号
2026-04-01   1     K1
2026-04-02   2     K1
2026-04-03   3     K1
2026-04-05   4     K2
```

这样连续区间就会落到同一个组里。

### 5.4 累计值和滑动值

窗口函数不只做排名，还可以做：

- 累计求和
- 累计平均
- 环比/同比
- 与上一条记录比较

例如累计支付金额：

```sql
SELECT
  user_id,
  pay_time,
  amount,
  SUM(amount) OVER (
    PARTITION BY user_id
    ORDER BY pay_time
  ) AS running_amount
FROM payment_record;
```

## 6. 测试和排查里怎么用窗口函数

窗口函数在测试工作里最常见的用法有三类：

### 状态抽取

例如每个订单的最终状态、每个账号最新一次封禁状态。

### 去重和取最新

例如接口多次重试后，要确认最终只认哪一条记录。

### 找异常切换点

例如某用户前 10 次登录都成功，第 11 次开始连续失败，你可以结合排序和相邻比较找出转折点。

这也是窗口函数相比普通聚合更适合测试排查的原因：  
它能保留明细上下文，不容易把证据压扁。

---

## 7. 常见错误和误区

### 7.1 把“每组最新一条”写成 `MAX(time)`

只拿到最大时间，不一定能拿到对应整行。

### 7.2 不理解并列时排名差异

`ROW_NUMBER()`、`RANK()`、`DENSE_RANK()` 用错，会直接导致“第 N 名”题答案不对。

### 7.3 分组粒度错了

`PARTITION BY` 多一个字段或少一个字段，结果就可能完全变样。

### 7.4 排序规则不稳定

如果只按时间排，但时间会重复，最好补次级排序字段，否则“最新一条”可能不稳定。

---

## 8. 面试里怎么答窗口函数

更稳的答法不是背一串函数名，而是先说一句总逻辑：

“窗口函数的核心价值是保留明细行，同时在组内按顺序补充排名、编号或累计结果，所以像每组最新一条、Top N、连续登录、累计金额这类问题都很适合用它处理。”

然后再补：

1. `GROUP BY` 会压缩行，窗口函数不会
2. `ROW_NUMBER()` / `RANK()` / `DENSE_RANK()` 的区别
3. 一个真实例子，比如每组最新状态

这样面试官更容易判断你是真的理解，而不是背题。

---

## 小结

窗口函数的本质，不是“高级语法”，而是“在保留明细的前提下做组内计算”。只要把题目还原成“按什么分组、按什么排序、每行要补什么信息”，很多看起来复杂的 SQL 题都会变得稳定可解。
