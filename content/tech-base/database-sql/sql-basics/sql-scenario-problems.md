# 04 测试场景 SQL

## 摘要

测试岗位里常用的 SQL，不是为了做复杂报表，而是为了把“接口返回正确吗、数据库有没有脏数据、链路有没有断、状态有没有写错”这类问题查清楚。它的价值在于把抽象的业务问题落成可验证的数据证据。

---

## 核心问题

- 真实测试工作里最常见的 SQL 场景有哪些
- 每类场景应该先找什么表、什么字段、什么业务键
- 为什么很多 SQL 写不出来，本质上不是语法问题，而是验证目标没想清楚

---

## 1. 测试场景 SQL 真正解决什么

测试工作里写 SQL，核心不是“把题做对”，而是把下面几类问题查明白：

- 接口返回的结果和数据库最终落库是否一致
- 一次业务操作有没有写漏表、写错状态、重复写入
- 某个异常反馈到底是页面展示问题、服务返回问题还是数据落库问题
- 某条链路的中间状态有没有丢失、延迟或者乱序

所以 SQL 的角色更像“证据提取工具”，而不是单独的编程练习。

---

## 2. 写测试 SQL 前先做什么

拿到一个问题时，不要立刻开始写 `SELECT *`。更稳的顺序是：

1. 明确你要验证什么结论
2. 找到这条业务链路经过哪些表
3. 确认哪几个字段是业务唯一键
4. 再决定查异常、查重复、查缺失还是查最新状态

最常用的定位信息包括：

- 订单号、用户 ID、角色 ID、流水号
- 状态字段
- 创建时间、更新时间
- 金额、库存、次数这类关键业务字段

如果业务键和表关系没找清，SQL 往往会写成“能跑，但证明不了任何事”。

---

## 3. 四类最高频测试 SQL 场景

### 3.1 查异常数据

这类 SQL 关心的是“字段值本身就不合理”。

典型场景：

- 奖励数量是负数
- 支付成功但支付时间为空
- 状态字段写成了非法枚举值

示例：

```sql
SELECT order_id, status, pay_time, amount
FROM orders
WHERE status = 'PAID'
  AND (pay_time IS NULL OR amount <= 0);
```

这类 SQL 的关键不是条件多，而是要先知道：

- 什么值是合法区间
- 哪些字段之间应该同时出现
- 哪些状态不应该和哪些字段组合在一起

### 3.2 查重复数据

这类问题通常出现在：

- 接口重试导致重复写入
- 幂等没有做好
- 消息重复消费
- 补单脚本重复执行

示例：

```sql
SELECT biz_order_no, COUNT(*) AS cnt
FROM payment_record
GROUP BY biz_order_no
HAVING COUNT(*) > 1;
```

真正要点在于先找“业务唯一键”，而不是机械地对主键计数。  
主键天然不重复，真正要查的是“本来业务上只能有一条”的字段组合。

常见业务唯一键：

- 订单号
- 用户 ID + 奖励类型 + 发放批次
- 请求流水号 / 幂等号

### 3.3 查缺失数据

缺失数据本质是在查“链路有没有断”。

典型场景：

- 订单表里有下单记录，但支付表里没有支付记录
- 发货成功了，但背包表没写入
- 主表有数据，明细表没有对应行

示例：

```sql
SELECT o.order_id, o.user_id, o.status
FROM orders o
LEFT JOIN payment_record p
  ON o.order_id = p.order_id
WHERE o.status = 'PAID'
  AND p.order_id IS NULL;
```

为什么常用 `LEFT JOIN ... IS NULL`：

- 主表代表“理论上一定存在”的数据
- 从表代表“链路后续动作”
- 连不上时，就说明后续动作缺失

### 3.4 查最新状态

很多业务表都会留历史记录，测试时真正关心的是“现在最终状态是什么”。

典型场景：

- 每个订单的最新状态
- 每个用户最近一次登录
- 每个角色最近一次封禁或解封结果

示例：

```sql
WITH latest_status AS (
  SELECT
    order_id,
    status,
    update_time,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY update_time DESC
    ) AS rn
  FROM order_status_log
)
SELECT order_id, status, update_time
FROM latest_status
WHERE rn = 1;
```

这里窗口函数的价值是：

- 保留原始记录
- 在每个业务键内部做排序
- 从多条历史记录里筛出最新一条

## 4. 一个完整测试场景怎么拆

假设问题是：

“用户投诉充值接口返回成功，但账户余额没有增加。”

更稳的 SQL 排查路径通常是：

1. 先查充值请求日志表或充值订单表，确认请求有没有成功受理
2. 再查支付流水表，确认支付是否成功
3. 再查账户变更表，确认余额有没有写入
4. 再查账户当前余额表，确认最终状态是否正确

可能对应的链路是：

```text
充值请求表 -> 支付流水表 -> 账户变更表 -> 账户余额表
```

每一步要验证的不是同一件事：

- 请求表：有没有收到请求
- 流水表：扣款是否成功
- 账户变更表：业务动作有没有执行
- 余额表：最终结果是否正确

这样拆，你写 SQL 就不会只盯着一张表。

---

## 5. 常用模式为什么有用

### `GROUP BY ... HAVING`

适合查重复、聚合异常、批量统计。

### `LEFT JOIN ... IS NULL`

适合查缺失链路、缺少关联记录。

### 窗口函数

适合查每组最新一条、排名、连续状态。

### 条件过滤

适合查非法值、状态不一致、字段组合错误。

这些模式的共同点是：  
它们都在帮你把“业务怀疑”转成“数据证据”。

---

## 6. 测试视角下最容易错的地方

### 6.1 先写 SQL，后想验证目标

正确顺序应该反过来：  
先明确“我要证明什么”，再决定 SQL 怎么写。

### 6.2 只查单表，不查链路

很多线上问题不是某张表值错了，而是整条链路少了一步。

### 6.3 只看接口返回，不做数据库闭环

接口返回成功，只能说明“这一层说自己成功了”，不代表后面的写库、发消息、状态流转都完成了。

### 6.4 不找业务唯一键

查重复、查幂等、查链路时，不先找业务唯一键，结果很容易失真。

---

## 7. 如何把 SQL 和接口测试结合起来

接口测试里，SQL 最常见的作用有三类：

### 结果校验

接口返回的数据是否真的落到了数据库里。

### 副作用校验

一次调用除了主返回值之外，有没有顺带改动库存、积分、状态、日志、消息。

### 问题定位

接口失败时，到底是请求没进来、业务没执行、数据库没写成，还是写了但查错表。

这也是为什么真正成熟的接口测试，不会只停在断言 HTTP 状态码和 JSON 字段。

---

## 8. 面试里怎么答这类问题

如果面试官问“测试工作里 SQL 主要怎么用”，可以按这个顺序回答：

1. 先说用途：主要用于数据校验和问题定位
2. 再说高频场景：查异常、查重复、查缺失、查最新状态
3. 再说方法：先明确验证目标，再找表关系和业务键
4. 最后补一个真实例子：比如订单、支付、发奖、库存

比起背语法，这种回答更像真实工作经验。

---

## 小结

测试场景 SQL 的核心，不是“我会多少语法”，而是“我能不能把业务问题拆成可验证的数据问题”。真正有价值的 SQL，往往都围绕异常、重复、缺失、最新状态和链路一致性这几类问题展开。
