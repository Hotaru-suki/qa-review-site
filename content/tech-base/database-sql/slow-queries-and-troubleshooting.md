# 08 慢查询与排查基础

## 1. 什么是慢查询

慢查询就是：

> **执行时间明显过长、影响系统性能的 SQL。**

“慢”没有绝对固定标准，要结合业务场景、数据量、并发量来判断。

---

## 2. 为什么慢查询重要

慢查询会导致：

- 接口响应慢
- 页面加载慢
- 数据库压力上升
- 并发能力下降
- 系统整体吞吐下降

所以面试里问慢查询，考的其实是：

> 你有没有基本的性能排查思路。

## 4. 慢查询排查的基本思路

一个比较稳的回答思路如下：

### 第一步：先确认慢在哪里

先搞清楚：

- 是哪条 SQL 慢
- 是稳定复现，还是偶发
- 是查询本身慢，还是在等锁
- 是单次请求慢，还是并发一高就慢

### 第二步：看 SQL 本身

检查 SQL 写法有没有明显问题，例如：

- 是否写了不必要的 `SELECT *`
- 是否返回了太多列、太多行
- 是否缺少精准过滤条件
- 是否做了不必要的多表关联
- 是否深分页
- 是否存在过重的排序、分组

### 第三步：看索引情况

确认：

- 相关字段是否建了索引
- 联合索引设计是否合理
- 查询是否真的用了索引
- 有没有明显的索引失效场景

### 第四步：用 EXPLAIN 看执行计划

重点看：

- 是否全表扫描
- 是否用了索引
- 扫描了多少行
- 是否有 `Using filesort`
- 是否有 `Using temporary`

### 第五步：看业务和数据分布

### 第六步：看并发与锁等待

有些 SQL “慢”不是因为它查得慢，而是因为它在等待。

## 6. 深分页为什么慢（基础理解）

例如：

```sql
SELECT *
FROM orders
ORDER BY id
LIMIT 100000, 20;
```

数据库往往需要先跳过前 100000 行，再取后面 20 行。  
偏移量越大，通常开销越高，所以深分页常常慢。

---

## 7. 为什么 SELECT * 不好

`SELECT *` 的问题包括：

- 可能查出很多根本不需要的字段
- 增加 I/O 和网络传输开销
- 某些情况下不利于覆盖索引

---

## 8. 慢查询与锁等待的区别

有些 SQL “慢”不是因为查得慢，而是因为：

- 等锁
- 被别的事务阻塞
- 资源竞争严重

### 8.1 一句话理解

> 有些慢是“执行慢”，有些慢是“等待慢”。

## 10. 面试标准答法模板

### 10.1 什么是慢查询

> 慢查询就是执行时间明显过长、影响系统性能的 SQL 查询。

### 10.2 如果遇到慢查询，你会怎么排查

> 如果遇到慢查询，我会先定位具体是哪条 SQL 慢，再结合业务场景看是否查询条件不合理、返回数据过多或多表关联过重。然后我会检查相关字段是否有索引，并使用 `EXPLAIN` 查看执行计划，重点关注是否走索引、扫描行数、访问方式，以及是否有 `Using filesort`、`Using temporary` 等额外开销。如果 SQL 本身没有明显问题，我还会考虑是否存在锁等待、并发冲突或数据量增长带来的性能退化。

### 10.3 为什么建了索引还会慢

> 因为可能存在索引失效、联合索引未按最左前缀使用、查询命中数据太多、SQL 本身返回数据过多，或者并不是 SQL 本身慢，而是在等待锁。

---

## 11. 当前阶段必须记住的核心点

- 慢查询就是执行时间过长的 SQL
- 常见原因：没索引、索引失效、扫描数据多、排序分组重、JOIN 不合理
- 排查顺序通常是：定位 SQL -> 看写法 -> 看索引 -> 看 EXPLAIN -> 看数据分布 -> 看锁等待
- 深分页通常慢
- `SELECT *` 通常不够好
- 有些慢不是 SQL 执行慢，而是锁等待慢
