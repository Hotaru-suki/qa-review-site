# 查询思路与结果整形

## 这个知识点在解决什么问题
很多人会写 `select ... from ... where ...`，但一遇到真实业务查询就乱：不知道从哪张表起手，不知道结果要按什么粒度返回，也不知道该先过滤还是先聚合。查询思路和结果整形，解决的是“如何把业务问题翻译成 SQL”。

## 写查询前先确定三件事
1. 主体对象是谁
- 是订单、用户、支付记录，还是某个事件流水

2. 结果粒度是什么
- 一行代表一条订单
- 一行代表一个用户
- 一行代表一天的统计

3. 筛选条件和关联关系是什么
- 时间范围
- 状态条件
- 业务键
- 表之间用什么字段关联

## 为什么结果粒度这么重要
很多 SQL 写错，不是语法问题，而是粒度错了。

例如“统计每个用户最近一次支付时间”：
- 如果先把支付表和订单表连起来再直接查，容易一人多行
- 正确思路通常是先明确“一行代表一个用户”，再决定用聚合、子查询还是窗口函数

## 结果整形常见手法
### 过滤
先把范围缩小，避免后面关联太多无关数据。

### 关联
把主体对象和补充信息表连起来，但要明确是一对一还是一对多。

### 聚合
用 `count`、`sum`、`max` 等把多行压成目标粒度。

### 排序和截断
只保留最相关、最新、最高或前 N 条结果。

## 一个完整查询是怎么想出来的
以“查出最近 7 天内支付成功但库存未扣减的订单”为例，思路不是直接写 SQL，而是先拆问题：

1. 主体对象
- 订单

2. 结果粒度
- 一行代表一笔异常订单

3. 必要条件
- 最近 7 天
- 支付成功
- 库存未扣减或库存流水缺失

4. 需要关联的表
- 订单表
- 支付表
- 库存流水表

5. 结果展示字段
- 订单号
- 用户 ID
- 支付时间
- 库存状态

这样拆完后再写 SQL，出错概率会低很多。

## 为什么很多 SQL 会写成一团
通常是因为同时在做三件事：
- 定义结果粒度
- 找关联关系
- 写筛选条件

如果这三件事没有先在脑子里分开，就会出现“join 很多表但自己也说不清一行代表谁”的情况。

## 测试工作里最常见的查询模式
- 查单对象当前状态
- 查链路是否完整
- 查异常对象集合
- 查重复、漏处理、错状态
- 查按时间、渠道、角色分组的统计

这些场景本质上都离不开“主体 + 粒度 + 关联 + 过滤”这套框架。

## 一个最容易犯错的点
很多人写查询时，会在还没确定“结果一行代表谁”的情况下就开始 join 多张表。这样最后出来一堆重复行时，很难判断是业务本来一对多，还是 SQL 写错了。

所以真正的顺序应该是：
1. 先确定主体
2. 再确定结果粒度
3. 再决定是否需要 join、group by、窗口函数

## 典型场景
### 测试校验订单链路
你想知道一次下单是否完整成功：
- 从订单表定位业务键
- 关联支付表、库存表、优惠表
- 看是否所有关键节点都已更新

这时 SQL 不是为了“查数据”，而是为了还原业务状态。

## 常见误区
- 一上来就连很多表，没有先确定主体表
- 结果要的是“每用户一行”，却写成“每明细一行”
- 先把全量数据 join 再过滤，导致结果和性能都失控
- 不知道重复行是正常一对多，还是 SQL 写错了
