# 连接顺序与结果基数怎么判断

## 这个知识点在解决什么问题
很多 SQL 一写上 `join` 就开始重复行、漏行或结果爆炸。真正的问题往往不是语法不会，而是没有判断清楚每张表之间是一对一、一对多还是多对多，也没有想明白结果到底会膨胀成多少行。

## 什么叫结果基数
基数可以理解成“结果会有多少行、每一行代表什么”。  
当你把两张表连起来时，如果一边一条能对应另一边多条，结果就会膨胀。

例如：
- 用户表 `users`
- 订单表 `orders`

一个用户对应多条订单，那么：
- `users join orders`
- 结果通常是一用户多行，不再是“一用户一行”

## 为什么连接顺序会影响理解
虽然优化器可能重排执行顺序，但你在写查询时，脑子里的“主体表”仍然很重要。

如果主体是用户：
- 你要问结果是不是仍然一用户一行

如果主体是订单：
- 你要问是不是每一行都代表一笔订单

## 典型坑
### 想查“每个用户最近一笔订单”
如果你直接把用户表和订单表连起来再 `order by`，很容易得到重复用户行。

更合理的思路通常是：
1. 先在订单侧求出“每个用户最近一笔”
2. 再把结果连回用户表

## 一个简单判断方法
在写 `join` 前先问：
- 左表一行，对右表是几行
- 最终结果想保留哪边的粒度
- 是否需要先聚合、去重或窗口函数

## 测试开发为什么要懂
很多业务对账 SQL 都会跨多张表：
- 订单
- 支付
- 库存
- 营销

如果不先判断基数，查出来的结果很容易“看着有数据，其实重复放大了”。

## 常见误区
- 只要能查出结果就算对
- 不先定义一行代表谁，就直接多表 join
- 多对多 join 后结果爆炸，再靠 `distinct` 生硬去重
