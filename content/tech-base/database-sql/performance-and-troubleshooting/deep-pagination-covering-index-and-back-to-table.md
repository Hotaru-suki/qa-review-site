# 深分页、回表与覆盖索引为什么会慢

## 摘要

很多列表查询在数据量小时跑得很快，数据量一大就开始明显变慢，最常见的原因之一就是深分页和回表成本。这个知识点不是单独讲一个 SQL 技巧，而是帮你理解：为什么 `LIMIT 100000, 20` 这种写法会越来越贵，为什么“建了索引”也不一定够，以及覆盖索引到底解决了哪一段成本。

---

## 核心问题

- 深分页为什么会慢
- 回表到底在慢查询里增加了什么成本
- 覆盖索引为什么经常能改善列表查询，但又不是万能解法

---

## 1. 深分页慢在哪

例如：

```sql
SELECT id, status, created_at
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 100000, 20;
```

这类 SQL 的问题不在于“只返回了 20 行”，而在于：

- 数据库为了拿到这 20 行，往往要先定位并跳过前面 100000 行

也就是说，真正贵的是：

- 查找并排序候选数据
- 扫描并跳过大量不需要返回的记录

所以分页越深，代价通常越大。

### 用一个更直观的方式理解

可以把它想成：

- 你要第 100001 到 100020 条
- 但数据库往往不能直接“跳到”那一页并只读 20 条
- 它通常仍要沿着访问路径把前面大量候选记录处理过去

所以深分页慢的核心，不是返回结果多，而是“为得到结果而处理的中间数据太多”

---

## 2. 什么叫回表

如果索引里只有一部分列，而查询又要取索引外的列，数据库可能会：

1. 先通过索引找到候选记录
2. 再回到主表 / 聚簇数据页里把完整列取出来

这一步就叫回表。

当候选记录很多时：

- 每一行都可能多一次随机访问

于是“走了索引”也可能还是慢。

### 一个字符图

```text
二级索引: (user_id, created_at, id)
                    |
                    | 先找到候选 id
                    v
主表记录: id -> status, amount, address, remark ...
```

如果查询列里包含了 `status, amount, address` 这些不在索引里的字段，就需要从索引拿到 `id` 后，再回主表把完整列取出来。这一步在候选记录很多时会很贵。

---

## 3. 覆盖索引解决了哪段成本

覆盖索引指的是：

- 查询需要的列，索引本身就已经包含了

这样数据库只看索引就能得到结果，不需要回表拿额外列。

例如：

```sql
SELECT user_id, created_at
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 20;
```

如果存在合适的联合索引：

```sql
(user_id, created_at)
```

那这条查询就更有机会只在索引层完成。

覆盖索引真正减少的是：

- 额外的数据页访问
- 回表次数

### 但它不是“索引列越多越好”

因为索引列变多也意味着：

- 索引更大
- 写入维护成本更高
- 缓存命中和页利用率可能变差

所以覆盖索引是针对高频查询的一种有意识取舍，不是把所有列都堆进索引里。

---

## 4. 为什么“有覆盖索引”也可能还不够

因为深分页的核心问题不只是“取哪几列”，而是：

- 前面要跳过多少记录

即使完全不回表，如果仍要从索引里扫过大量记录再丢掉，成本依然会高。

所以：

- 覆盖索引主要改善“单条记录读取成本”
- 深分页问题还涉及“总扫描范围”

这就是为什么很多分页优化不能只停在“补索引”。

### 一个常见误判

很多人看到：

- 查询走了索引
- `type` 也不算差

就直接判断“索引没问题”。但如果：

- `rows` 很大
- `Extra` 里还有排序或回表相关成本
- 实际分页偏移量又很深

那依然可能慢得很明显。

---

## 5. 一个更完整的优化思路

面对深分页列表，更稳的思路通常是：

### 先确认排序键和过滤条件

- 是否有稳定排序
- 是否有高选择性的过滤条件

### 再看当前索引能否匹配

- 条件列和排序列是否在同一条索引路径里
- 是否能避免额外排序
- 是否能减少回表

### 再考虑是否改分页方式

很多场景更适合：

- 基于上次最后一条记录的游标分页
- 基于时间 / 主键的范围翻页

而不是一直用大偏移量的 `LIMIT offset, size`。

### 一个更具体的改写思路

原始写法：

```sql
SELECT id, created_at, status
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 100000, 20;
```

改成基于上次最后一条数据的游标方式，会更像：

```sql
SELECT id, created_at, status
FROM orders
WHERE user_id = 1001
  AND created_at < '2026-04-01 10:00:00'
ORDER BY created_at DESC
LIMIT 20;
```

这里的核心变化是：

- 不再跳过前面 100000 条
- 而是从上次位置继续往后取

这往往比单纯补索引更能从根上降低分页代价

---

## 6. 一个典型测试场景

假设后台订单列表在前几页很快，但翻到后面越来越慢。

更稳的排查路径是：

1. 看查询 SQL 是否用了大偏移量分页
2. 看 `EXPLAIN` 是否出现大量扫描
3. 看是否存在回表和额外排序
4. 看返回列是否过多
5. 再决定是补联合索引、改返回列，还是改分页方案

如果再具体一点，可以补看：

- 同样条件下第一页和第 5000 页 RT 差多少
- 扫描行数是不是随着页码明显变大
- 返回列里是否包含大字段
- 该场景到底是“翻页查看”，还是其实更适合搜索 / 导出

如果测试时只看“第一页很快”，很容易漏掉这个问题。

---

## 7. 这类问题和测试开发有什么关系

测试开发在做接口和后台性能验证时，经常会遇到：

- 列表页越翻越慢
- 管理后台报表后几页超时
- 导出 / 检索在大数据量下明显退化

这时如果你知道深分页和回表的机制，就不会只写“数据量大导致变慢”，而能更具体地说明：

- 当前更像偏移量分页导致扫描膨胀
- 或者查询列过多，索引命中了但回表成本仍然高

也能在测试报告里更准确地写：

- 问题触发条件是什么
- 是页码加深后退化，还是所有页都慢
- 更像执行方式问题，还是资源竞争问题
- 建议是索引优化、查询裁剪，还是交互方案调整

这样报告就不只是“慢”，而是已经接近根因判断

---

## 8. 常见误区

- 只要走索引就一定快
- 覆盖索引能解决所有分页慢的问题
- 只看返回 20 行，就忽略前面被跳过的海量数据
- 不区分“排序成本”“扫描成本”“回表成本”
- 只看第一页性能，不验证深页退化
- 把“后台需要翻很多页”当成必然，不思考是否应该改交互和访问方式

---

## 小结

深分页慢，不是因为返回行数多，而是因为扫描和跳过的数据越来越多；回表慢，不是因为索引没用，而是因为索引之外还要反复取完整数据；覆盖索引则是在减少这一步额外访问。把这三件事分清楚，列表类慢查询的很多现象就会更容易解释和定位。
