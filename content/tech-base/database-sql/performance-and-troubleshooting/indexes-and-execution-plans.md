# 07 索引与执行计划

## 摘要

索引与执行计划是数据库性能排查里最实用的一组内容。它真正解决的是：SQL 为什么慢、索引为什么没生效、数据库到底打算怎么跑这条语句，以及优化应该先改索引、改 SQL，还是改查询范围和返回列。

---

## 核心问题

- 索引为什么能快，为什么有时又帮不上忙
- `EXPLAIN` 里最关键的字段应该怎么看
- 一条慢 SQL 应该怎样从现象一路收束到执行计划和优化方向

---

## 1. 为什么一条 SQL 会慢

SQL 慢通常不是一个原因，而是下面几类问题叠加：

- 没有合适索引
- 有索引但没有被有效利用
- 扫描数据量太大
- 排序、分组、回表成本高
- `JOIN` 条件或过滤条件设计不合理
- 返回列过多，IO 成本本来就高

所以排查 SQL 慢，重点不是先拍脑袋改语句，而是先搞清数据库到底怎么执行。

---

## 2. 索引为什么能提速

索引的本质是：

- 帮数据库更快缩小扫描范围

而不是逐行做全表查找。

它通常更适合帮助下面这几类操作：

- `WHERE` 精确筛选
- 高频 `JOIN`
- 排序和范围查询
- 联合条件过滤

但索引不是万能的。  
如果查询本身就要返回大量数据，或者 SQL 写法让优化器难以使用索引，建了索引也未必会快。

---

## 3. 为什么建了索引也可能没用

这是实际排查和面试里最常见的问题之一。

典型原因包括：

- 在索引字段上做函数、表达式或隐式类型转换
- 联合索引没有按最左前缀使用
- 查询条件选择性太差
- 返回列太多，回表成本高
- 数据量太小，优化器判断全表扫描更划算
- SQL 写法本身不利于利用索引

关键不是背规则，而是理解：

- 索引必须和过滤方式、字段顺序、返回内容配合，数据库才愿意用

---

## 4. 联合索引到底怎么理解

联合索引本质上是多个字段的有序组合，例如：

```text
(user_id, status, create_time)
```

它更适合支持下面这些查询：

- `user_id`
- `user_id + status`
- `user_id + status + create_time`

而只按 `status` 或只按 `create_time` 查询时，通常无法完整发挥这个联合索引的价值。

最左前缀原则的本质就是：

- 索引本来就是按字段顺序组织的

字段顺序一旦不匹配，索引能力就会明显下降。

---

## 5. 覆盖索引和回表为什么重要

如果查询需要的字段在索引里已经都能拿到，数据库就不需要再回到数据页取整行，这通常会更快。

这就是覆盖索引。

反过来，如果：

- 过滤靠索引
- 但最终还得回表拿大量列

性能优势就会被削弱。

所以很多优化不是“有没有索引”的问题，而是：

- 这个索引是否同时支撑过滤和返回

---

## 6. `EXPLAIN` 最先看什么

`EXPLAIN` 用来查看数据库打算如何执行一条 SQL。  
初学阶段不用一口气看完所有字段，先抓四个最重要的：

- `type`
- `key`
- `rows`
- `Extra`

### `type`

表示访问方式，可以粗略理解为“扫描得粗还是细”。

常见值包括：

- `const`
- `ref`
- `range`
- `index`
- `ALL`

其中 `ALL` 往往意味着全表扫描风险较大。

### `key`

表示实际使用了哪个索引。

如果 `key = NULL`，通常说明没有用到有效索引。

### `rows`

表示优化器预估要扫描多少行。

一般来说，扫描行数越大，潜在代价越高。

### `Extra`

用于补充说明执行细节，常见需要重点关注：

- `Using index`
- `Using where`
- `Using filesort`
- `Using temporary`

其中：

- `Using filesort` 往往提示排序开销大
- `Using temporary` 往往提示临时结果成本高

---

## 7. 一个慢 SQL 的排查路径

一条更稳的排查路径通常是：

1. 先确认 SQL 是否真的慢，以及慢在什么场景
2. 用 `EXPLAIN` 看访问方式、索引使用和扫描行数
3. 判断问题更像是索引设计、SQL 写法还是数据量导致
4. 再决定是改索引、改 SQL、改返回列，还是改业务查询方式

这条路径的重点在于：

- 先看数据库怎么跑
- 再决定你该改哪里

如果没有执行计划支撑，优化很容易变成盲改。

---

## 8. 一个具体例子怎么分析

假设有一条查询：

```sql
SELECT *
FROM orders
WHERE user_id = 10001
  AND status = 'PAID'
ORDER BY create_time DESC
LIMIT 20;
```

如果表上有联合索引：

```text
(user_id, status, create_time)
```

那这条查询通常很适合走索引。

但如果你把条件改成：

```sql
WHERE DATE(create_time) = '2026-04-05'
```

就可能导致：

- 对索引字段做函数处理
- 索引难以有效使用

这就是为什么：

- “建了索引”不等于“写法就能用上索引”

---

## 9. 常见优化方向怎么判断

### 更像索引问题

常见特征：

- `key = NULL`
- `rows` 很大
- 过滤字段本来就适合建索引

### 更像 SQL 写法问题

常见特征：

- 对索引列做函数、计算、类型转换
- 联合索引使用顺序不合理
- 返回列过多

### 更像业务查询方式问题

常见特征：

- 本来就需要扫大量数据
- 分页过深
- 统计逻辑过重
- 一条 SQL 承担了过多业务职责

这时不一定只靠“再加一个索引”就能解决。

---

## 10. 常见误区

### 10.1 建了索引就觉得一定会快

索引要和 SQL 写法配合。

### 10.2 只看有没有索引，不看扫描行数和回表成本

这会导致判断过浅。

### 10.3 只背最左前缀，不理解字段顺序为什么重要

一旦换场景就不会分析。

### 10.4 看到 SQL 慢就立刻加索引

不先看执行计划，很容易盲改。

---

## 小结

索引与执行计划的核心，是理解数据库实际怎么执行 SQL。排查慢 SQL 时，先用 `EXPLAIN` 看访问方式、实际索引、扫描行数和执行细节，再判断问题落在索引设计、SQL 写法还是业务查询方式上。只有这样，优化才不是凭经验乱猜。
