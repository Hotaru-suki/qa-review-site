# 08 慢查询与排查基础

## 摘要

慢查询排查解决的是“到底是哪条 SQL 慢、为什么慢、应该先看哪里”这类实战问题。它强调的是排查路径，而不是只记几个性能术语。真正会排慢查询的人，不是上来就说“建索引”，而是先分清到底是执行慢、等待慢，还是 SQL 本身没问题但被锁、资源或数据量拖慢了。

---

## 核心问题

- 慢查询从定位到分析的基本路径是什么
- “执行慢”和“等待慢”为什么必须分开看
- 为什么慢查询经常和索引、数据量、排序、分页、锁等待一起出现

---

## 1. 什么叫慢查询

慢查询就是执行时间明显过长、已经影响接口响应、数据库压力或系统吞吐的 SQL。

“慢”没有绝对阈值，要结合：

- 业务要求
- 数据量
- 并发量
- 接口 SLA

一起判断。

例如：

- 后台报表跑 2 秒，可能还能接受
- 核心交易接口跑 2 秒，可能已经不可接受

所以慢查询判断首先要带业务背景。

---

## 2. 常见慢的原因有哪些

最常见的原因包括：

- 没有合适索引
- 索引没有被有效利用
- 扫描数据量过大
- 排序、分组代价高
- `JOIN` 设计不合理
- 返回数据过多
- 深分页
- 锁等待或资源竞争

但要注意，这些只是“常见原因清单”，不是排查顺序。

---

## 3. 一个更稳的排查顺序

遇到慢查询时，更稳的顺序通常是：

1. 先确认具体是哪条 SQL 慢
2. 判断是稳定复现还是偶发
3. 看 SQL 写法是否有明显问题
4. 看索引设计和执行计划
5. 看数据量和数据分布
6. 再看是否存在锁等待和并发冲突

这条顺序很重要，因为很多“慢”并不只是 SQL 写法问题。

---

## 4. 执行慢和等待慢到底怎么区分

这是慢查询排查里最容易被忽略的一步。

### 执行慢

执行慢指的是：

- SQL 本身要扫很多数据
- 排序很多数据
- 分组很多数据
- 计算代价很大

也就是说，数据库真正在“干活”时就很慢。

### 等待慢

等待慢指的是：

- SQL 本身不复杂
- 但它被锁、连接池、资源竞争卡住了

这时候你优化 SQL 文本本身，可能效果很小，因为真正瓶颈不在执行计划，而在等待资源。

所以慢查询排查第一件事之一，就是分清到底是哪一类慢。

---

## 5. 执行计划为什么重要

`EXPLAIN` 的价值，不是背几个字段名，而是帮你看清：

- 走没走索引
- 扫了多少行
- 有没有临时表
- 有没有文件排序
- JOIN 顺序和访问方式是否合理

如果一条查询看起来不复杂，却扫了非常多行，那大概率是：

- 过滤条件不够精准
- 索引设计不匹配
- 写法导致索引失效

---

## 6. 一个典型场景

例如：

```sql
SELECT * FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 100000, 20;
```

这条 SQL 常见慢点可能来自：

- `SELECT *` 取了过多列
- 深分页导致前面大量数据也要被跳过
- `ORDER BY` 需要排序
- 如果索引不合适，还可能全表扫描

这时真正排查时就不会只说“建索引”，而会继续问：

- 现有索引是什么
- 扫描行数多少
- 是否存在覆盖索引机会
- 是否应改分页方式

---

## 7. 一个更完整的排查场景

假设某个接口在高峰期超时，你定位到一条 SQL 很慢。

更稳的分析路径是：

1. 先确认慢查询日志里是不是这条 SQL
2. 判断它是一直慢，还是只有高峰期慢
3. 看 `EXPLAIN` 是否扫了很多行、是否走索引
4. 看数据量是否变大导致原写法不再适用
5. 如果执行计划还不错，再继续看是否有锁等待或资源争用

这样就不会在“SQL 看起来没问题”和“实际接口很慢”之间来回摇摆。

---

## 8. 常见优化方向

只有定位清楚之后，优化才不会跑偏。

常见方向包括：

- 建或调整高频索引
- 改写 SQL，让过滤更精准
- 避免不必要的 `SELECT *`
- 缩小返回列和返回行
- 优化排序与分页方式
- 拆分过重查询
- 降低锁冲突或调整并发策略

重点不是“优化手段多”，而是“先知道慢在哪”。

---

## 9. 常见误区

- 一看到慢就直接说“加索引”
- 不区分执行慢和等待慢
- 只看 SQL 文本，不看数据量和执行计划
- 不带业务 SLA 就说某 SQL 一定慢

慢查询排查最怕的是跳过定位，直接进入优化。

---

## 小结

慢查询排查最重要的不是背更多原因，而是形成“定位问题 -> 看执行方式 -> 判断瓶颈 -> 再优化”的稳定路径。只要先分清执行慢还是等待慢，再结合 SQL、索引、数据量和锁等待去分析，优化方向通常就不会偏。
