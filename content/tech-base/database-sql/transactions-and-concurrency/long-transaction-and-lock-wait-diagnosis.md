# 长事务与锁等待排查

## 这个知识点在解决什么问题
数据库并发问题里，最常见也最隐蔽的，不一定是死锁，而是长事务和锁等待。它们会让系统看起来“没挂，但越来越慢”，尤其在写多、状态更新频繁的业务里特别常见。

## 什么叫长事务
事务开始后，迟迟不提交或回滚，占着锁和资源不放。

它的影响包括：
- 阻塞别的事务
- 放大锁等待
- 增加回滚和恢复成本
- 拖慢整条业务链

## 锁等待在看什么
锁等待意味着：
- 你不是执行得慢
- 而是在等别人把资源释放出来

这类问题如果只看 SQL 文本，很容易误判成“语句太复杂”。

## 一个典型场景
### 库存扣减
- 事务 A 更新库存但长时间不提交
- 事务 B、C、D 都在等同一行锁

外部看起来像：
- 写接口 RT 变长
- 错误率不一定马上升高
- 但队列和长尾越来越差

## 排查时怎么想
1. 先确认慢的是执行还是等待
2. 再找谁持有锁
3. 再看持锁事务为什么这么久不结束

常见原因：
- 事务里做了太多逻辑
- 事务里混入外部调用
- 批量更新和联表操作拖太长

## 测试开发为什么要懂
压测时如果只盯 QPS 和平均耗时，很容易错过锁等待。  
但如果能结合：
- 写请求长尾
- 锁等待视图
- 热点业务键

就能更快判断是不是长事务在拖垮链路。

## 常见误区
- 看到慢写就先怪 SQL 复杂
- 只看死锁，不看长事务和锁等待
- 把事务范围设计得过大，却没意识到并发代价
