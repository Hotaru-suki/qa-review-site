# 查询变慢时如何建立排查路径

## 这个知识点在解决什么问题
SQL 变慢时，很多人第一反应是“加索引”。但真实问题可能是执行计划变了、数据量涨了、条件选择性变差、锁等待、回表太多，甚至根本不是数据库而是应用层拿连接太慢。排查路径的价值，是先判断慢在哪一层，再决定怎么改。

## 一条基础排查路径
```text
确认现象
  -> 确认是单条 SQL 慢还是整体请求慢
  -> 看执行计划
  -> 看数据量与条件选择性
  -> 看锁等待和事务阻塞
  -> 看资源瓶颈与上下游影响
```

## 第一步先问三个问题
- 是一直慢，还是最近才慢
- 是所有参数都慢，还是某些条件组合特别慢
- 是数据库执行慢，还是应用侧等待时间长

这一步决定你后面是看 SQL 语句、索引、事务，还是连接池和网络。

## 典型慢的原因
### 扫描范围变大
- 缺索引
- 索引选择性差
- where 条件命中不了索引

### 执行计划变差
- 统计信息过旧
- 优化器选择了不合适路径
- join 顺序改变

### 锁与事务阻塞
- 更新事务长时间不提交
- 热点行竞争
- 死锁重试

### 结果集过大
- 本来只该查 100 行，却拉了 10 万行回应用层

## 实际排查时要看什么
- `EXPLAIN` 看走了什么索引、预估扫描多少行
- 慢查询日志看发生时间和完整 SQL
- 事务视图或锁视图看是否被阻塞
- 业务侧监控看连接池、线程池、超时重试

## 一次排查最好怎么走
### 第一步：先分层
先区分到底是：
- 数据库执行真的慢
- 应用拿连接慢
- 网络传输慢
- 下游重试放大导致整体接口看起来慢

很多“SQL 慢”其实是接口慢，不是数据库内核在慢。

### 第二步：锁定问题 SQL
不要泛泛地说“查询很慢”，而要拿到：
- 完整 SQL
- 参数
- 发生时间
- 数据量
- 调用链上下文

### 第三步：看执行计划和数据分布
执行计划告诉你理论路径，数据分布决定理论路径是否合理。

### 第四步：看并发和阻塞
同一条 SQL 在单独执行时正常，不代表在高并发下正常。要继续确认：
- 是否被锁等待
- 是否连接池耗尽
- 是否同时间段有批量任务

## 一个例子
假设“查询用户订单列表”突然从 80ms 变到 2s：
- 先查慢日志，确认是这条 SQL 的数据库执行时间变长，还是接口总耗时变长
- 再看 `EXPLAIN` 是否仍走用户 ID + 时间的联合索引
- 再看最近数据量是否暴涨、分页是否拉太深
- 再看同时段是否有补偿任务在批量更新订单状态

这样最后才能得出是：
- 索引问题
- 数据量问题
- 锁等待问题
- 还是应用层连接等待问题

## 测试视角有什么不同
测试不是只证明“慢”，而是要把结论落到可复现条件上：
- 哪类 SQL
- 什么参数
- 什么数据量
- 什么并发条件
- 慢到什么程度
- 证据链指向哪里

这样开发和 DBA 才能高效处理。

## 测试报告里怎么写更有用
例如可以写成：
- 在 50 并发下，订单列表 SQL 在用户订单量 > 10 万时，P95 从 120ms 上升到 1.8s
- `EXPLAIN` 显示仍走索引，但扫描行数显著上升
- 同时间段存在订单状态批量更新任务，锁等待明显增加
- 当前判断根因更接近数据量扩大叠加锁竞争，而不是单纯缺索引
