# 排序、分组与临时表成本

## 这个知识点在解决什么问题
很多查询变慢，并不是因为过滤没走索引，而是排序、分组和中间临时结果代价太高。这个知识点解决的是：为什么一条 SQL 过滤条件看起来不重，但一加 `order by`、`group by` 就明显慢下来。

## 为什么排序和分组会贵
因为数据库可能需要：
- 先读出大量候选行
- 再做排序或分组
- 甚至创建临时表保存中间结果

如果候选集很大，代价就会快速上升。

## 什么叫临时表成本
临时表不是业务表，而是数据库在执行过程中为了组织中间结果额外创建的结构。

一旦出现大量临时结果：
- 内存消耗上升
- 可能落磁盘
- 整体查询时间变长

## 一个典型场景
你查最近 30 天的订单，并按某字段排序：
- 如果过滤后仍有几十万行
- 排序字段又没法利用索引

那即使 where 条件本身不复杂，排序也可能成为主要成本来源。

## 测试开发为什么要懂
你在排查慢 SQL 时，会经常遇到：
- 过滤看起来不重，但排序很慢
- 分组统计类 SQL 压测下明显退化
- 加一个排序条件后 RT 翻倍

这时就要想到是不是中间结果和排序成本在放大。

## 常见误区
- 只盯 where 条件，不看 order by / group by
- 以为“命中索引”就代表排序一定便宜
- 没有意识到中间结果可能比最终结果大很多
