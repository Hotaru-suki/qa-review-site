# 索引失效与扫描范围膨胀怎么判断

## 这个知识点在解决什么问题
慢 SQL 很多时候不是“没有索引”，而是“有索引但没用上”或者“虽然用了索引，但扫描范围还是太大”。判断索引失效和扫描范围膨胀，是排查性能问题的关键一步。

## 什么叫索引失效
索引失效指的是 SQL 本来有相关索引，但优化器最终没有有效利用它，或者只能利用一部分，导致回到大范围扫描。

## 常见原因
- 查询条件不满足最左前缀
- 对索引列做函数、计算或隐式类型转换
- 范围查询后面的列难以继续利用联合索引
- 选择性太差，优化器认为全表扫更便宜

## 什么叫扫描范围膨胀
即使走了索引，也可能发生：
- 命中了索引，但返回候选行太多
- 回表次数很多
- 排序和分组仍然要处理大结果集

这时你会看到“用了索引，但还是慢”。

## 怎么判断
### 看执行计划
- 访问类型
- 预估扫描行数
- 是否使用索引条件
- 是否有 `Using filesort`、`Using temporary`

### 看业务条件
- 时间范围是不是过大
- 状态值是不是低选择性字段
- 是否一次查了太多历史数据

## 一个真实排查过程
假设你有一条查询：

```sql
select *
from orders
where user_id = 1001
  and status = 'PAID'
  and created_at >= '2026-04-01'
order by created_at desc
limit 20;
```

排查时至少要看四件事：
1. 有没有和查询条件、排序方向匹配的联合索引
2. `EXPLAIN` 预估扫描了多少行
3. 是否出现回表很多、排序很多
4. `user_id=1001` 这类条件到底筛掉了多少数据

如果 `user_id` 下订单本来就很多，而 `status='PAID'` 占比也高，扫描范围仍可能很大。

## 怎么区分“没走索引”和“走了但仍然慢”
### 没走索引的典型信号
- 扫描类型偏全表
- 扫描行数很大
- SQL 对索引列做了函数或隐式转换

### 走了索引但仍然慢的典型信号
- 扫描行数仍然很高
- 回表很多
- 排序或分组在大结果集上进行
- 结果集传回应用层太大

这两类问题解决思路并不一样。

## 典型场景
用户表上有 `(status, created_at)` 联合索引，但查询用了：

```sql
where date(created_at) = '2026-04-05'
```

因为对列做了函数处理，索引利用率会变差。  
即使换成范围条件，如果 `status` 基本所有行都一样，扫描范围也仍然会很大。

另一个常见场景是：
- 联合索引有 3 列
- where 只稳定命中第一列，后两列过滤能力弱
- 看起来“有联合索引”，但其实真正筛掉的数据很少

## 常见误区
- 只要 `EXPLAIN` 显示用了索引，就觉得没问题
- 不看扫描行数和回表成本
- 不结合数据分布，只谈索引名字
- 只想“补一个索引”，不先明确查询真正慢在扫描、排序还是锁等待
