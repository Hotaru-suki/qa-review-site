# 事务、锁与并发高频题

## 这组问题通常在考什么
这类高频题不是单纯考名词，而是在考你是否知道：
- 为什么会有事务
- 并发时为什么会出错
- 锁和 MVCC 各自解决什么问题
- 出现阻塞和死锁时你会怎么判断

## 高频问题 1：为什么需要事务
回答重点：
- 多步操作要么一起成功，要么一起失败
- 核心是保证一致性，不是单纯“防止报错”
- 典型场景是转账、扣库存、支付成功后写多张表

一个更完整的答法可以是：
“事务本质上是把多步更新当成一个整体来管理，解决的是中间步骤成功、后面步骤失败时系统状态不一致的问题。比如支付成功后要写订单表、支付流水表、营销表，如果中途失败，事务能保证这些操作一起提交或一起回滚。”

## 高频问题 2：隔离级别在解决什么
回答重点：
- 并发事务同时读写会出现脏读、不可重复读、幻读
- 隔离级别是在一致性和并发性能之间做取舍
- 不同数据库的实现细节会不同，要把原则和实现分开说

继续追问时可以补一句：
“隔离级别越高，一般一致性越强，但并发代价通常更大，所以真实业务里不是一味追求最高隔离，而是看能接受什么并发异常。”

## 高频问题 3：锁和 MVCC 什么关系
回答重点：
- 锁偏向控制写冲突和互斥访问
- MVCC 偏向提升读并发，让读不用总被写阻塞
- 两者不是互斥关系，而是一起构成并发控制机制

更落地的说法是：
“写和写冲突通常仍要靠锁解决，读和写并发时很多数据库会通过 MVCC 让读看到一致快照，从而减少阻塞。”

## 高频问题 4：死锁怎么理解
回答重点：
- 多个事务互相等待对方释放资源
- 常见于更新顺序不一致、热点资源竞争
- 处理不只是“重启数据库”，而是统一访问顺序、缩短事务、降低锁粒度

如果面试官追问“遇到死锁怎么办”，可以按这个顺序答：
- 先确认死锁日志和涉及的 SQL
- 看事务持锁顺序和访问资源顺序
- 统一更新顺序
- 缩短事务和锁持有时间
- 对热点资源考虑拆分或降低竞争

## 面试时怎么避免答散
每题都尽量按这个顺序：
- 定义
- 解决的问题
- 代价或副作用
- 一个业务场景
- 一个排查或优化角度

## 常见追问怎么接
- 问事务：接 ACID、回滚、业务一致性
- 问隔离级别：接脏读、不可重复读、幻读和快照读
- 问锁：接行锁、表锁、锁等待、死锁
- 问 MVCC：接 undo log、版本链、快照可见性

## 一个更像面试现场的答法
### 问：事务为什么重要
可以答：
“事务的核心不是语法层面的 begin/commit，而是保证一组业务更新要么全部成功，要么全部失败。像支付成功后要写订单、支付流水、优惠核销，如果中间一步失败，没有事务就可能造成用户已扣款但订单状态没更新。”

### 问：锁和 MVCC 什么关系
可以答：
“写写冲突通常还是靠锁解决，MVCC 主要让读写并发时读请求不一定被写阻塞。它们不是二选一，而是一起构成并发控制能力。”

### 问：怎么回答才不会显得只会背
最好补上：
- 真实业务场景
- 代价和副作用
- 如果出问题你会怎么排查
