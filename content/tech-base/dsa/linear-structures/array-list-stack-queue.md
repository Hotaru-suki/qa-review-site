# 数组、链表、栈与队列怎么区分

## 摘要

这些结构表面上都像“存一串元素”，但真正决定它们差异的，不是名字，而是底层组织方式和高频操作成本。数组靠连续内存换来快速随机访问，链表靠节点链接换来局部插删灵活；栈和队列则是在这些线性结构上进一步限制访问方式，让某类操作更自然。

---

## 核心问题

- 数组和链表在底层到底怎么存
- 插入、删除、访问分别为什么会快或慢
- 栈和队列为什么本质上是“操作受限的线性结构”
- 选型时应该先看什么操作模式

---

## 1. 数组为什么适合随机访问

数组的关键前提是：元素在内存中连续存放。

可以把它想成：

```text
index:   0    1    2    3    4
value:  [7] [12] [5] [9] [3]
```

如果已知数组起始地址和每个元素大小，那么访问 `arr[i]` 时，可以直接通过地址偏移算出目标位置：

```text
address(arr[i]) = base + i * element_size
```

所以数组按下标访问的时间复杂度通常是 `O(1)`。

这就是数组最大的优势：随机访问非常快。

---

## 2. 数组为什么中间插删会慢

假设数组里有：

```text
[10, 20, 30, 40]
```

如果要在下标 `1` 插入 `15`，为了保持连续存储，后面的元素都得后移：

```text
原来: [10, 20, 30, 40]
插入: [10, 15, 20, 30, 40]
```

`20`、`30`、`40` 都要搬位置。

所以数组在中间插入或删除的时间复杂度通常是 `O(n)`。

这也解释了为什么数组适合“读多改少”的场景，而不适合频繁中间插删。

---

## 3. 动态数组和普通数组有什么区别

很多语言里的 `list`、`vector` 并不是固定长度数组，而是动态数组。

它们在逻辑上允许追加元素，但底层仍然依赖连续内存。常见策略是：

- 容量没满时，直接尾部追加
- 容量满了时，申请更大空间
- 把旧元素整体搬过去

所以：

- 尾部追加平均复杂度通常接近 `O(1)`
- 但触发扩容时，单次可能是 `O(n)`

这也是“摊还复杂度”的典型场景。

---

## 4. 链表为什么适合局部插删

链表不要求连续内存，它靠节点之间的指针连接。

单链表可以直观看成：

```text
[10 | next] -> [20 | next] -> [30 | next] -> null
```

如果你已经拿到了某个位置的前驱节点，那么插入一个新节点通常只需要改几个指针：

```text
原来:
A -> B -> C

插入 X 之后:
A -> X -> B -> C
```

这类局部操作本身通常是 `O(1)`。

但问题在于：如果你不知道这个位置在哪，就得先顺着链表一路找过去，这部分又会回到 `O(n)`。

所以链表真正擅长的是：

- 已知位置附近的插删
- 不强调随机访问

---

## 5. 链表为什么随机访问慢

数组访问第 `i` 个元素时，可以直接算地址。

链表不行。因为节点不连续，只能从头一个个顺过去：

```text
head -> node1 -> node2 -> node3 -> ...
```

要访问第 `k` 个节点，就得走 `k` 步左右，因此访问复杂度通常是 `O(n)`。

这就是链表和数组最根本的差别：

- 数组快在“按下标定位”
- 链表快在“局部改链接”

---

## 6. 栈为什么是后进先出

栈本质上不是全新底层结构，而是一种访问约束：只允许在一端插入和删除。

字符图可以理解成：

```text
top
 |
[30]
[20]
[10]
```

新元素压入顶部：

```text
push(40)

top
 |
[40]
[30]
[20]
[10]
```

删除时也只能从顶部弹出：

```text
pop() -> 40
```

因此栈是后进先出，LIFO。

它适合：

- 函数调用栈
- 撤销回退
- 括号匹配
- 深度优先搜索

如果底层用动态数组或链表实现，`push` / `pop` 通常都可以做到接近 `O(1)`。

---

## 7. 队列为什么是先进先出

队列也是访问受限结构，但规则变成：

- 尾部入队
- 头部出队

字符图：

```text
front -> [10] [20] [30] <- rear
```

执行：

```text
enqueue(40)
front -> [10] [20] [30] [40] <- rear

dequeue() -> 10
front -> [20] [30] [40] <- rear
```

这就是先进先出，FIFO。

它适合：

- 任务排队
- 消息处理
- 广度优先搜索
- 生产者消费者模型

---

## 8. 时间复杂度怎么比较

| 结构 | 按下标访问 | 头部插删 | 中间插删 | 尾部追加 | 典型优势 |
| --- | --- | --- | --- | --- | --- |
| 数组 | `O(1)` | `O(n)` | `O(n)` | 平均 `O(1)` | 随机访问 |
| 链表 | `O(n)` | `O(1)` | 已知位置时 `O(1)`，查找位置后总体常为 `O(n)` | 若有尾指针可 `O(1)` | 局部插删 |
| 栈 | 不强调 | `O(1)` | 不支持 | `O(1)` | 最近元素优先 |
| 队列 | 不强调 | 出队 `O(1)` | 不支持 | 入队 `O(1)` | 先进先出 |

这里要注意：

- 栈和队列不是拿来做随机访问的
- 链表“插入快”不等于“任何情况下都快”
- 动态数组“尾插快”是平均意义，不是每次都绝对 `O(1)`

## 9. 一个简单实现示例

### 用数组实现栈

```python
class Stack:
    def __init__(self):
        self.data = []

    def push(self, x):
        self.data.append(x)

    def pop(self):
        return self.data.pop()

    def top(self):
        return self.data[-1]
```

### 用链表实现队列

```python
class Node:
    def __init__(self, value):
        self.value = value
        self.next = None


class Queue:
    def __init__(self):
        self.head = None
        self.tail = None

    def enqueue(self, x):
        node = Node(x)
        if not self.tail:
            self.head = self.tail = node
        else:
            self.tail.next = node
            self.tail = node

    def dequeue(self):
        if not self.head:
            return None
        val = self.head.value
        self.head = self.head.next
        if not self.head:
            self.tail = None
        return val
```

这两段代码重点不是实现技巧，而是帮助理解：栈在操作顶部，队列在操作头尾。

## 10. 一个工程或测试场景

例如你在做日志处理：

- 如果只是按顺序读取并按下标访问某些位置，数组或列表更自然
- 如果你要维护待处理任务队列，队列更自然
- 如果你要处理撤销操作或解析括号嵌套，栈更自然
- 如果你要频繁在中间插入并删除节点，而且已持有位置引用，链表才更有优势

结构选型永远要回到“我最频繁做什么操作”。

---

## 小结

数组、链表、栈、队列的差别，不是它们都能不能存一串元素，而是它们分别把“访问、插删、顺序约束”优化到了不同方向。只要底层组织方式和高频操作模式想清楚，选型就不会停留在死记定义。
