# 时间复杂度与空间复杂度怎么理解

## 摘要

复杂度不是数学装饰，而是描述“输入规模变大后，算法代价如何增长”的语言。它回答的不是“这次代码跑了几毫秒”，而是“如果数据从 100 条涨到 10 万条，这个解法还能不能撑住”。真正会用复杂度的人，不只是会写 `O(n log n)`，而是能把循环层数、数据结构选择和额外空间代价翻译成实际性能判断。

---

## 核心问题

- 复杂度为什么看趋势而不是某次具体耗时
- 时间复杂度和空间复杂度分别在描述什么
- 为什么同样正确的代码，复杂度不同会导致完全不同的可用性

---

## 1. 为什么复杂度关注“增长趋势”

同一段代码在不同机器、不同语言、不同输入分布下，实际毫秒数都会变。

但复杂度关注的是更稳定的东西：

- 输入规模变大时，操作次数怎么增长
- 额外使用的空间怎么增长

例如：

- 一次循环通常对应 `O(n)`
- 双重嵌套循环常见是 `O(n^2)`
- 每次把问题规模折半，常见是 `O(log n)`

这类增长趋势，比一次跑了多少毫秒更能说明解法是否可扩展。

---

## 2. 时间复杂度到底在看什么

时间复杂度本质上是在估算“关键操作的执行次数”。

例如：

```python
for x in nums:
    print(x)
```

如果 `nums` 有 `n` 个元素，循环执行 `n` 次，所以时间复杂度是 `O(n)`。

再例如：

```python
for i in nums:
    for j in nums:
        if i == j:
            ...
```

外层 `n` 次，内层每次又 `n` 次，总量接近 `n * n`，所以常见是 `O(n^2)`。

要点不是死背公式，而是看“随着输入规模增长，核心操作被放大了几层”。

---

## 3. 空间复杂度又在看什么

空间复杂度关注的是“额外占用了多少空间”，不是程序总内存，而是为了解题额外申请的辅助空间。

例如：

```python
seen = set()
for x in nums:
    seen.add(x)
```

如果 `nums` 里有 `n` 个不同元素，`seen` 可能会存下 `n` 个值，所以额外空间复杂度通常是 `O(n)`。

而如果只用了几个固定变量：

```python
max_val = nums[0]
for x in nums:
    max_val = max(max_val, x)
```

额外变量数量不随 `n` 增长，空间复杂度通常是 `O(1)`。

---

## 4. 为什么同样正确，复杂度却会决定能不能用

例如两种找重复元素的方法：

### 方法一：双重循环

```python
def has_duplicate(nums):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] == nums[j]:
                return True
    return False
```

时间复杂度通常是 `O(n^2)`，额外空间 `O(1)`。

### 方法二：哈希集合

```python
def has_duplicate(nums):
    seen = set()
    for x in nums:
        if x in seen:
            return True
        seen.add(x)
    return False
```

时间复杂度平均 `O(n)`，额外空间 `O(n)`。

两者都正确，但当 `n` 很大时，差异会非常明显。

这就是复杂度真正的意义：帮助你判断“正确答案”是否具有现实可用性。

---

## 5. 常见复杂度图景怎么建立直觉

### `O(1)`

数据规模变大，操作次数基本不变。

典型例子：

- 数组按下标访问
- 栈顶 `push/pop`

### `O(log n)`

每次都能排除一大半范围。

典型例子：

- 二分查找
- 平衡树查找

### `O(n)`

需要把数据大致扫一遍。

典型例子：

- 遍历数组
- 用哈希统计频次

### `O(n log n)`

比线性慢，但通常仍可接受，是很多高效排序和分治算法常见量级。

### `O(n^2)`

两层数据规模相关循环叠在一起，输入一大很容易失控。

典型例子：

- 暴力两两比较

---

## 6. 如何从代码里看出复杂度

### 看循环层数

最直观的方法是先看有没有多层嵌套循环。

### 看递归规模如何缩小

如果每次递归把问题规模减半，常见是 `O(log n)` 或 `O(n log n)`；如果每次只减 1，往往更接近线性或更高。

### 看是否引入了额外数据结构

如果你引入：

- 哈希表
- 额外数组
- 队列
- 递归调用栈

就要考虑空间复杂度是不是随输入增长。

### 看某个操作本身是不是常数时间

例如：

- 哈希查找平均接近 `O(1)`，但不是绝对
- 列表头插可能是 `O(n)`
- 堆插入是 `O(log n)`

复杂度不是只看你写了几个循环，还要看每个操作本身代价。

## 7. 一个测试或工程里的直觉例子

复杂度不只出现在算法题里，也会出现在测试工具和业务代码里。

例如你要在 10 万条日志中找重复请求 ID：

### 暴力做法

每来一条日志，就和之前所有日志比一遍。

如果有 `n` 条日志，总比较次数会接近：

```text
1 + 2 + 3 + ... + (n - 1) ≈ O(n^2)
```

### 更合理做法

用一个 `set` 记录已出现的请求 ID：

```python
seen = set()
for req_id in req_ids:
    if req_id in seen:
        ...
    seen.add(req_id)
```

这样时间复杂度平均能降到 `O(n)`，代价是额外空间 `O(n)`。

这正是“用空间换时间”的典型例子。

---

## 8. 复杂度分析时常见误区

- 只盯一次运行时间，不看规模增长
- 只看自己写了几层循环，不看底层操作成本
- 忽略额外数据结构带来的空间代价
- 把平均复杂度误当成最坏复杂度

例如哈希查找常说是 `O(1)`，但如果冲突严重，最坏情况并不是真正常数时间。

---

## 9. 为什么测试和工程里也需要复杂度意识

如果没有复杂度意识，很容易写出：

- 小数据测试完全正常，大数据直接卡死的脚本
- 查日志、聚合数据、比对结果时低效的工具代码
- 看起来功能正确，但线上一放量就明显变慢的逻辑

复杂度意识并不要求每次都做严格数学证明，但至少要能提前判断：

- 这个解法会不会随着数据量膨胀
- 有没有必要换数据结构
- 有没有必要用空间换时间

---

## 小结

复杂度是算法选型和代码实现的基础判断标准。它真正帮助你的，不是写出几个符号，而是让你在看到代码结构、数据结构和输入规模时，能迅速判断这段实现将来是“还能撑住”，还是“规模一大就失效”。
