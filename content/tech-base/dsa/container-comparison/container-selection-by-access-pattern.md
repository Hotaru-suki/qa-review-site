# 按访问模式选择容器

## 这个知识点在解决什么问题
容器选型不应该靠背名字，而应该先看访问模式。你到底是在做随机访问、头尾插入、中间删除、去重查找，还是按优先级取最值，不同模式决定了底层组织方式和复杂度。

## 先问自己几个问题
- 元素是主要按下标访问，还是按键查找
- 插入删除主要发生在头部、尾部，还是中间
- 是否需要去重
- 是否需要保持顺序
- 是否经常拿最小值/最大值

## 常见选择路径
```text
频繁随机访问 -> 数组 / vector / list 不合适
头尾频繁插入删除 -> deque / 双端队列
只在一端进出 -> stack / queue
按键快速查找 -> hash map / unordered_map
需要有序查找 -> tree map / map / set
频繁取最值 -> heap / priority queue
```

## 选型时顺手带上复杂度
| 需求模式 | 更常见的结构 | 关键操作 | 常见复杂度 |
| --- | --- | --- | --- |
| 按下标随机访问 | array / vector | `a[i]` | `O(1)` |
| 尾部追加 | vector / list | `push_back` | vector 摊还 `O(1)` |
| 头部插入删除 | deque | `push_front/pop_front` | `O(1)` |
| 按键查找 | hash map / unordered_map | `find` | 平均 `O(1)` |
| 有序查找 | map / set | `lower_bound` | `O(log n)` |
| 频繁取最大/最小 | heap | `top/pop` | `O(1)` / `O(log n)` |

这里的重点不是死背表，而是知道“为什么这个结构的复杂度适合当前访问模式”。

## 具体例子
### 日志批量收集
数据不断追加，偶尔遍历输出：
- 更像 `vector` 或 Python `list`
- 因为尾插和顺序遍历很常见

### 消息消费队列
一端入队、一端出队：
- 更像 `queue`
- 如果还要支持两端插入删除，可以考虑 `deque`

### 用户 ID 去重
核心操作是“是否已存在”：
- 更像 `set` 或哈希集合
- 不是数组，因为线性查找太慢

### 定时任务取最近到期任务
频繁取最小值：
- 更像最小堆
- 不是普通数组或链表

### 聊天消息窗口
既要按顺序追加，也可能只保留最近 N 条：
- 更像 `deque` 或环形缓冲区
- 不一定适合链表，因为大多数时候仍然需要顺序遍历和缓存友好性

### 排行榜前 K 名
- 如果是持续插入并只关心 TopK，堆很常见
- 如果需要全量有序遍历，可能更适合平衡树或排序后的数组

## 为什么这比背复杂度更重要
复杂度表只是抽象结果，访问模式才决定实际使用场景。很多人知道 `list` 插入删除是 O(1)，却忘了前提是你已经拿到了那个位置；如果为了找到位置要遍历，整体成本仍然很高。

## 还要考虑哪些非复杂度因素
- 内存占用和缓存友好性
- 是否允许重复元素
- 是否需要稳定迭代器/引用
- 库语言义是否易错，例如哈希冲突、扩容失效、迭代器失效

例如 `vector` 在遍历性能上通常很好，因为连续内存对 CPU cache 更友好；这也是为什么很多场景下即使中间插入不优，也仍然优先考虑 `vector`。

## 真正选型时可以这样问自己
- 读多还是写多
- 是头尾操作还是中间操作
- 有没有按键查找
- 结果是否必须有序
- 数据量大概多大

如果这五个问题能回答清楚，容器选型通常就不会太偏。

## 常见误区
- 只因“插删快”就选链表
- 需要有序结果却选哈希表
- 需要按键查找却用数组硬扫
- 需要取 TopK 却先全部排序
- 只看单次操作复杂度，不看整体访问模式和数据规模
