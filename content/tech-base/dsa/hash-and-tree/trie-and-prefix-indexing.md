# Trie 与前缀索引

## 这个知识点在解决什么问题
哈希表擅长精确查找，但不擅长前缀关系。Trie 和前缀索引解决的是：当你关心“是不是以某个前缀开头”“有哪些字符串共享前缀”时，怎样比普通哈希查找更自然地组织数据。

## Trie 在表达什么
Trie 可以看成一棵按字符展开的树：

```text
root
 └─ a
    ├─ p
    │  └─ p
    └─ r
```

它特别适合：
- 字典匹配
- 自动补全
- 敏感词前缀过滤
- 路由或命令前缀匹配

## 为什么这和哈希不同
哈希更适合完整 key 命中。  
Trie 更适合利用“公共前缀”这件事。

## 一个简单例子
如果你要判断：
- `/api/user/*`
- `/api/order/*`
- `/api/admin/*`

这类路径前缀关系，用 Trie 或前缀树思维更自然；  
而哈希更适合判断“这个完整路径是不是存在”。

## 测试开发里的实际落点
- 接口路径按前缀归类
- 日志关键字树状匹配
- 命令补全
- 配置项前缀提示

## 测试开发里的例子
- 一批接口路径按前缀归类
- 日志关键字按前缀快速匹配
- 命令补全或参数提示

## 常见误区
- 遇到字符串查找就只想到哈希表
- 不区分“完整匹配”和“前缀匹配”
- 只会背 Trie 的定义，不知道它为什么适合前缀场景
- 把前缀问题也硬塞进普通哈希查找里，后续逻辑越来越绕
