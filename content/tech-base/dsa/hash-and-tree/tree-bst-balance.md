# 树、二叉搜索树与平衡性的意义

## 摘要

树结构的价值，不只是“有父子节点”，而是它能把数据组织成层次关系，并在某些规则下支持高效查找。二叉搜索树进一步用“左小右大”的规则把查找、插入、删除控制在和树高相关的复杂度上，而平衡性的意义，就是防止这棵树退化成链表，导致性能从接近 `O(log n)` 掉回 `O(n)`。

---

## 核心问题

- 树为什么天然适合表达层级关系
- 二叉搜索树是怎么利用大小关系组织查找的
- 平衡性为什么会直接影响查找、插入、删除效率

---

## 1. 树和线性结构的根本区别

数组、链表、栈、队列都是线性结构，元素之间主要是前后关系。

树不一样，它更强调层级和分叉关系：

```text
          A
        /   \
       B     C
      / \     \
     D   E     F
```

这种结构非常适合表达：

- 文件目录
- 组织架构
- UI 组件树
- 表达式语法树

也就是说，树的第一层价值是“组织层次”。

---

## 2. 二叉搜索树到底是什么

二叉搜索树，BST，额外增加了一条规则：

- 左子树所有值都比当前节点小
- 右子树所有值都比当前节点大

例如：

```text
        8
      /   \
     3     10
    / \      \
   1   6      14
      / \     /
     4   7   13
```

这棵树的意义在于，每次比较后，你都可以直接排除一半方向：

- 目标比当前值小，就去左边
- 目标比当前值大，就去右边

因此 BST 的核心价值不是“长得像树”，而是“用大小关系指导查找路径”。

---

## 3. 查找、插入、删除分别怎么做

### 查找

例如要找 `7`：

```text
先看 8，7 < 8，往左
再看 3，7 > 3，往右
再看 6，7 > 6，往右
找到 7
```

查找路径长度取决于树高。

### 插入

插入逻辑和查找类似：

1. 从根开始比较
2. 小于当前值就往左
3. 大于当前值就往右
4. 走到空位置时插入

### 删除

删除稍复杂，通常分三种情况：

- 叶子节点：直接删
- 只有一个孩子：让孩子顶上来
- 有两个孩子：通常找中序后继或前驱替代

例如删掉 `8` 时，不能直接把整棵树断开，通常会找右子树中的最小值或左子树中的最大值来补位。

---

## 4. 为什么树高这么关键

BST 的性能本质上和树高 `h` 有关：

- 查找 `O(h)`
- 插入 `O(h)`
- 删除 `O(h)`

如果树比较平衡，树高通常接近 `log n`，那操作复杂度就比较好。

但如果树严重倾斜，例如不断插入递增序列：

```text
1
 \
  2
   \
    3
     \
      4
       \
        5
```

这时候 BST 几乎退化成链表，查找、插入、删除都会退化到 `O(n)`。

所以平衡性的意义，本质上是在防止结构退化。

---

## 5. 平衡树到底在解决什么

平衡树不是为了“让树看起来更好看”，而是为了让树高始终控制在较低水平。

它解决的问题是：

- 普通 BST 的性能高度依赖插入顺序
- 一旦输入数据有序，树很容易退化
- 退化后有序查找优势几乎消失

所以像 AVL 树、红黑树这类平衡树，会在插入删除后通过旋转等操作调整结构，让树不要长期偏斜。

你不一定要手写这些树，但一定要理解它们存在的原因：稳定复杂度。

---

## 6. 时间复杂度怎么看

| 结构 | 查找 | 插入 | 删除 | 说明 |
| --- | --- | --- | --- | --- |
| 普通 BST，理想情况下 | `O(log n)` | `O(log n)` | `O(log n)` | 树较平衡 |
| 普通 BST，最坏情况下 | `O(n)` | `O(n)` | `O(n)` | 退化成链 |
| 平衡树 | `O(log n)` | `O(log n)` | `O(log n)` | 通过调整保持树高 |

空间复杂度通常是 `O(n)`，因为需要存所有节点。

---

## 7. 一个简单实现示意

下面给一个极简 BST 插入和查找的 Python 示例：

```python
class Node:
    def __init__(self, val):
        self.val = val
        self.left = None
        self.right = None


def insert(root, x):
    if root is None:
        return Node(x)
    if x < root.val:
        root.left = insert(root.left, x)
    elif x > root.val:
        root.right = insert(root.right, x)
    return root


def find(root, x):
    if root is None:
        return False
    if x == root.val:
        return True
    if x < root.val:
        return find(root.left, x)
    return find(root.right, x)
```

这段代码足够说明 BST 的核心：每次通过比较把搜索空间缩到一个子树里。

---

## 8. BST 和哈希表怎么选

这两个结构都能做查找，但侧重点不同。

哈希表更适合：

- 快速判断存在性
- key 到 value 的直接映射
- 不在乎顺序

BST 或平衡树更适合：

- 需要有序遍历
- 需要范围查询
- 需要找前驱、后继、最接近值

例如：

- 判断请求 ID 是否出现过，更适合哈希
- 维护一组有序时间点并频繁查区间，更适合树

---

## 9. 一个工程场景

比如你在做排行榜、时间区间事件、调度窗口等问题：

- 如果只想快速查某个 key 在不在，哈希很自然
- 如果要维护顺序、查某个范围内所有元素、找第一个大于某值的元素，树更自然

这时“树”真正有价值的地方，不只是能查找，而是它同时保留了结构上的有序关系。

---

## 小结

学树结构时，重点不是只记遍历顺序，而是看懂三件事：树为什么适合表达层级，BST 为什么能利用比较关系缩小查找范围，平衡性为什么决定性能能否稳定。把这三点想清楚，树就不再只是抽象定义，而是清楚知道它在解决什么问题。
