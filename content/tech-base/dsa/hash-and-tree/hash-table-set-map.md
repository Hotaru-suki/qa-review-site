# 哈希表、集合和映射的核心思想

## 摘要

哈希结构的核心价值，是把“顺序扫描找目标”变成“先算位置，再在局部确认目标”。这让查找、插入、删除在平均情况下都可以接近 `O(1)`。但这不是白来的，它依赖额外空间、哈希函数、冲突处理和扩容策略。`set` 和 `map` 只是建立在这种按键定位思想上的两种不同语义。

---

## 核心问题

- 哈希表到底是怎么把 key 映射到位置上的
- 插入、查找、删除的过程分别是什么
- 冲突为什么一定会发生，常见处理方式有哪些
- 集合和映射在底层和语义上分别表达什么

---

## 1. 哈希表的底层结构长什么样

可以先把哈希表理解成“数组 + 哈希函数 + 冲突处理”。

最外层通常是一段桶数组：

```text
index:   0      1      2      3      4      5      6      7
bucket: [ ]    [ ]    [ ]    [ ]    [ ]    [ ]    [ ]    [ ]
```

当我们插入一个 key 时，不是从头找空位，而是先通过哈希函数算出一个位置：

```text
index = hash(key) % capacity
```

例如：

```text
hash("apple") % 8 = 3
hash("banana") % 8 = 6
hash("grape") % 8 = 3
```

那它们在桶数组里的直觉位置可能变成：

```text
index:   0      1      2      3                 4      5      6         7
bucket: [ ]    [ ]    [ ]   [apple, grape]    [ ]    [ ]   [banana]   [ ]
```

这里 `"apple"` 和 `"grape"` 同时落到 `3` 号桶，这就叫冲突。

---

## 2. 为什么哈希表平均能很快

普通数组里找一个元素，如果没有额外结构，通常只能线性扫描，复杂度是 `O(n)`。

哈希表的思路是：

1. 先用哈希函数把 key 转成整数
2. 再映射到桶位置
3. 只在该桶对应的小范围里确认目标

如果哈希分布足够均匀，单个桶里的元素数量通常不多，那么查找、插入、删除的平均复杂度都能接近 `O(1)`。

这里的关键是“平均”。如果大量 key 都撞到同一个桶里，局部就会退化，性能也会变差。

---

## 3. 插入、查找、删除到底怎么做

### 查找

查找 `key` 的过程通常是：

1. 计算 `hash(key)`
2. 映射到某个桶
3. 在桶中逐个比对 key 是否相等

伪代码：

```python
def find(table, key):
    idx = hash(key) % len(table)
    bucket = table[idx]
    for k, v in bucket:
        if k == key:
            return v
    return None
```

### 插入

插入过程通常是：

1. 先算位置
2. 去对应桶里看 key 是否已存在
3. 存在则更新，不存在则追加

### 删除

删除过程通常是：

1. 算位置
2. 到对应桶中定位 key
3. 删除该节点或标记为空

这三类操作的共同点是：都先按哈希定位，再在局部处理。

---

## 4. 冲突为什么一定会发生

冲突不是异常，而是哈希表的常态。

原因很简单：

- key 的取值空间通常非常大
- 桶数组长度通常远小于 key 空间

把无限多或非常多的 key 映射到有限桶里，必然会有不同 key 落到同一个桶。

所以哈希表设计不是“如何避免冲突”，而是“冲突发生后如何高效处理”。

---

## 5. 常见冲突处理方式

### 方式一：拉链法

拉链法的意思是，每个桶里不只放一个元素，而是放一个链表或小型容器。

字符图可以这样看：

```text
index 0 -> empty
index 1 -> (tom, 18) -> (sam, 23)
index 2 -> empty
index 3 -> (apple, 5)
index 4 -> (grape, 8) -> (peach, 2)
```

优点：

- 实现直观
- 删除方便
- 装载更多元素时不需要严格连续空位

缺点：

- 如果某桶元素过多，局部查找会变慢

### 方式二：开放寻址

开放寻址不在桶里挂链，而是冲突后继续找下一个可用位置。

例如线性探测：

```text
index:   0      1      2      3      4      5      6      7
bucket: [ ]  [apple] [grape] [ ]    [ ]    [ ]  [banana] [ ]
```

假设 `"apple"` 算到 `1`，`"grape"` 也算到 `1`，那就向后找：

- `1` 被占用
- 看 `2`
- `2` 空着，就放进去

优点：

- 数据更紧凑，缓存友好

缺点：

- 删除更麻烦
- 容易形成聚集，影响查找效率

---

## 6. 扩容为什么重要

如果不断插入元素，桶越来越满，冲突概率也会越来越高。

这时通常会维护一个装载因子：

```text
load_factor = 元素个数 / 桶数量
```

当装载因子过高时，就需要扩容并重新哈希。

例如从 8 个桶扩到 16 个桶后，原来很多撞在一起的 key 可能会被重新分散：

```text
旧表: hash(key) % 8
新表: hash(key) % 16
```

扩容代价通常是 `O(n)`，因为要把所有已有元素重新分布到新桶里。但它不是每次插入都发生，所以摊还下来，插入平均复杂度仍然可以接近 `O(1)`。

---

## 7. 时间复杂度和空间复杂度怎么看

以常见的拉链法哈希表为例：

| 操作 | 平均时间复杂度 | 最坏时间复杂度 | 说明 |
| --- | --- | --- | --- |
| 查找 | `O(1)` | `O(n)` | 冲突集中时会退化 |
| 插入 | `O(1)` | `O(n)` | 扩容或严重冲突时变慢 |
| 删除 | `O(1)` | `O(n)` | 先定位再删除 |
| 空间 | `O(n)` | `O(n)` | 需要桶数组和元素存储 |

为什么最坏会到 `O(n)`？

因为如果所有 key 都哈到同一个桶，哈希表就会局部退化成链表或顺序结构。

---

## 8. `set` 和 `map` 分别是什么

### `set`

集合更强调“元素是否存在”。

例如：

- 判断用户 ID 是否重复
- 判断请求 ID 是否已经处理过
- 去重

底层通常还是哈希表，但只存 key，不存额外 value。

### `map`

映射更强调“key 对应什么 value”。

例如：

- 用户 ID -> 用户名
- 订单号 -> 订单状态
- 单词 -> 出现次数

它底层也是按 key 做哈希定位，只是桶里存的是 `(key, value)` 对。

## 9. 一个简化实现示例

下面用 Python 写一个非常简化的拉链法哈希表，帮助理解过程：

```python
class SimpleHashMap:
    def __init__(self, capacity=8):
        self.capacity = capacity
        self.buckets = [[] for _ in range(capacity)]

    def _index(self, key):
        return hash(key) % self.capacity

    def put(self, key, value):
        idx = self._index(key)
        bucket = self.buckets[idx]
        for i, (k, _) in enumerate(bucket):
            if k == key:
                bucket[i] = (key, value)
                return
        bucket.append((key, value))

    def get(self, key):
        idx = self._index(key)
        for k, v in self.buckets[idx]:
            if k == key:
                return v
        return None

    def remove(self, key):
        idx = self._index(key)
        bucket = self.buckets[idx]
        for i, (k, _) in enumerate(bucket):
            if k == key:
                bucket.pop(i)
                return True
        return False
```

这段代码没有处理扩容、哈希攻击、线程安全等工程细节，但足够说明哈希表“先算桶，再在桶中处理”的基本思想。

## 10. 什么时候不该优先想到哈希

哈希结构通常不擅长：

- 范围查询
- 有序遍历
- 求前驱后继
- 按大小关系组织数据

如果问题要求：

- 找最小/最大且持续维护顺序
- 查某个区间内所有元素
- 需要稳定有序输出

树结构、排序数组或堆可能更自然。

---

## 11. 一个测试或工程场景

例如你要在 10 万条日志里判断有没有重复请求 ID。

如果每来一条都和之前所有请求比一遍，时间复杂度接近 `O(n^2)`。

如果改成：

1. 新请求 ID 先查 `set`
2. 不存在就插入
3. 已存在就记为重复

整个过程就可以压到接近 `O(n)`。

这就是哈希结构在“快速判断存在性”和“快速建立映射关系”场景里的核心价值。

---

## 小结

哈希表真正解决的是“如何快速按 key 定位数据”。集合和映射只是这套思想上的两种不同语义。理解哈希时，重点不在于会背容器名，而在于真正看懂：桶数组怎么组织、冲突怎么处理、扩容为什么必要、复杂度为什么是平均 `O(1)` 而不是永远 `O(1)`。
