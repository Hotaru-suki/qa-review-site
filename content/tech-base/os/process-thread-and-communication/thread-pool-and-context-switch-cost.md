# 线程池与上下文切换成本

## 这个知识点在解决什么问题
很多人知道“线程太多不好”，但不知道坏在哪里。线程池与上下文切换成本解决的是：为什么线程不是越多越快，为什么大量线程反而会让系统更慢。

## 线程池在解决什么
线程池的核心价值是复用线程，避免每个任务都频繁创建和销毁线程。

它通常负责：
- 控制并发数量
- 缓冲任务
- 统一调度执行

## 上下文切换成本是什么
线程切换不是免费的。每次切换都可能带来：
- 寄存器和执行状态保存/恢复
- CPU cache 命中下降
- 调度开销增加

线程一多，系统可能把大量时间花在切换上，而不是花在真正做事上。

## 一个典型现象
- 线程数不断加
- CPU 使用率很高
- 吞吐却没有提升，甚至下降

这时问题不一定是业务逻辑慢，而可能是切换成本和锁竞争把收益吃掉了。

## 为什么线程池不能只看“线程数”
线程池至少还涉及：
- 核心线程数
- 最大线程数
- 队列长度
- 拒绝策略

如果只把最大线程数拉很大，常见结果是：
- 大量线程争抢 CPU
- 队列堆积
- 上下文切换急剧上升

## 一个测试开发场景
例如自动化执行器同时跑很多任务：
- 任务数翻倍后，线程数也暴涨
- CPU 很高，但任务完成时间反而更久

这时你要怀疑：
- 线程池配置是否不合理
- 任务是否被锁或 IO 阻塞
- 系统是不是把时间花在切换上，而不是执行任务

## 观察指标可以看什么
- 活跃线程数
- 队列积压长度
- 平均任务耗时和长尾耗时
- CPU 使用率和上下文切换次数

## 一个简单的调优思路
如果你发现：
- 线程很多
- 切换很多
- 吞吐没起来

可以优先怀疑：
1. 线程池开太大
2. 任务粒度太碎
3. 任务里有锁竞争
4. IO 阻塞把线程占住了

这时不是继续加线程，而是先判断瓶颈类型。

## 一个测试开发场景
自动化执行框架里，100 个任务同时下发：
- 线程池开太小，队列会积压
- 线程池开太大，切换和锁竞争又会升高

所以合理配置线程池，本质上是在吞吐、响应时间和系统开销之间找平衡。

## 为什么测试开发要懂
- 压测时线程池配置是常见瓶颈点
- 日志采集、任务分发、自动化执行器经常会用线程池
- 排查性能问题时，要区分是“任务处理慢”还是“线程调度成本高”

## 常见误区
- 觉得线程越多并发越高
- 线程池只配最大线程数，不看队列和拒绝策略
- 看到 CPU 高就直接认为业务计算重
- 不区分“线程多导致慢”和“任务本身慢”
- 看到队列堆积就只加线程，不先查任务为什么慢
