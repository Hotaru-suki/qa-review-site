# 进程间通信该怎么选

## 这个知识点在解决什么问题
进程之间地址空间隔离，不能像线程那样直接共享普通变量。所以当两个进程要交换数据、传递事件、同步状态时，就必须选合适的进程间通信方式。

## 常见 IPC 方式
- 管道 / 命名管道
- 消息队列
- 共享内存
- 信号
- Socket

## 先按几个维度判断
- 通信双方是在同一台机器，还是跨机器
- 传的是事件通知，还是大量数据
- 是单向流，还是双向请求响应
- 更关心实时性，还是更关心实现复杂度
- 是否需要持久化和排队能力

## 各自解决什么问题
### 管道
适合父子进程或串行数据流传递，模型简单，但不适合复杂双向通信。

### 消息队列
适合按消息边界传递结构化数据，发送方和接收方耦合较低。

### 共享内存
速度快，因为多进程共享同一段物理内存；但同步控制复杂，通常要配合锁或信号量。

### 信号
适合通知事件，不适合承载大量业务数据。

### Socket
最通用，既能本机进程通信，也能跨机器；代价是协议和序列化开销更高。

## 用字符图看差异
### 管道

```text
Process A  --->  kernel pipe buffer  --->  Process B
```

更像一条流，适合顺序写入、顺序读取。

### 消息队列

```text
Producer A ---\
Producer B ----> [ message queue ] ---> Consumer
Producer C ---/
```

天然带消息边界，更适合任务派发和事件通知。

### 共享内存

```text
Process A ----\
               > [ shared memory ] <---- Process B
Process C ----/
```

数据复制少，但“谁先写、谁后读、什么时候可见”都要自己处理。

## 怎么选
```text
小通知事件 -> 信号
简单流式传输 -> 管道
结构化消息 -> 消息队列
高性能大数据共享 -> 共享内存
需要跨机或统一通信模型 -> socket
```

## 典型场景
### 本机采集进程和分析进程交换大量图像帧
- 更适合共享内存
- 因为拷贝成本很高

### 主进程向工作进程派发任务
- 更适合消息队列或 socket
- 因为任务天然按消息组织

### 服务进程收到退出通知
- 可以用信号

### 本机测试代理和被测进程交换控制命令
- 如果命令量不大、需要清晰边界，消息队列或本地 socket 都可以
- 如果同时还要返回执行结果，本地 socket 往往更统一

## 选择时的代价也要看
- 管道简单，但不适合复杂拓扑
- 消息队列解耦好，但可能引入排队和顺序问题
- 共享内存快，但同步、清理、异常恢复最麻烦
- socket 通用，但要自己定义协议和处理连接状态

## 一句话记忆
- 要简单流式传输，先想管道
- 要结构化任务派发，先想消息队列
- 要同机高吞吐共享大块数据，先想共享内存
- 要跨机或希望模型统一，先想 socket

## 和测试/排障有什么关系
做系统测试时，IPC 问题常表现成：
- 子进程收不到命令
- 进程退出后资源没清理
- 队列堆积导致消息延迟
- 共享内存里数据被覆盖或没同步

所以排障时不能只看“进程活着没有”，还要看：
- 通信对象有没有建起来
- 消息有没有发送成功
- 接收端有没有消费
- 异常退出后残留资源有没有影响下一次运行

## 常见误区
- 为了“快”一律上共享内存，却忽略同步和一致性复杂度
- 只谈 IPC 名字，不谈具体数据量、实时性和部署范围
- 忽略异常退出后的清理和资源泄漏
- 只看吞吐，不看调试和维护成本
