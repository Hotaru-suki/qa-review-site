# 线程安全与同步原语怎么取舍

## 这个知识点在解决什么问题
一旦多个线程同时访问共享数据，就会出现覆盖写、脏读、顺序错乱等问题。线程安全不是一句“加锁就行”，而是要理解共享状态在哪里、竞争发生在哪一步、应该用哪种同步原语控制。

## 什么叫线程安全
如果多个线程同时执行同一段逻辑，最终结果仍然正确、状态仍然一致，这段逻辑才算线程安全。

典型不安全例子：

```text
count = count + 1
```

这看起来是一行，但底层通常包含：
- 读取 count
- 加 1
- 写回 count

两个线程交错执行，就可能少加。

可以把它想成：

```text
T1: read count = 0
T2: read count = 0
T1: write count = 1
T2: write count = 1
```

最后结果应该是 2，却变成了 1，这就是竞态条件。

## 常见同步原语在解决什么
### 互斥锁
同一时刻只允许一个线程进入临界区。适合保护共享变量和复杂状态更新。

### 读写锁
读多写少时，允许多个读线程并发，但写线程独占。

### 条件变量
让线程在某个条件满足前等待，适合生产者消费者模型。

### 信号量
控制可同时访问某类资源的线程数量。

### 原子变量
适合简单计数、状态位更新，避免重锁开销。

## 怎么选
```text
共享状态复杂 -> 互斥锁
读多写少 -> 读写锁
需要等待条件 -> 条件变量
控制并发配额 -> 信号量
单变量轻量更新 -> 原子操作
```

## 典型场景
### 日志缓冲区
多个线程写同一个缓冲区，需要互斥保护或无锁队列。

### 任务队列
生产者投递任务，消费者等待任务到来，通常需要“锁 + 条件变量”。

### 请求计数器
简单累计访问次数，适合原子计数。

## 代码上怎么理解差异
### 互斥锁保护复杂状态

```cpp
std::mutex mtx;
std::vector<int> tasks;

void push_task(int id) {
    std::lock_guard<std::mutex> lock(mtx);
    tasks.push_back(id);
}
```

这里用锁，是因为 `vector` 的修改涉及内部状态，不适合只靠原子变量。

### 原子变量保护简单计数

```cpp
std::atomic<int> request_count{0};

void on_request() {
    request_count.fetch_add(1, std::memory_order_relaxed);
}
```

这里如果只是计数，原子变量通常比大锁更轻量。

### 条件变量做生产者消费者

```cpp
std::mutex mtx;
std::condition_variable cv;
std::queue<int> q;

void consumer() {
    std::unique_lock<std::mutex> lock(mtx);
    cv.wait(lock, [] { return !q.empty(); });
    int value = q.front();
    q.pop();
}
```

这类场景只靠互斥锁还不够，因为消费者需要在“没有任务时等待”。

## 死锁通常怎么来的
- 线程 A 先拿锁 1 再拿锁 2
- 线程 B 先拿锁 2 再拿锁 1

这时两个线程都可能永远等下去。

## 怎么避免
- 统一加锁顺序
- 缩小持锁范围
- 不在持锁期间做阻塞 IO
- 把能拆开的共享状态拆开，减少大锁

## 常见误区
- 所有地方都用大锁，结果并发性能很差
- 只给写加锁，不保护读
- 持锁范围过大，把 IO 和日志也包进去
- 忘了死锁风险，例如多把锁顺序不一致
- 以为“用了原子变量”就天然解决了所有复合状态一致性问题
