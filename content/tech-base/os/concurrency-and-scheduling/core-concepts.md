# 并发、并行、同步、异步、阻塞、非阻塞

## 摘要

这组概念之所以难，是因为很多人把“多个任务怎么推进”“结果什么时候返回”“线程会不会被挂住”这三件事混成了一件事。真正学明白的关键，不是背定义，而是先搞清每个概念到底在回答什么问题，再把它们放回同一个场景里看。只要维度没分清，后面越学越乱；维度一旦分清，这些词其实是在描述系统运行的不同侧面。

---

## 核心问题

- 并发、并行到底差在哪
- 同步 / 异步 和 阻塞 / 非阻塞 为什么不是一回事
- 这些概念放在同一个例子里时应该怎么区分

---

## 1. 先把问题维度拆开

这组概念其实分别在回答不同问题：

- 并发 / 并行：多个任务怎么推进
- 同步 / 异步：结果什么时候交回给调用方
- 阻塞 / 非阻塞：等待过程中线程会不会被挂住

一旦把这三条线混在一起，后面所有定义都会越看越乱。

所以最稳的第一步不是背名词，而是先问：

- 我现在讨论的是任务推进方式？
- 还是结果返回方式？
- 还是线程等待方式？

---

## 2. 并发和并行到底差在哪

### 并发

并发强调的是：

> 在同一段时间内，多个任务都在往前推进。

它不要求同一时刻真的同时执行。

例如单核 CPU 上：

```text
时刻1: 任务A跑一点
时刻2: 切到任务B跑一点
时刻3: 再切回任务A
```

虽然同一时刻只有一个任务在执行，但从一段时间看，A 和 B 都在推进，这就是并发。

### 并行

并行强调的是：

> 在同一时刻，多个任务真的同时执行。

这通常依赖多核 CPU、多处理单元或分布式资源。

例如：

```text
CPU核1 跑任务A
CPU核2 跑任务B
```

这时 A 和 B 在同一时刻都在运行，这才叫并行。

### 最稳的区分方式

- 并发看“时间段内是否一起推进”
- 并行看“同一时刻是否一起执行”

所以：

- 并行一定属于并发
- 并发不一定是并行

---

## 3. 同步和异步到底在说什么

### 同步

同步强调的是：

> 调用方必须等待当前操作的结果，流程上才能继续往下衔接。

重点不在于线程是不是睡住了，而在于逻辑上你是不是必须先拿到这个结果。

例如：

```python
result = query_db()
handle(result)
```

这里后续逻辑依赖 `query_db()` 的结果，属于同步关系。

### 异步

异步强调的是：

> 任务发出去之后，调用方不必原地等待，可以先做别的，结果完成后再通知回来或稍后处理。

例如：

```text
提交任务
  ↓
当前线程先继续干别的
  ↓
任务完成后回调 / 通知 / future 完成
```

所以异步说的是：结果返回机制改变了。

---

## 4. 阻塞和非阻塞到底在说什么

### 阻塞

阻塞强调的是：

> 当前线程在等待条件满足之前，不能继续执行。

常见场景：

- 等 IO
- 等锁
- 等线程结束

### 非阻塞

非阻塞强调的是：

> 即使条件不满足，调用也立刻返回，不把当前线程挂住。

例如非阻塞读取 socket，没有数据时会立刻告诉你“现在还读不到”，而不是让线程睡在那里。

所以阻塞/非阻塞描述的是线程等待行为，不是业务逻辑依赖关系。

---

## 5. 为什么同步/异步和阻塞/非阻塞不是一回事

因为它们描述的是不同维度，所以可以交叉组合。

### 同步阻塞

最常见。

```text
发起调用 -> 线程原地等待 -> 拿到结果 -> 继续
```

### 同步非阻塞

例如你不断轮询一个状态：

```text
发起调用 -> 立刻返回“还没好”
发起调用 -> 立刻返回“还没好”
发起调用 -> 终于返回结果
```

逻辑上你还是得等结果才能继续，所以是同步；但线程每次没被挂住，所以是非阻塞。

### 异步阻塞

例如你先异步提交任务，但后面又马上调用 `future.get()` 去等结果，那等待这一步仍然可能阻塞。

### 异步非阻塞

提交任务后当前线程继续做别的，结果完成后再通知回来，这是最典型的异步非阻塞。

所以不能简单地说：

- 同步 = 阻塞
- 异步 = 非阻塞

这是错误的。

---

## 6. 放到同一个例子里看

以读 socket 数据为例：

### 阻塞同步

```text
read() -> 没数据 -> 卡住等 -> 有数据 -> 返回
```

### 非阻塞同步

```text
read() -> 没数据 -> 立刻返回
稍后再 read() -> 还没数据 -> 立刻返回
直到某次 read() -> 有数据 -> 返回
```

### 异步

```text
提交读取请求
当前线程继续做别的
数据准备好后，系统通知你来处理
```

这个例子最能说明：同步/异步和阻塞/非阻塞是在不同坐标轴上。

---

## 7. 这些概念为什么在工程里重要

因为它们直接决定：

- 线程是否会被长时间占住
- 系统能否同时推进很多任务
- 高并发下资源开销是否可控
- 程序结构是回调驱动还是顺序等待

例如服务端一连接一线程的阻塞模型，在连接数少时很直观；但连接数很多时，线程和切换成本会迅速上升。这时就会自然过渡到非阻塞、多路复用、异步回调等模型讨论。

---

## 8. 常见误区

- 把并发和并行当成同义词
- 把同步和阻塞直接画等号
- 把异步和高性能直接画等号
- 觉得非阻塞一定更好，忽略忙等会浪费 CPU

这些误区的根源，都是没先把维度拆开。

---

## 9. 一句话记忆法

- 并发 / 并行：多个任务怎么推进
- 同步 / 异步：结果什么时候交回来
- 阻塞 / 非阻塞：等待时线程会不会被挂住

只要始终守住这三条线，很多混淆会自然消失。

---

## 小结

这组概念最重要的不是记单独定义，而是始终知道它们分别在回答哪一类问题。任务推进、结果返回、线程等待，这三条线一旦分清，并发相关知识就不再是抽象名词堆，而会变成很清楚的系统行为描述。
