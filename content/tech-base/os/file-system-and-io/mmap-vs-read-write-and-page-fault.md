# mmap、普通读写与缺页异常

## 摘要

很多人知道 `mmap` 可以把文件映射到内存，但不知道它到底快在哪里、慢在哪里，也不知道它和 `read/write` 的核心差别是什么。这个知识点真正要理解的是：数据访问路径、页缓存、缺页异常和拷贝成本。

---

## 核心问题

- `mmap` 和 `read/write` 的访问路径有什么差别
- 为什么 `mmap` 不等于一定更快
- 缺页异常在文件映射里扮演什么角色

---

## 1. 普通读写在做什么

使用 `read` 时，可以粗略理解为：

1. 内核从页缓存或磁盘拿到数据
2. 再把数据拷贝到用户态缓冲区

它的好处是：

- 模型直接
- 生命周期清楚

代价是：

- 用户态和内核态之间有明确的数据拷贝

---

## 2. `mmap` 在做什么

`mmap` 更像是：

- 把文件的一段内容映射进进程虚拟地址空间

之后程序访问这块内存时，内核再按需把对应页准备好。

它最大的特点不是“完全没有内核参与”，而是：

- 访问方式变成了内存式访问
- 不必每次显式 `read`

### 一个更直观的区别

可以粗略把它们理解成：

```text
read/write: 你主动要一段数据，内核拷给你
mmap:      你先建立映射，真正访问时再按页触发准备
```

这个差别决定了：

- `read/write` 更像显式拉数据
- `mmap` 更像把访问时机推迟到内存访问阶段

---

## 3. 缺页异常为什么关键

映射建立后，数据不一定立刻都在物理内存里。

当程序第一次访问某个尚未准备好的映射页时，就会触发：

- 缺页异常

内核会在这时：

- 建立页映射
- 必要时把数据从页缓存或磁盘加载进来

所以 `mmap` 的成本并没有消失，而是被部分延后到了“访问那一刻”。

### 为什么这会带来抖动

如果程序第一次碰到某一页时才触发缺页，那这次访问就可能比后续访问明显慢。

于是你会看到：

- 平均吞吐不错
- 但首次访问或冷页访问 RT 抖一下

这类现象在排障时很容易被误判成“偶发卡顿”。

---

## 4. 为什么 `mmap` 不一定更快

因为性能不仅取决于 API 名字，还取决于：

- 访问模式是否顺序
- 数据是否局部
- 页缺失是否频繁
- 是否会产生大量随机访问

如果访问模式很差，频繁触发缺页和随机页加载，`mmap` 未必比 `read` 更占优。

### 什么时候它更容易有优势

- 访问的是结构化文件
- 需要随机定位某些片段
- 希望把数据当内存结构处理

### 什么时候它未必合适

- 顺序大块扫描
- 页面很多但实际只读一次
- 对访问延迟抖动很敏感

---

## 5. 一个测试开发视角

如果你在看大文件解析、日志扫描、索引文件访问等场景：

- 顺序大块读取时，普通读写通常更直观
- 随机访问固定结构数据时，映射方式可能更方便

测试或排障时真正该关注的是：

- 访问模式
- 页缓存命中情况
- 缺页和 IO 抖动

而不是简单问“哪个 API 更快”。

### 一个测试开发里的例子

如果你在看日志分析工具、倒排索引文件、二进制资源文件解析：

- 需要随机跳转时，映射方式可能让代码更自然
- 但如果测试观察到冷启动阶段卡顿，就要怀疑是不是大量首访页缺失叠加在一起

这时问题已经不是“API 用错没”，而是访问模式和页行为是否匹配。

---

## 6. 常见误区

- 把 `mmap` 理解成完全不用 IO
- 认为 `mmap` 一定零成本、一定更快
- 不理解页缺失带来的延迟抖动
- 只看接口形式，不看访问模式

---

## 小结

`read/write` 和 `mmap` 的差别，不只是调用方式不同，而是数据进入进程的路径和时机不同。理解页缓存、访问模式和缺页异常后，才能真正判断一个场景更适合哪种方式。
