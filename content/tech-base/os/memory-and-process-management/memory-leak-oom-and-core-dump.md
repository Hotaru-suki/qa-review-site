# 内存泄漏、OOM 与 core dump

## 这个知识点在解决什么问题
程序内存问题常被混成一件事，但其实至少有三类不同现象：
- 内存泄漏：内存持续增长却没释放
- OOM：系统或进程已经拿不到足够内存
- core dump：程序崩溃时留下的现场快照

把这三者分清，排查路径才不会乱。

## 内存泄漏是什么
内存泄漏的重点不是“程序申请了内存”，而是：
- 申请之后失去了引用
- 或者长期不再需要，却一直不释放

结果是进程长期运行时内存越来越高。

## OOM 是什么
OOM 表示内存资源不够了。常见表现：
- 申请内存失败
- 进程被系统 OOM killer 杀掉
- 服务开始频繁抖动或重启

OOM 不一定都是泄漏造成，也可能是：
- 瞬时内存峰值过高
- 缓存配置过大
- 并发请求同时占用大量内存

## core dump 是什么
core dump 是程序异常崩溃时保存下来的内存和寄存器现场。它的价值是：
- 让你回到崩溃时刻看栈帧
- 找到崩溃位置和上下文

## 典型排查思路
### 内存持续上涨
- 看是否存在长期未释放对象
- 看是否有缓存无限增长
- 看是否有连接、句柄、容器长期累积

### 进程突然被杀
- 看系统日志是否有 OOM 记录
- 看同时间段内存、swap、容器限制

### 程序崩溃
- 看是否生成 core dump
- 结合日志、堆栈、最近变更定位问题

## 测试开发为什么要懂这些
长时间稳定性测试、压测、弱网恢复测试时，经常会遇到：
- 内存只涨不回
- 压力一大进程被杀
- 客户端或服务端偶发崩溃

这些都不是简单的“功能 bug”，而是资源和系统行为问题。

## 常见误区
- 进程内存高就等于泄漏
- OOM 就等于代码一定没 `free`
- 有 core dump 却不去分析，只看表面错误日志
