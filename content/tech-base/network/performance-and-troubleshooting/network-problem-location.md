# 网络问题如何按层定位

## 摘要

网络故障最容易陷入“看起来什么都像网络问题”的混乱。真正高效的定位方式，不是凭经验先猜，而是按链路分层判断问题停在哪一段。对客户端、测试和服务端排障来说，最重要的是知道：一个请求从域名解析到建立连接、完成 TLS、发出 HTTP、拿到业务响应，中间每一层失败时分别会长什么样。

---

## 核心问题

- 为什么网络定位必须分层，而不能上来就猜“后端挂了”
- 从 DNS、连接、TLS、HTTP 到应用逻辑，怎样逐步收窄范围
- 哪些现象更像链路问题，哪些更像服务逻辑问题

---

## 1. 为什么网络定位一定要分层

“请求失败”只是最终表象，它背后可能停在完全不同的位置：

- 域名没解析对
- 端口不通
- TLS 握手失败
- HTTP 返回错误
- 业务接口本身报错

如果不分层，最常见的结果就是：

- 前端说后端挂了
- 后端说网络有问题
- 测试说接口超时但说不清超时发生在哪一步

分层定位的价值，就是把模糊的“请求失败”拆成一个可验证链路。

---

## 2. 一个请求链路可以怎么拆

可以先把最常见的访问路径想成：

```text
输入域名
  ↓
DNS 解析
  ↓
建立 TCP 连接
  ↓
如果是 HTTPS，再做 TLS 握手
  ↓
发送 HTTP 请求
  ↓
服务端处理并返回响应
  ↓
客户端解析结果
```

只要知道这条链，你就知道任何“访问失败”都应该先问：它停在了哪一层。

---

## 3. 每一层失败时通常会有什么现象

### DNS 层

常见表现：

- 域名无法解析
- 解析到错误地址
- 某地区或某网络下才能复现

常见判断点：

- `nslookup` / `dig` 结果
- hosts 是否被改写
- DNS 解析结果是否和预期一致

### TCP 连接层

常见表现：

- 超时
- 连接被拒绝
- 端口不通

常见判断点：

- 目标 IP 是否可达
- 目标端口是否监听
- 三次握手是否完成

### TLS 层

常见表现：

- 证书错误
- 协议版本不匹配
- 握手失败

常见判断点：

- 证书是否过期
- 域名是否匹配
- TLS 协议和加密套件是否兼容

### HTTP 层

常见表现：

- 4xx
- 5xx
- 请求被重定向
- 响应格式异常

这里要注意，HTTP 层有响应，不代表链路没问题，只代表前面几层大概率已经通了。

### 应用逻辑层

常见表现：

- HTTP 200，但业务码失败
- 接口返回成功，但数据不对
- 页面看起来像网络问题，实际是业务状态异常

这时就已经不该再泛泛说“网络问题”了，而要回到应用日志和业务链路。

---

## 4. 一个具体定位场景

假设用户反馈：“接口一直超时”。

更稳的排查顺序通常是：

1. 先确认域名是不是解析到了正确地址
2. 再确认目标 IP 和端口能不能连上
3. 如果是 HTTPS，再确认 TLS 是否正常
4. 再看 HTTP 请求有没有真正发出去
5. 最后才继续看服务端处理和业务日志

这里每一步都对应不同结论：

- DNS 错了，问题在解析层
- 连不上端口，问题在连接层或服务监听
- TLS 挂了，问题在证书或协议协商
- 有请求无响应，要继续看服务端处理
- HTTP 已返回但业务失败，就不是网络链路问题了

---

## 5. 一个测试视角下怎么收证据

测试排查网络问题时，不一定能直接看所有服务端状态，但至少可以建立一条更完整的证据链：

- 复现时间和环境
- 域名和目标地址
- 抓包结果
- 请求发出时间、响应返回时间
- 状态码、错误码、超时位置
- 是否只在某网络、某地区、某运营商出现

这样提交给开发或运维时，问题描述会从“接口不稳定”变成“DNS 正常、TCP 建连正常、TLS 正常，但 HTTP 响应超过客户端超时阈值”，价值完全不同。

---

## 6. 常见误区

- 一看到超时就直接归因给后端
- 一看到 5xx 就认定一定是服务逻辑问题
- 连 DNS、连接、TLS 都没排，就直接怀疑业务代码
- 客户端报“网络错误”，就以为一定是用户网络差

网络定位最怕跳层。前面的层不确认，后面很容易一直在错方向上查。

---

## 小结

网络定位的核心不是经验猜测，而是分层排除。只要把请求链路拆清楚，并知道每一层失败分别会产生什么现象，网络问题就能从模糊的“请求失败”逐步收敛成具体、可验证、可交接的技术问题。
