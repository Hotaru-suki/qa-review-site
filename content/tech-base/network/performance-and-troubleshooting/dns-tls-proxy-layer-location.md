# DNS、TLS 与代理层怎么分层定位

## 这个知识点在解决什么问题
很多“网络问题”其实不是单纯的 TCP 连不通，而是卡在：
- DNS 解析
- TLS 握手
- 代理/网关转发

这个知识点解决的是：当请求“发不出去”或“发得很慢”时，怎样把问题从应用层继续拆到 DNS、TLS 和代理层。

## DNS 层在看什么
DNS 负责把域名解析成目标地址。  
如果这里慢或错，后面的连接根本建不起来。

典型现象：
- 域名偶发解析失败
- 首次请求明显慢
- 不同网络环境解析结果不一致

## TLS 层在看什么
TLS 负责安全握手和证书校验。  
这里的问题常表现为：
- 握手超时
- 证书错误
- 协议/套件不兼容

## 代理层在看什么
代理、网关、负载均衡层会影响：
- 路由转发
- 请求头处理
- 超时策略
- 连接复用

所以“服务端没收到请求”未必是客户端问题，也可能是中间代理拦住了。

## 一个排查顺序
1. 先确认是不是 DNS 解析异常
2. 再确认 TCP/TLS 是否建立成功
3. 最后确认代理层是否转发到了目标服务

## 测试开发为什么要懂
接口联调时，经常会遇到：
- Postman 偶发超时
- 浏览器访问报证书问题
- 服务端日志里压根没有请求

这时如果不把 DNS、TLS、代理层拆开，就很难快速定位。

## 常见误区
- 一看到超时就只怪服务端
- 不区分“连不上”“握手失败”“转发失败”
- 代理层存在时仍然把问题只看成点对点连接
