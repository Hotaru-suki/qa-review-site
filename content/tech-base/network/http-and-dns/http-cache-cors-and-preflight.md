# 缓存、CORS 与预检请求

## 这个知识点在解决什么问题
前后端联调和接口测试里，常见问题不只是“接口有没有返回数据”，还包括：
- 为什么我改了接口却像没生效
- 为什么浏览器里报跨域
- 为什么明明只有一个请求，却看到了多一次 OPTIONS

这背后通常是缓存、CORS 和预检请求。

## 缓存在解决什么
缓存的目标是减少重复请求和传输成本，但副作用是：
- 你以为接口或资源没变，其实浏览器在复用旧结果
- 联调时容易误判“服务端没更新”

常见缓存相关头包括：
- `Cache-Control`
- `Expires`
- `ETag`
- `Last-Modified`

## CORS 在解决什么
CORS 不是服务端“允许跨域访问”这么简单，它本质上是浏览器的同源安全限制下，允许特定跨源请求被放行的机制。

常见头包括：
- `Access-Control-Allow-Origin`
- `Access-Control-Allow-Methods`
- `Access-Control-Allow-Headers`

## 为什么会有预检请求
当浏览器认为一次跨域请求风险较高时，会先发一个 `OPTIONS` 预检，请服务端确认：
- 这个源能不能访问
- 允许哪些方法
- 允许哪些自定义头

只有预检通过，真正请求才会发出。

## 典型场景
### 浏览器里调接口报跨域
要判断是：
- 服务端没返回正确 CORS 头
- 预检请求被拦截
- 带了自定义头触发预检，但服务端没放行

### 前端说接口改了但页面没变化
要判断是：
- 资源缓存
- CDN 缓存
- 浏览器本地缓存
- 还是根本没发新请求

## 测试时要注意什么
- Postman 能调通，不代表浏览器能调通
- 跨域问题要在浏览器环境下验证
- 看到两个请求时，要先确认第一个是不是 `OPTIONS`

## 常见误区
- 把缓存问题误判成接口问题
- 把 Postman 调通误判成浏览器也会调通
- 不知道 `OPTIONS` 是预检，请求链路看起来像“多发了一次”
