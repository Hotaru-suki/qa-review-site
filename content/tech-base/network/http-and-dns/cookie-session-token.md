# Cookie、Session 与 Token 怎么区分

## 摘要

Cookie、Session 和 Token 经常一起出现，但它们并不处在同一层级。Cookie 更像浏览器侧的存储与携带机制，Session 更像服务端保存会话状态的方式，Token 更像客户端持有、服务端校验的身份凭证。真正要分清的，不是背三者定义，而是看清客户端存什么、服务端记什么，以及认证状态怎样在多个请求之间延续。

---

## 核心问题

- Cookie、Session 和 Token 各自承担什么角色
- 为什么它们常一起出现，却不能互相替代
- 登录状态到底存在哪一边，怎样在请求间延续
- 测试身份链路时，三者分别该看什么

---

## 1. 为什么三者总被混在一起

因为它们都和“登录后如何保持身份”有关，但关注点不同：

- Cookie：客户端怎么保存并自动带回一段数据
- Session：服务端怎么记住这次会话是谁
- Token：客户端怎么持有一个可校验的身份凭证

也就是说，它们是在同一条身份链路上的不同位置，不是三个平级替代品。

---

## 2. Cookie 到底在做什么

Cookie 本质上是浏览器帮助保存并在后续请求中自动携带的一段小数据。

典型流程：

```text
服务端响应 Set-Cookie
   ↓
浏览器保存 Cookie
   ↓
后续访问同域名时自动带上 Cookie
```

所以 Cookie 更像“载体”或“传递工具”，它并不天然等于登录方案。

它可以存：

- 会话 ID
- 偏好设置
- 某些轻量状态

但 Cookie 自己不负责说明“这个用户到底是谁、服务器是否信任他”，这些要看它承载的内容是什么。

---

## 3. Session 到底是什么

Session 更强调：服务端保存一段和当前登录会话相关的状态。

典型模式是：

1. 用户登录成功
2. 服务端创建一份 session 数据
3. 生成一个 session id
4. 把 session id 通常通过 Cookie 发给浏览器
5. 浏览器后续请求自动带回这个 session id
6. 服务端根据 session id 找到对应会话状态

字符图可以写成：

```text
浏览器: Cookie(session_id=abc123)
                 ↓
服务端: session_store["abc123"] = {user_id: 1001, role: "admin"}
```

所以 Session 的关键不是浏览器带了什么，而是服务端有没有保存这份会话上下文。

---

## 4. Token 又是什么

Token 更像一段由客户端持有、服务端可校验的身份凭证。

常见流程：

1. 用户登录成功
2. 服务端签发 token
3. 客户端保存 token
4. 后续请求通过 Header 或其他方式携带 token
5. 服务端验证 token 是否合法、是否过期、是否对应某用户

例如：

```text
Authorization: Bearer <token>
```

和 Session 不同的是，Token 模式下，服务端可以不必像传统 Session 那样强依赖集中保存完整会话状态，而是把一部分身份信息或校验信息封进 token 本身，或通过 token 去查关联信息。

---

## 5. 三者怎么放到一张图里理解

可以这样记：

- Cookie：怎么带
- Session：状态存在哪
- Token：凭什么证明你是谁

所以：

- Cookie 是传输载体
- Session 是服务端会话状态
- Token 是认证凭证

它们可能组合使用，也可能分开使用。

---

## 6. 一个登录场景分别长什么样

### Session 模式

```text
用户登录
  ↓
服务端验证账号密码
  ↓
服务端创建 session
  ↓
Set-Cookie: session_id=abc123
  ↓
浏览器后续自动带 Cookie
  ↓
服务端查 session_id -> 找到用户状态
```

### Token 模式

```text
用户登录
  ↓
服务端验证账号密码
  ↓
签发 token
  ↓
客户端保存 token
  ↓
后续请求手动或程序带 Authorization 头
  ↓
服务端校验 token
```

这两个模式的关键区别是：

- Session 更依赖服务端持久会话状态
- Token 更强调客户端持有凭证、服务端校验凭证

---

## 7. 测试时三者分别该关注什么

### 测 Cookie

重点看：

- 是否正确设置
- 域、路径、过期时间是否正确
- 是否按预期自动携带
- 安全属性如 `HttpOnly`、`Secure`、`SameSite` 是否合适

### 测 Session

重点看：

- 登录后是否真正建立会话
- 退出登录后是否失效
- 超时是否失效
- 多端登录、并发登录是否符合设计
- 会话清理是否及时

### 测 Token

重点看：

- 是否签发成功
- 是否过期
- 伪造、篡改后是否被拒绝
- 刷新机制是否正确
- 权限变化后旧 token 如何处理

这样就能看出，三者的测试关注点根本不是同一件事。

---

## 8. 一个常见误区：Cookie 不等于 Session

很多人会说“我们用 Cookie 登录”，这其实容易说混。

更准确地说，通常是：

- 用 Cookie 携带 session id
- 服务端根据 session id 找 session

所以真正的登录状态管理方案是 Session，Cookie 只是它常见的承载方式。

同理，也有人说“Token 更安全”，这也过于绝对。安全性最终取决于：

- 存储位置是否安全
- 是否防泄漏、防篡改
- 是否处理过期和撤销
- 整体链路设计是否合理

---

## 9. 一个测试排障视角

如果用户反馈“明明登录了，但接口提示未登录”，排查时就要分层看：

- Cookie 有没有带上
- Session 是否还存在
- Token 是否过期
- 服务端校验逻辑是否一致

如果不把 Cookie、Session、Token 分层理解，就很容易把所有问题都归成“登录态丢了”，却说不清到底是哪里丢的。

---

## 小结

Cookie、Session 和 Token 最稳的理解方式，是把它们看成身份状态管理链路上的不同位置，而不是三种并列技术名词。Cookie 解决“怎么带”，Session 解决“服务端怎么记”，Token 解决“客户端拿什么证明自己”。把这条链理顺，登录和认证相关问题就会清楚很多。
