# TCP 与 UDP 的核心区别

## 摘要

TCP 和 UDP 的差别不只是“一个可靠、一个不可靠”。它们本质上是在做两种不同的传输取舍：TCP 通过连接状态、确认应答、重传和流控换取更稳定的结果完整性；UDP 则尽量减少协议负担，把是否补偿丢包、是否恢复顺序交给应用层处理，以换取更轻的开销和更低的实时延迟。

---

## 核心问题

- TCP 和 UDP 分别想优先保证什么
- 为什么 TCP 要维护连接状态，而 UDP 通常不维护
- 顺序、重传和可靠性到底是怎么换来更高开销的
- 什么样的业务更适合 TCP，什么样的业务更适合 UDP

---

## 1. 先不要把它理解成“高级和低级”

很多人第一次学这两个协议时，容易形成一个错误印象：

- TCP 功能多，所以更高级
- UDP 功能少，所以只是“简化版”

这种理解是错的。

更准确的说法是：

- TCP 是“结果完整性优先”的传输协议
- UDP 是“协议负担更轻、实时性更优先”的传输协议

它们不是高低关系，而是目标不同。

---

## 2. TCP 和 UDP 头部视角下的直觉区别

### TCP 更像“带状态的传输通道”

TCP 在通信前要建立连接，通信中维护序号、确认号、窗口、连接状态，通信结束后还要有关闭流程。

这意味着 TCP 不只是“把包发出去”，而是在维护一段持续的连接关系。

### UDP 更像“尽快把一个个数据报发出去”

UDP 不强调建立连接，也不默认承担重传、顺序恢复、流量控制这些责任。发送方发出一个数据报，协议层尽力发，但不保证对端一定收到，也不保证一定按顺序交付给应用层。

所以 UDP 的协议本体更轻，但上层如果需要可靠效果，就要自己补机制。

---

## 3. TCP 到底多做了哪些事

TCP 的“可靠”不是一句口号，而是下面这些机制共同完成的：

- 连接建立与状态维护
- 序号
- 确认应答
- 超时重传
- 流量控制
- 拥塞控制
- 顺序重组

换句话说，TCP 为了保证“字节流最终按顺序、尽量完整地送达”，在协议层承担了大量额外工作。

而 UDP 基本不帮你做这些。

---

## 4. 为什么 TCP 要先建立连接

TCP 建立连接，不是为了“走流程”，而是为了让双方先同步一段通信关系。

建立完连接之后，双方才有一致上下文去做这些事：

- 确认彼此都能收发
- 确认初始序号
- 确认当前连接状态
- 后续根据状态持续确认、重传、控流

这让 TCP 能更稳地处理长连接、顺序、重发和关闭。

UDP 通常不需要这些前置协商，所以可以更快开始发送，但代价是协议层也不会给你这些保障。

---

## 5. 一个直观字符图

### TCP 的思路

```text
应用数据
   ↓
TCP: 建连接 -> 编序号 -> 发包 -> 等确认 -> 丢了就重传 -> 按序交付
   ↓
IP
```

### UDP 的思路

```text
应用数据
   ↓
UDP: 封一个数据报 -> 发出去
   ↓
IP
```

从这个角度看，TCP 是“协议层多做事”，UDP 是“协议层少做事，把决定权交给应用层”。

---

## 6. 顺序和重传为什么会增加成本

如果一个协议要保证顺序和可靠性，就至少要解决：

- 这一段数据是第几个
- 对方到底有没有收到
- 如果没收到，什么时候重发
- 如果后面的包先到了，要不要先缓存等待

这些能力都需要额外状态和等待。

因此 TCP 的代价通常表现为：

- 更高协议开销
- 更复杂状态维护
- 丢包后等待恢复，时延可能抖动
- 连接管理成本更高

UDP 因为默认不做这些，所以更轻，但如果应用真的需要类似保证，又得自己补回来。

---

## 7. 业务到底怎么选

### 更适合 TCP 的场景

这类业务更看重结果完整性和顺序一致性：

- 网页访问
- 文件下载
- 订单提交
- 支付请求
- 配置同步

如果丢一段、乱一次序、少一部分内容就会出大问题，那 TCP 更自然。

### 更适合 UDP 的场景

这类业务更在意时延和实时反馈，对少量丢失容忍度更高：

- 实时语音
- 实时音视频中的一部分流
- 在线对战状态同步
- 局域网广播或发现协议

例如语音通话中，丢一两帧通常比整体延迟 2 秒更可接受。

---

## 8. 一个测试视角下的例子

假设你在测在线游戏角色同步：

- 如果选 TCP，丢包后会重传，最终数据更完整
- 但网络一抖，后续状态可能因为等待重传而产生明显延迟

如果选 UDP：

- 个别状态包可能丢失
- 但后续状态能更快继续到达
- 应用层通常会用“新状态覆盖旧状态”或插值方式做补偿

这就是为什么很多实时系统宁可接受局部丢失，也不愿意接受长期等待。

---

## 9. 复杂度不是重点，代价模型才是重点

TCP 和 UDP 不太适合用算法题那种时间复杂度去比较，更重要的是理解它们的代价模型：

- TCP：协议层做得多，状态更重，时延波动更大，但一致性更强
- UDP：协议层做得少，时延更轻，但很多可靠性工作要上移到业务层

所以面试或工作里更好的回答，不是“TCP 慢，UDP 快”，而是：

- TCP 为可靠性和顺序做了更多工作
- UDP 把这些责任更多留给应用层

---

## 10. 常见误区

- 把 TCP 理解成“绝对不会丢”
- 把 UDP 理解成“完全不可靠所以没法用于重要业务”
- 只背“TCP 可靠、UDP 不可靠”，却说不出为什么
- 忽略应用层其实也可以在 UDP 之上补可靠性

真正重要的不是标签，而是理解它们各自把责任放在了哪一层。

---

## 小结

TCP 和 UDP 的本质区别，是两种不同的传输取舍。TCP 把可靠性、顺序和状态管理放进协议层，UDP 把更多自由度留给应用层。理解这点之后，你就不会再把它们当成“一个完整、一个简化”，而会知道它们分别适合什么目标。
