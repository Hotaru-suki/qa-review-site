# TCP 流量控制与拥塞控制

## 这个知识点在解决什么问题
TCP 可靠传输不只是重传和握手。真正决定“为什么有时候越发越慢、为什么网络一抖吞吐会掉”的，是流量控制和拥塞控制。它们都在限制发送速度，但解决的问题完全不同。

## 流量控制在控制什么
流量控制解决的是：接收方处理不过来怎么办。

接收方会通过接收窗口告诉发送方：
- 我现在还能接收多少数据
- 你别发太快

所以它更像“终点能力限制”。

## 拥塞控制在控制什么
拥塞控制解决的是：网络路径本身扛不住怎么办。

即使接收方还能收，如果链路已经拥塞，继续猛发只会导致丢包、重传和更差的吞吐。所以发送方会根据网络反馈，动态调整拥塞窗口。

## 两者的区别
```text
流量控制 -> 防止接收方来不及处理
拥塞控制 -> 防止网络中间路径被打爆
```

发送端真正能发多少，通常同时受这两者限制。

## 为什么测试和排障会碰到它
例如接口在弱网下：
- RTT 变大
- 丢包增加
- 重传增多

这时吞吐下降不一定是服务端性能差，也可能是 TCP 自身在保护网络和接收端。

## 典型现象
- 带宽看着够，但吞吐一直上不去
- 丢包率一上来，RTT 和重传明显增多
- 某些长连接在弱网环境下抖动很明显

## 常见误区
- 把所有吞吐下降都归因到服务端慢
- 知道窗口这个词，但分不清接收窗口和拥塞窗口
- 只会背算法名，却不知道它们在限制什么
