# TCP 三次握手、四次挥手与可靠性

## 摘要

三次握手和四次挥手不是孤立流程题，它们体现的是：TCP 如何在通信前建立一致状态、在通信中维护可靠字节流、在结束时有序释放连接。真正理解 TCP，不能只背“客户端发、服务端回、客户端再回”，而要知道每一步到底在确认什么，为什么不是两次，为什么关闭通常又不是三次。

---

## 核心问题

- 三次握手到底在确认什么
- 为什么建立连接通常是三次，而关闭连接通常是四次
- TCP 的可靠性到底是靠哪些机制拼起来的
- 这些机制在测试、抓包和排障时分别意味着什么

---

## 1. 三次握手不是礼貌流程，而是在同步状态

可以把 TCP 握手理解成：双方在正式传数据前，先确认彼此都能建立一段可靠连接，并对初始序号达成一致。

一个简化字符图：

```text
客户端                        服务端
  | ---- SYN(seq=x) --------> |
  | <--- SYN(seq=y),ACK(x+1)- |
  | ---- ACK(y+1) ----------> |
```

这三步分别在做：

### 第一次

客户端告诉服务端：

- 我想建立连接
- 我的初始序号从 `x` 开始

### 第二次

服务端告诉客户端：

- 我收到了你的建立请求
- 我也愿意建立连接
- 我的初始序号从 `y` 开始

### 第三次

客户端再告诉服务端：

- 我收到了你的确认和你的初始序号
- 我们现在对这段连接状态达成一致

也就是说，三次握手的核心不是“打三次招呼”，而是让双方都知道：

- 对方能收
- 对方能发
- 双方的连接初始状态一致

---

## 2. 为什么不是两次握手

如果只有两次：

```text
客户端 -> 服务端：我要连
服务端 -> 客户端：好，连吧
```

服务端并不能百分之百确认客户端已经收到自己的那次确认。

第三次握手的作用，就是让服务端知道：

- 自己发出的确认已经成功到达客户端
- 客户端确实知道这条连接可以开始使用

没有这一步，双方对连接状态的认知可能不一致。

---

## 3. 为什么关闭连接通常是四次挥手

关闭连接时，最容易误解的是：为什么建立是三次，关闭却常常四次。

原因在于：TCP 是全双工连接，双方“发送数据”和“接收数据”是两条相对独立的方向。

一方不再发送，不代表另一方也立刻发完了。

字符图可以看成：

```text
客户端                        服务端
  | ---- FIN ---------------> |
  | <--- ACK ---------------- |
  | <--- FIN ---------------- |
  | ---- ACK ---------------> |
```

意思是：

1. 客户端说：我这边发完了
2. 服务端说：我知道你发完了
3. 服务端等自己也发完后，再说：我这边也发完了
4. 客户端最后确认：我知道你也发完了

所以关闭连接更像“双方分别结束各自发送通道”，而不是简单对称回去。

---

## 4. TCP 的可靠性到底靠什么

TCP 可靠性不是某一个功能单独完成的，而是一组机制协作的结果。

### 序号

每段数据都有序号，用来识别：

- 哪些数据已经到
- 哪些还没到
- 数据顺序应该怎样重组

### 确认应答 ACK

接收方会告诉发送方自己已经收到哪些数据。

这让发送方知道：

- 可以继续发哪些
- 哪些还需要等待确认

### 超时重传

如果某段数据发出后长时间没有收到确认，发送方会重新发送。

这就是“丢了还能补回来”的核心。

### 流量控制

接收方会通过窗口大小告诉发送方：

- 我现在还能接多少

避免发送方把接收方压垮。

### 拥塞控制

网络本身也不是无限承载的。TCP 会根据拥塞情况调整发送速率，避免网络过载。

所以“TCP 可靠”不是一句标签，而是一整套状态和反馈机制的组合。

---

## 5. 一个抓包视角的理解

如果你在抓包工具里看 TCP，常见会看到：

- `SYN`
- `SYN, ACK`
- `ACK`
- 后续数据包和确认包
- 可能的重传
- `FIN, ACK`

这时候不能只看包名，还要知道它们在表达什么。

例如看到大量重传，意味着：

- 包可能丢了
- 或确认迟迟没回来
- 或链路质量差、延迟大、拥塞严重

看到三次握手迟迟起不来，可能说明：

- 服务端没监听
- 网络不可达
- 防火墙拦截
- 服务端 backlog 或资源有问题

这就是为什么理解握手和可靠性，会直接影响排障。

---

## 6. 一个测试或定位场景

假设用户反馈“偶尔请求超时，但服务端其实收到了请求”。

这时你不能只停留在“可能网络不好”，而要进一步判断：

- 是请求包丢了
- 还是响应包丢了
- 是连接建立慢
- 还是中间发生重传
- 是客户端等待超时
- 还是服务端处理慢

如果从 TCP 视角看抓包和日志，往往会重点关注：

- 三次握手是否正常完成
- 数据包是否出现重传
- ACK 是否异常延迟
- 关闭连接是否异常

这时候 TCP 机制知识就不再是背诵题，而是定位工具。

---

## 7. 常见误区

- 只会背三次握手流程，不知道每一步在确认什么
- 只会说“TCP 可靠”，却说不出可靠性由什么机制构成
- 以为 TCP 可靠就代表“绝不会丢、绝不会慢”
- 把四次挥手死记成模板，不理解全双工关闭的原因

这些误区一旦遇到追问，回答很快就会散。

---

## 小结

三次握手、四次挥手和 TCP 可靠性，本质上是同一件事的不同阶段：先建立一致状态，再通过序号、确认、重传和控制机制维持可靠传输，最后再有序关闭连接。只要把这条线串起来，TCP 就不再只是流程图，而是一套可理解、可抓包、可排障的协议机制。
