# 自定义异常、边界转换与错误语义

## 摘要

异常处理真正难的地方，不是会不会 `try/except`，而是异常到底要不要往外透、要不要转换、怎样让上层一看就知道错误属于哪一类。只会到处抛 `Exception` 的代码，很快就会让调用边界变得模糊，日志和排障也会越来越难看。

---

## 核心问题

- 为什么异常需要有明确语义，而不是都抛通用异常
- 什么叫“在边界做异常转换”
- 测试开发里怎样通过异常设计让问题更容易定位

---

## 1. 为什么自定义异常有价值

如果所有错误都抛：

```python
raise Exception("failed")
```

上层几乎无法一眼看出：

- 是参数错误
- 是网络错误
- 是业务状态错误
- 还是依赖系统错误

自定义异常真正提供的是：

- 错误语义
- 处理边界

---

## 2. 什么叫边界转换

边界转换指的是：

- 底层抛出的技术异常
- 在跨层时被转换成更符合上层语义的异常

例如：

- 底层 HTTP 超时
- 在业务层转换成“下游订单服务不可用”

这样上层读到异常时，理解成本会明显更低。

### 一个更实际的边界例子

比如测试平台调用订单服务：

- 底层可能抛 `TimeoutError`
- 平台层更希望知道“订单服务暂不可用”
- 用例层更希望区分“环境依赖失败”和“业务断言失败”

如果每一层都保留自己的语义边界，排障和失败归类会清楚很多。

---

## 3. 一个简单例子

```python
class OrderServiceError(Exception):
    pass

def query_order(order_id):
    try:
        return http_client.get(f"/orders/{order_id}")
    except TimeoutError as e:
        raise OrderServiceError("query order timeout") from e
```

这里不是把底层信息丢掉，而是：

- 用更高一层的语义包住它
- 同时保留原始异常链

### 为什么 `raise ... from e` 很关键

因为它能同时满足两件事：

- 上层拿到更可理解的业务语义
- 底层原始根因还在异常链里

如果只改异常名不保留链路，最后日志里就只剩抽象结论，很难追到真正的超时、连接失败或解析错误。

---

## 4. 为什么测试开发很需要这件事

自动化、测试平台、工具脚本里常见很多边界：

- 文件边界
- 网络边界
- 数据库边界
- 业务平台边界

如果这些边界都只抛原始异常，上层脚本会很难判断：

- 是环境问题
- 是依赖问题
- 还是业务断言失败

一旦异常语义清楚，测试报告和失败分类也会更稳。

### 这对自动化平台有什么直接价值

很多平台都需要把失败自动分桶成：

- 环境错误
- 依赖错误
- 用例逻辑错误
- 业务断言错误

如果异常都是 `Exception`，平台很难做稳定分类；如果边界语义清楚，很多统计和告警就能自动化。

---

## 5. 一个更完整的设计思路

更稳的异常层次通常是：

- 底层保留技术异常
- 边界层做语义转换
- 顶层根据语义决定重试、告警还是失败归档

这意味着不是每一层都应该造新异常，而是：

- 真正跨边界时再转换
- 同一层内部尽量保持语义一致

---

## 6. 常见误区

- 什么都抛 `Exception`
- 转换异常时把原始异常链丢掉
- 每一层都随意改名，导致语义更乱
- 业务异常和系统异常不区分

---

## 小结

异常设计不是“多写几个类”，而是让错误在跨层传播时保持可理解、可分类、可追踪。只要系统开始有明显边界，自定义异常和边界转换就值得认真设计。
