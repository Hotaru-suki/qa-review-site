# 类、对象与面向对象怎么理解

## 摘要

Python 里的类不是为了“写得更像 Java”，而是为了把状态和行为组织成更稳定的抽象。对象真正有价值的地方，是它让数据和操作有了明确归属。什么时候应该继续用函数，什么时候值得引入类，关键看的是状态是否长期存在、行为是否围绕同一份状态反复发生，以及边界是否需要更清楚地表达。

---

## 核心问题

- 类和对象在 Python 里分别代表什么
- 为什么面向对象的重点不是语法，而是建模
- 什么场景下用函数就够了，什么场景下更适合引入类
- 一个类最起码应该把哪些边界说清楚

---

## 1. 类和对象分别是什么

可以把类理解成“某类对象的设计图”，把对象理解成“按这张设计图创建出来的具体实例”。

例如：

```python
class User:
    def __init__(self, name, role):
        self.name = name
        self.role = role

    def can_publish(self):
        return self.role in {"admin", "editor"}
```

这里：

- `User` 是类
- `User("alice", "admin")` 是对象

对象的价值在于：它把相关状态和行为组织在一起。

---

## 2. 为什么重点不是 `class` 语法，而是建模

面向对象最重要的不是会写：

```python
class X:
    ...
```

而是知道：

- 什么状态应该放在一起
- 什么行为应该归这个对象负责
- 哪些细节应该封装
- 哪些信息应该作为对外接口暴露

如果这些边界没想清楚，类就只是把一堆过程式代码包进了壳里，复杂度反而更高。

---

## 3. 什么场景下函数就够了

如果任务只是：

- 读一个文件
- 做一次转换
- 输出结果

函数通常更简单直接。

例如：

```python
def normalize_username(name):
    return name.strip().lower()
```

这里没有长期状态，没有多个相关动作，也没有对象身份，没必要为了“面向对象”强行套类。

---

## 4. 什么场景下更适合引入类

当下面几种需求出现时，类的价值会更明显：

- 一组状态会一起变化
- 有多个操作都围绕同一份状态工作
- 需要创建多个相似实例
- 需要把内部状态和对外接口区分开

例如一个下载任务对象：

```python
class DownloadTask:
    def __init__(self, url):
        self.url = url
        self.status = "pending"
        self.retry_count = 0

    def start(self):
        self.status = "running"

    def mark_failed(self):
        self.status = "failed"
        self.retry_count += 1
```

这里类的意义就在于：状态和操作是天然绑定的。

---

## 5. `self` 到底是什么

`self` 不是关键字，只是约定俗成的第一个实例参数名。它表示“当前这个对象自己”。

例如：

```python
class Counter:
    def __init__(self):
        self.value = 0

    def inc(self):
        self.value += 1
```

调用：

```python
c = Counter()
c.inc()
```

本质上是把 `c` 这个对象传给了方法中的第一个参数。

所以 `self.value` 表示的不是“全局变量 value”，而是“当前这个对象自己的 value”。

---

## 6. 类最重要的边界是什么

一个类至少要把下面几件事说清楚：

- 它负责管理什么状态
- 哪些方法负责改变状态
- 对象创建后允许处于哪些合法状态
- 外部应该通过什么接口和它交互

如果这些边界不清楚，就会出现常见问题：

- 外部代码到处直接改对象内部字段
- 一个类承担太多职责
- 方法只是在搬运外部逻辑，没有真正围绕对象状态工作

这时类会变得越来越难维护。

---

## 7. 一个脚本到类的演化例子

假设最开始你只有一个简单函数：

```python
def run_case(case_id):
    print(f"run {case_id}")
```

后来需求变成：

- 需要记录执行状态
- 需要记录开始时间和结束时间
- 需要重试次数
- 需要输出执行结果

这时继续用零散函数也不是不行，但状态会分散在很多变量里。

更自然的改法通常是：

```python
class TestCaseRunner:
    def __init__(self, case_id):
        self.case_id = case_id
        self.status = "pending"
        self.retry_count = 0

    def run(self):
        self.status = "running"

    def mark_passed(self):
        self.status = "passed"

    def mark_failed(self):
        self.status = "failed"
        self.retry_count += 1
```

这时类的价值就体现出来了：它把“这个测试用例执行过程中的状态”组织成了一个明确对象。

---

## 8. 类不是越多越好

很多初学者一接触面向对象，就容易犯两个极端错误：

- 所有逻辑都写成函数，没有状态归属
- 所有东西都强行封进类里，最后类比函数更混乱

更稳的原则是：

- 无状态、一次性处理逻辑，优先函数
- 有状态、反复演化、行为围绕状态展开，优先类

类的价值在于表达“谁拥有这份状态，谁负责操作它”，不是为了写出更多面向对象名词。

---

## 9. 一个测试和工程视角

在测试工具或自动化框架里，类常见适合表达：

- 测试用例对象
- 页面对象
- 接口客户端
- 配置上下文
- 任务执行器

因为这些对象通常都有持续状态和一组相关行为。

但像“把 JSON 转成字典”“提取日志字段”这种一次性数据处理，函数往往更直接。

---

## 小结

类的核心不是继承层级，也不是语法形式，而是把“谁拥有状态、谁负责操作这些状态”这件事说清楚。面向对象最重要的不是写出多少类，而是知道什么时候类真的能让结构更清楚，什么时候函数已经足够。
