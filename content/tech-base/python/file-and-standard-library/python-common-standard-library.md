# 高频标准库怎么理解和选用

## 摘要

Python 标准库的价值不在于“自带很多模块”，而在于它已经把大量通用问题抽象成了稳定工具。会选标准库，通常比从零手写轮子更重要。对测试、脚本和自动化场景来说，这意味着你需要先识别自己面对的到底是路径问题、文本处理问题、时间问题、进程调用问题，还是数据结构问题，再选合适模块，而不是见到需求就先写一堆底层逻辑。

---

## 核心问题

- 哪些标准库能力在脚本和自动化里最常见
- 为什么标准库选择本质上是在选择抽象层次
- 什么时候该先找标准库，什么时候才考虑第三方库

---

## 1. 为什么标准库值得优先考虑

标准库的最大价值，不只是“官方自带”，而是：

- 接口相对稳定
- 文档和社区资料丰富
- 大量常见问题已经被抽象好
- 不需要额外安装，环境依赖更简单

对测试和自动化来说，这点尤其重要。很多脚本最终运行在：

- CI 环境
- 服务器
- 临时调试机
- 同事的本地环境

如果能先用标准库解决问题，部署和维护成本通常更低。

---

## 2. 高频标准库分别在解决什么问题

### `pathlib` 和 `os`

这组模块主要解决：

- 路径拼接
- 文件和目录存在性判断
- 遍历目录
- 环境变量读取
- 基础文件系统操作

优先使用 `pathlib` 的原因，是它把路径操作变成了对象化表达，比字符串拼接更清晰、更不容易出错。

例如：

```python
from pathlib import Path

log_dir = Path("logs") / "api"
if log_dir.exists():
    for file in log_dir.iterdir():
        print(file.name)
```

这比手工拼 `"/"` 或 `"\\\\"` 更稳。

### `json`

主要解决结构化数据的序列化和反序列化，常见于：

- 接口返回处理
- 配置文件读写
- 自动化结果输出

例如：

```python
import json

data = {"case": "login", "passed": True}
text = json.dumps(data, ensure_ascii=False, indent=2)
obj = json.loads(text)
```

这里要注意，`json` 适合结构化数据，不适合拿来处理任意对象或大规模流式日志。

### `re`

主要解决规则匹配和文本提取。

常见场景包括：

- 从日志中抽取时间、状态码、请求 ID
- 校验字符串格式
- 批量替换文本

但 `re` 不是所有文本处理问题的首选。如果只是简单切分、包含判断、前后缀处理，字符串方法往往更直接。

### `datetime`

主要解决：

- 时间戳转换
- 时间加减
- 日期格式化
- 日志时间处理

它的重要性在于：只要脚本碰到“昨天的日志”“超时 5 分钟”“按小时汇总”等需求，时间处理通常不能靠字符串硬拼。

### `collections`

这一组更像“更适合的基础数据结构”，比如：

- `defaultdict` 适合聚合统计
- `Counter` 适合计数
- `deque` 适合队列和滑动窗口

它们的价值在于：让代码表达更贴近问题，而不是手工维护一堆边界逻辑。

### `subprocess`

主要解决 Python 脚本和系统命令的协作问题。

典型场景：

- 调用 `adb`
- 调用 `git`
- 调用 `ffmpeg`
- 拉起测试工具或系统脚本

例如：

```python
import subprocess

result = subprocess.run(
    ["adb", "devices"],
    capture_output=True,
    text=True,
    check=False,
)
print(result.stdout)
```

这里比直接用 `os.system` 更稳，因为你能明确拿到返回码、标准输出和标准错误。

## 3. 标准库选择本质上是在选抽象层次

很多人把“会用标准库”理解成“知道有哪些模块”，其实不够。

更重要的是识别问题属于哪一层：

- 如果你要处理路径，先想 `pathlib`
- 如果你要做结构化序列化，先想 `json`
- 如果你要做模式提取，先想 `re`
- 如果你要和系统命令交互，先想 `subprocess`
- 如果你需要更合适的容器，先想 `collections`

这就是抽象层次的选择。抽象选对了，代码自然短、稳、清晰；抽象选错了，代码就会充满手工判断和重复逻辑。

---

## 4. 一个更完整的脚本场景

例如你要写一个“读取日志文件，提取失败请求，按日期输出 JSON 报告”的脚本。

这时通常会组合用到：

- `pathlib`：定位日志目录和输出路径
- `re`：提取日志里的请求 ID、状态码、时间
- `datetime`：把字符串时间转成日期对象，按天归类
- `collections`：统计某类错误出现次数
- `json`：把最终结果输出成结构化报告

这说明标准库真正的价值不是单个模块，而是它们可以作为稳定积木协作起来。

---

## 5. 什么时候该用标准库，什么时候再考虑第三方库

通常优先考虑标准库的情况：

- 问题常见且不复杂
- 对环境依赖敏感
- 团队希望脚本可直接运行
- 需求是路径、文本、时间、基础数据处理这类通用问题

再考虑第三方库的情况：

- 标准库表达明显繁琐
- 需要专业能力，例如 HTTP 客户端、数据分析、图像处理
- 性能、协议支持或生态整合要求更高

例如：

- 做 HTTP 请求，标准库能做，但大多数时候 `requests` 更顺手
- 做复杂时间处理，标准库能覆盖基础需求，但某些高级时区需求可能更适合专门库

重点不是迷信“只用标准库”，而是先判断问题规模和复杂度。

---

## 6. 常见错误

- 明明标准库已足够，却习惯性引第三方依赖
- 所有文本处理都用正则，结果代码难维护
- 用字符串拼路径，跨平台时容易出错
- 用 `os.system` 调命令，却拿不到可靠的返回结果和错误信息

这些错误本质上都和“没有先识别问题类型”有关。

---

## 小结

标准库使用能力的核心，不是背 API，而是先判断问题属于哪一类通用能力，再选合适的抽象。只要你能把“路径、文本、时间、结构化数据、系统调用、容器”这些常见问题和对应标准库建立稳定映射，写脚本和自动化时就会顺很多。
