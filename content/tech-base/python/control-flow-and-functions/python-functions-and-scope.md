# 函数、作用域与参数传递

## 摘要

函数不是简单的“把代码包起来复用”，而是程序组织边界。它决定一段逻辑的输入输出、局部状态、返回值和副作用。作用域规则决定变量在什么位置可见，参数传递则决定对象怎样跨函数边界流动。很多 Python 新手常见 bug，例如变量名覆盖、外部列表被悄悄改掉、默认参数累积状态，本质上都和这三件事没有想清楚有关。

---

## 核心问题

- 函数为什么是代码组织边界，而不是代码压缩工具
- Python 的作用域规则如何影响变量读写
- 参数传递为什么常和可变对象、副作用一起出问题
- 写函数时怎样把输入、输出和副作用设计清楚

---

## 1. 函数到底在解决什么问题

函数最核心的价值有三点：

- 把一段逻辑封装成明确输入和输出
- 隐藏内部实现细节
- 让调用方只关心“我要什么结果”

例如：

```python
def normalize_username(name):
    return name.strip().lower()
```

调用方只要知道：

- 传进去一个用户名字符串
- 拿回来一个标准化结果

不需要关心内部是怎么去空格、怎么转小写的。

这就是函数的意义：建立边界。

---

## 2. 作用域到底是什么

作用域可以简单理解成：一个变量名在当前代码位置能不能被访问。

Python 里最常接触的是：

- 局部作用域：函数内部定义的变量
- 全局作用域：模块顶层定义的变量

例如：

```python
x = 10

def f():
    y = 20
    print(x)  # 可以访问全局变量
    print(y)  # 可以访问局部变量
```

函数执行结束后，`y` 这个局部变量就不会再作为函数外部可见名字继续存在。

所以作用域的价值，是让不同层级的变量边界更清晰，避免所有变量混在一起。

---

## 3. Python 读取和赋值时为什么感觉不一样

很多人第一次碰作用域问题时会困惑：

- 为什么有时函数里能直接读全局变量
- 但一旦赋值，就会报局部变量相关错误

例如：

```python
count = 0

def add():
    count = count + 1
```

这段代码会报错，因为 Python 在函数里看到 `count = ...`，会把 `count` 当成本地变量。但右侧又先读取了它，此时本地变量还没初始化。

如果你真的要修改外层变量，需要明确声明：

```python
count = 0

def add():
    global count
    count += 1
```

但在工程里，频繁改全局变量通常不是好习惯，更推荐通过参数和返回值传递状态。

---

## 4. 参数传递到底发生了什么

Python 的参数传递常被口语化地说成“传引用”或“传值”，这两种说法都容易引起误解。

更准确的理解是：

- 函数形参会绑定到传入对象
- 如果对象是可变的，函数内部修改它，外部就可能感知到变化
- 如果对象是不可变的，重新绑定通常不会影响外部变量

例如：

```python
def add_tag(tags):
    tags.append("new")

items = ["old"]
add_tag(items)
print(items)  # ['old', 'new']
```

这里 `items` 被改掉，是因为 `list` 是可变对象，函数内部直接修改了对象内容。

再看一个例子：

```python
def add_one(x):
    x = x + 1

n = 3
add_one(n)
print(n)  # 3
```

这里外部 `n` 没变，是因为整数是不可变对象，函数里只是让局部名字 `x` 重新绑定到了一个新整数。

## 5. 一个最常见的副作用问题

副作用指的是：函数除了返回值之外，还修改了外部可观察状态。

最常见的副作用来源，就是可变对象参数。

```python
def sort_in_place(nums):
    nums.sort()

arr = [3, 1, 2]
sort_in_place(arr)
print(arr)  # [1, 2, 3]
```

这段函数没有问题，但前提是你必须明确知道：它会直接修改原列表。

如果调用方本来期待“原数据不变，返回排序结果”，这个设计就会引发 bug。

因此写函数时要尽量说清楚：

- 这是原地修改
- 还是返回新对象

---

## 6. 一个更稳的函数设计方式

设计函数时可以优先问自己四件事：

1. 输入是什么
2. 输出是什么
3. 会不会修改传入对象
4. 会不会依赖外部隐藏状态

例如比起这样：

```python
cache = {}

def get_user_name(user_id):
    ...
```

更稳的写法往往是把依赖显式传入，或至少把副作用说清楚。

这会让函数更容易测试，也更容易排查问题。

---

## 7. 一个默认参数的经典坑

Python 里还有一个和函数边界强相关的经典问题：可变默认参数。

```python
def append_item(x, items=[]):
    items.append(x)
    return items
```

很多人以为每次调用都会拿到一个全新的空列表，实际上默认参数在函数定义时就创建好了，后续调用会复用同一个列表。

更稳的写法是：

```python
def append_item(x, items=None):
    if items is None:
        items = []
    items.append(x)
    return items
```

这个坑本质上也不是语法细节，而是函数边界和对象生命周期没有想清楚。

## 8. 一个测试或脚本场景

假设你写一个自动化脚本，用函数把接口返回结果整理成报告。

如果函数内部直接修改传入的原始响应字典：

```python
def normalize(resp):
    resp["status"] = resp["status"].lower()
    return resp
```

那么后续如果其他地方还要使用原响应，可能就会受到影响。

这时更稳的方式往往是：

- 要么明确说明这是原地修改函数
- 要么先拷贝，再返回新结果

否则你在排查“为什么原始数据后面变了”时，会很难定位问题。

---

## 9. 常见误区

- 用全局变量到处传状态
- 函数既返回结果又悄悄改输入对象
- 看到参数就默认以为“传进去的是副本”
- 忽略可变默认参数问题

这些问题表面看像语法问题，本质上都是边界设计问题。

---

## 小结

学函数时真正要掌握的，不只是 `def` 语法，而是边界感。输入输出清楚、作用域清楚、副作用可控，函数才会成为组织代码的工具，而不是新的混乱源头。Python 的作用域和参数传递规则并不难，难的是是否真的把“这个名字属于谁、这个对象会不会被改、这个状态从哪里来”想清楚。
