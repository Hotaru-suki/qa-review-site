# curl、ss、tcpdump 组合排查路径

## 这个知识点在解决什么问题
网络排查最怕只会单个命令。`curl`、`ss`、`tcpdump` 分别看的是不同层面：请求是否通、连接有没有建立、包到底发没发出去。把它们组合起来，才能形成从应用层到传输层的排查链。

## 三个命令各看什么
### `curl`
更偏应用层：
- URL 能不能访问
- HTTP 状态码是什么
- 返回头和返回体是什么
- TLS 握手、重定向、超时发生在哪

### `ss`
更偏连接层：
- 端口是否在监听
- 当前 TCP 连接状态
- 哪个进程占用了端口

### `tcpdump`
更偏抓包：
- 请求包有没有发出去
- 响应包有没有回来
- SYN、ACK、RST、FIN 这些包发生了什么

## 一个常见排查顺序
1. 先用 `curl` 看接口表面现象
- 是超时、拒绝连接，还是返回 500

2. 再用 `ss` 看端口和连接
- 服务有没有监听
- 是否有大量 `TIME_WAIT`、`CLOSE_WAIT`

3. 最后用 `tcpdump` 抓关键流量
- 包有没有走到对端
- 是否中途被重置或丢弃

## 典型场景
### `curl` 超时
可能是：
- DNS 解析慢
- 服务没监听
- 网络不通
- 对端处理太慢

这时如果 `ss` 看不到目标端口监听，问题就更像服务没起来；如果监听正常但 `tcpdump` 看不到响应包，问题更可能在网络或对端处理。

## 测试开发为什么要会这套
接口测试或联调时，经常会遇到：
- 本机能调，测试环境调不通
- 偶发超时
- 服务端说没收到请求

这时只会看接口返回不够，必须能把应用层、连接层、抓包层串起来。

## 常见误区
- 只会 `curl`，看到超时就不知道下一步
- 抓包抓得太大，没有先缩小目标 IP、端口和时间范围
- 不结合进程和端口信息，抓到包也不知道对应哪个服务
