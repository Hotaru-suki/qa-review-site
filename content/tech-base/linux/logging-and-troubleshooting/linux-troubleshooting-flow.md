# Linux 排障的一般路径

## 摘要

Linux 排障最怕的不是不会命令，而是顺序混乱：现象没确认就重启、网络没排就改配置、日志没看就怀疑代码。真正高效的排障，本质上是一个证据收束过程，要把现象、范围、分层判断、关键证据和根因逐步连成一条路径。

---

## 核心问题

- 为什么排障顺序会直接决定定位效率
- Linux 常见问题应该先从哪几层切入
- 怎样避免把网络、服务、资源、配置和业务问题混在一起

---

## 1. Linux 排障真正要做什么

排障不是“试几个命令看能不能好”，而是回答下面几件事：

- 问题是否真实存在、是否稳定复现
- 影响范围是单机、单环境还是全局
- 问题卡在哪一层
- 哪些证据能支撑当前判断
- 根因是现象层、依赖层还是业务层

如果这些问题没被逐步回答，排障通常就还停在猜。

---

## 2. 一个更稳的排障总路径

比较通用的顺序通常是：

```text
确认现象
  ↓
判断影响范围
  ↓
先做分层
  ↓
收集关键证据
  ↓
定位到具体模块或依赖
  ↓
验证修复是否生效
```

这条路径看起来简单，但它解决的是排障里最常见的问题：

- 太早动手
- 太晚分层
- 证据太散
- 结论跳太快

---

## 3. 第一步为什么一定要先确认现象

排障前最好先问清：

- 问题能否稳定复现
- 从什么时候开始
- 影响所有请求还是部分请求
- 只有某台机器出问题，还是多台一起出问题
- 失败表现是超时、拒绝连接、500、性能下降还是数据错误

这些信息会直接改变排查路径。

例如：

- 只有一台机器异常，更像单机环境问题
- 所有机器同一时段都异常，更像公共依赖或流量问题
- 只在高峰时段出现，更像资源瓶颈或竞争问题

所以“确认现象”不是废话，而是在决定排障入口。

---

## 4. 第二步先做分层，不要直接猜根因

Linux 上的很多问题，最终都要落到某一层：

### 网络层

- DNS 解析
- 端口可达性
- 路由
- 超时与丢包

### 服务层

- 进程是否存在
- 服务是否启动
- 监听端口是否正确
- systemd / 启动项是否正常

### 配置层

- 路径
- 权限
- 环境变量
- 配置值

### 资源层

- CPU
- 内存
- 磁盘
- 文件句柄
- 连接数
- 线程池

### 业务层

- 应用逻辑
- 数据库
- 缓存
- 下游依赖

分层的意义在于：  
先回答“问题大概停在哪一层”，再下钻，不然很容易出现：

- 服务没起，却一直查 SQL
- 磁盘满了，却一直改配置
- DNS 问题，却一直看代码逻辑

---

## 5. 各层通常看什么证据

### 网络层

常看：

- `ping`
- `curl`
- `telnet` / `nc`
- DNS 解析结果
- 超时与拒绝连接现象

重点问题：

- 是根本连不上，还是连上后响应异常
- 是域名问题，还是端口问题

### 服务层

常看：

- `ps`
- `systemctl status`
- `ss -lntp`
- 服务日志

重点问题：

- 服务没起
- 服务起了但没监听
- 服务反复重启

### 配置层

常看：

- 配置文件
- 环境变量
- 运行用户
- 目录权限

重点问题：

- 路径是否正确
- 配置值是否生效
- 权限是否匹配

### 资源层

常看：

- `top`
- `free -h`
- `df -h`
- `iostat`
- `ulimit`

重点问题：

- 资源是否打满
- 是否有明显竞争或耗尽

### 业务层

常看：

- 应用日志
- SQL 日志
- 下游返回
- trace / request_id

重点问题：

- 是逻辑异常，还是依赖异常
- 是单次失败，还是系统性波动

---

## 6. 一个典型接口访问失败场景怎么排

假设测试环境里接口调用失败。

更稳的排查顺序可以是：

1. 先确认失败表现是超时、连接拒绝还是 500
2. 如果是超时或拒绝连接，先查网络和服务层
3. 如果端口已监听、服务已启动，再查应用日志
4. 如果日志显示下游失败，再继续查数据库或依赖服务
5. 最后再判断是不是代码逻辑或环境配置问题

对应的思路可以画成：

```text
接口失败
 ├─ 连不上 -> 网络 / 服务
 ├─ 连上报 5xx -> 应用 / 依赖
 ├─ 只高峰失败 -> 资源 / 性能
 └─ 结果错但调用成功 -> 业务 / 数据
```

这个树状判断，比“遇到问题先看日志”更稳，因为它先利用现象做分流。

---

## 7. 为什么很多排障会低效

常见低效行为有：

### 7.1 还没确认现象就先重启

重启可能暂时掩盖问题，还会冲掉现场。

### 7.2 不先分层，直接全局搜

这会导致信息量很大，但方向很散。

### 7.3 只有命令，没有判断

会执行 `top`、`ps`、`grep` 不代表会排障，关键是知道这些命令各自在验证什么假设。

### 7.4 证据收集不成链

日志、端口、配置、监控各看一点，但最后拼不成明确结论。

---

## 8. 排障时怎样形成证据链

一条比较像样的证据链通常会长这样：

```text
用户反馈 / 监控告警
  ↓
确认失败时间和范围
  ↓
判断落在哪一层
  ↓
拿到关键日志 / 状态 / 资源数据
  ↓
把证据收束到某个服务、依赖或配置
  ↓
修复后复验
```

例如：

- 10:32 开始接口超时
- 只影响某一台机器
- `systemctl status` 显示服务 failed
- 日志显示配置文件路径不存在
- 修改路径后服务恢复

这就是完整排障链，而不是“后来改对了”。

---

## 9. 测试视角下为什么必须懂这套路径

测试工作里常见的问题不一定是代码 bug，也可能是：

- 环境部署错
- 服务没起来
- 配置不一致
- 权限不对
- 依赖服务未就绪
- 资源打满

如果不懂 Linux 排障路径，很多问题只能停留在“我复现了，交给开发看”。  
懂这套方法后，你至少能先把问题压缩到：

- 网络
- 服务
- 配置
- 资源
- 业务

这会显著提升问题定位和沟通效率。

---

## 小结

Linux 排障不是命令竞赛，而是按顺序收集证据、逐层缩小范围的过程。真正高效的做法始终是：先确认现象，再分层判断，再收集关键证据，最后把问题收束到具体模块或依赖，并用复验确认结论成立。
