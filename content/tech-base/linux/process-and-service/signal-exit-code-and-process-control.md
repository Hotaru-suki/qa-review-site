# 信号、退出码与进程控制

## 这个知识点在解决什么问题
服务启动、停止、异常退出时，很多信息都藏在信号和退出码里。理解信号、退出码与进程控制，能帮你区分：这是正常停止、被手动杀掉、被系统中断，还是程序自身崩溃。

## 信号在表达什么
信号是操作系统发给进程的一种异步通知，常见如：
- `SIGTERM`：请求优雅退出
- `SIGKILL`：强制终止
- `SIGINT`：中断
- `SIGSEGV`：非法内存访问

## 退出码在表达什么
退出码通常表示程序结束结果：
- `0`：正常结束
- 非 `0`：某类错误或异常结束

对于服务排障来说，退出码和信号一起看，比只看“进程没了”有用得多。

## 一个典型场景
服务频繁重启时，你要先分清：
- 是程序自己退出
- 是 supervisor/systemd 拉起
- 还是被信号杀掉

如果是 `SIGTERM`，更可能是外部控制；  
如果是 `SIGSEGV`，更可能是程序崩溃。

## 退出码和信号为什么能帮助定位
因为它们回答的是“进程为什么结束”，而不是只告诉你“进程结束了”。

例如：
- 正常退出但返回非 0，可能是程序自己判错退出
- 收到 `SIGKILL`，更像被外部强制终止
- 收到 `SIGSEGV`，更像内存访问崩溃

## 测试开发里的典型用法
- 自动化拉服务后先检查退出码
- 看健康检查失败是不是因为服务根本没启动起来
- 长稳测试里区分“业务失败”与“进程退出”

## 一个更完整的判断例子
### 现象：服务一启动就退出
你可以按这个顺序看：
1. 启动命令返回码是不是非 0
2. 服务管理器是否立刻再次拉起
3. 日志里是否出现异常堆栈
4. 是否有 `SIGTERM`、`SIGKILL`、`SIGSEGV` 之类的线索

如果是：
- 返回码非 0 且无外部信号，更像程序自己初始化失败
- 收到 `SIGKILL`，更像被外部强制终止或资源不足
- 收到 `SIGSEGV`，更像内存访问问题

## 为什么这和自动化稳定性有关
自动化脚本如果只判断“端口没起来”，定位信息很弱。  
但如果能同时拿到：
- 退出码
- 信号
- 对应日志

就能更快区分：
- 环境没配好
- 依赖没起来
- 程序自身崩了

## 一个简单思路
```text
进程退出
  -> 看退出码
  -> 看是否有信号
  -> 对齐 service 管理日志
  -> 对齐系统级日志
```

这样比单看 `ps` 或“进程没了”更有定位价值。

## 测试开发为什么要懂
- 自动化拉起被测服务时，要判断服务是否真的起来
- 异常退出时，要快速区分环境问题、资源问题还是程序崩溃
- 长稳测试里，进程被重启不一定就是应用逻辑问题

## 常见误区
- 只看进程在不在，不看退出原因
- 遇到重启就先怪服务管理器
- 不区分优雅退出和被杀死
- 把所有异常退出都当成应用崩溃
- 自动化只做存活判断，不采集退出上下文
