# 文本查看与处理命令

## 摘要

Linux 排障时，很多信息最终都会落成文本。会看文本、切片文本、过滤文本，比单独会很多系统命令更常用。真正的差别不在于你是否记住了 `grep`、`awk`、`sed` 这些名字，而在于面对一大段日志时，你能不能迅速把它压缩成关键时间点、关键请求 ID、关键错误行和上下文。

---

## 核心问题

- 为什么日志排查首先是文本处理问题
- `cat`、`less`、`tail`、`grep`、`awk`、`sed` 分别更适合什么场景
- 如何避免把大文件一次性全读出来

---

## 1. 为什么排障本质上常是文本问题

无论你在查的是：

- 应用日志
- Nginx 日志
- 系统日志
- 配置文件
- 命令输出

最后都要落到文本分析。

所以 Linux 文本处理能力的核心价值，是把大量原始输出快速压缩成：

- 时间范围
- 关键字
- 字段值
- 错误上下文

这一步如果做不好，后面再懂系统原理也很难快速收敛问题。

---

## 2. 最常用命令分别在做什么

### `cat`

适合：

- 看很小的文件
- 快速确认文件内容

不适合：

- 直接输出很大的日志文件

### `less`

适合：

- 交互式浏览大文件
- 搜索关键字
- 前后翻页看上下文

通常比直接 `cat` 大日志更稳。

### `head` / `tail`

适合：

- 看文件开头
- 看最新日志

例如：

```bash
tail -n 100 app.log
tail -f app.log
```

### `grep`

适合：

- 关键字过滤
- 按错误码、请求 ID、模块名筛日志

例如：

```bash
grep "ERROR" app.log
grep "request_id=abc123" app.log
```

### `awk`

适合：

- 按列提取
- 做简单统计
- 根据字段条件过滤

例如日志里如果按空格分列，`awk` 很适合拿某一列值。

### `sed`

适合：

- 看某段行号范围
- 做简单替换
- 批量文本处理

例如：

```bash
sed -n '100,140p' app.log
```

很适合看错误行附近上下文。

---

## 3. 一个更稳的排障顺序

假设你拿到一份很大的应用日志，不应该上来就：

```bash
cat app.log
```

更稳的顺序通常是：

1. `tail` 先看最新输出
2. `grep` 缩到错误关键字或请求 ID
3. `sed` / `less` 看上下文
4. `awk` 提取重点字段

这和在代码里先缩小问题范围是一个思路：先压缩，再细看。

---

## 4. 一个具体场景

例如排查“某个请求偶发 500”：

### 第一步

先看最近日志：

```bash
tail -n 200 app.log
```

### 第二步

如果知道请求 ID：

```bash
grep "request_id=abc123" app.log
```

### 第三步

如果找到异常行号，再看附近上下文：

```bash
sed -n '820,860p' app.log
```

### 第四步

如果日志按列有状态码或耗时，再用 `awk` 提取：

```bash
awk '/request_id=abc123/ {print $1, $2, $NF}' app.log
```

这样一条排查链，就从“海量文本”变成了“一个请求的关键上下文”。

---

## 5. 常见误区

- 直接 `cat` 很大的日志文件
- 没有先缩小时间范围和关键字
- 提取字段时全靠肉眼找
- 不看上下文，只盯一行错误

真正高效的文本处理，不是会很多命令，而是知道先缩哪里、后看哪里。

---

## 小结

Linux 文本处理的价值，在于把大量原始输出快速压缩成关键线索。真正常用的不是命令本身，而是：先缩时间和关键字，再取字段和上下文。只要这条思路稳，日志、配置和命令输出就都能快速转成可分析信息。
